# 4. AI 시스템 평가하기

- 3장에서 자동 평가 방식들을 다뤘다면, 4장은 평가 방식들을 사용해 애플리케이션의 모델을 평가하는 방법을 다룰 것
- 4장의 3가지 파트
  - 애플리케이션을 평가할 때 사용할 수 있는 기준들에 대한 정의 및 계산 방법
    - 도메인 특화 역량은 어떻게 측정할 수 있을까? 등
  - 어떤 모델을 선택할 것인가
    - 벤치마크를 신뢰할 수 있는가, 어떤 벤치마크를 선택할 것인가
    - 자체 모델을 호스팅할 것인가, 모델 API를 사용할 것인가
  - 평가 파이프라인 개발 방법
    - 애플리케이션 개발 과정을 지속적으로 개선하고 평가하는 데 도움을 주는 요소

## 4.1 평가 기준

- evaluation-driven development
  - 애플리케이션에 시간과 돈을 투자하기에 앞서, 애플리케이션을 어떻게 평가할지 정하는 방식
  - 개발하기 전에 평가 기준을 정의하는 것
  - 현명한 비즈니스 결정은 투자 수익률을 기준으로 하며, 애플리케이션은 배포되기 위해 그 가치를 명확히 보여줘야 함
  - 엔터프라이즈 애플리케이션들의 평가 기준들
    - 추천 시스템: 참여도, 구매 전환율의 증가로 성공 여부 판단
    - 사기 탐지 시스템: 예방한 사기로 얼마나 많은 돈을 절약했는지로 측정
    - 코딩: 생성된 코드의 기능적 정확성 평가
    - 파운데이션 모델: 의도 분류, 감성 분석, 다음 행동 예측 등 폐쇄형 작업 평가
- 일반적인 범주의 평가 기준들
  - 도메인 특화 능력
  - 생성 능력
  - 지시 준수 능력
  - 비용
  - 지연 시간

### 도메인 특화 능력(domain-specific capability)

- 모델의 도메인 특화 능력은 모델 구조, 크기 등의 설정이나 학습에 따라 달라짐
- 이를 평가하기 위해 공개 혹은 비공개 도메인 특화 벤치마크를 활용
- 다양한 도메인의 능력을 평가하기 위해 수천 개의 공개 벤치마크가 나왔으며 계속 늘어나고 있음
- 코딩 관련 도메인 특화 능력의 경우
  - 보통 정확성 평가로 판단
  - 효율성 및 비용도 함께 고려 (실행 시간 및 메모리 사용량으로 정확히 측정 가능)
  - 코드 가독성도 주관적으로나마 평가
- 코딩이 아닌 도메인 능력의 경우
  - 주로 객관식 문제 같은 폐쇄형 문자로 평가
    - 객관식 문제는 만들기 쉽고, 검증하기 쉬움
    - 다만 문제와 선택지를 조금만 다르게 표현해도 모델의 성능이 달라질 수 있음
  - 객관식 문제(Multiple-Choice Question, MCQ)는 하나 이상의 정답이 있을 수 있음
  - 일반적인 지표는 정확도
    - 모델이 맞힌 문제 수
    - 혹은 점수 체계를 사용해서 어려운 문제에 더 높은 점수를 받게끔 하는 경우도 있음
- 폐쇄형 벤치마크가 널리 사용되고 있지만 파운데이션 모델 평가에 적합한지는 불분명
- 객관식은 객관적 지식 및 분류를 평가할 순 있지만, 생성 능력을 평가하는 데에는 부적합

### 생성 능력

- 생성형 AI 이전부터 자연어처리(NLP) 분야의 연구자들은 개방형 출력의 품질 평가 방법을 연구해 왔음
  - 개방형 텍스트 생성 연구 분야 = 자연어 생성(Natural Language Generation, NLG)
- 당시 생성된 텍스트 품질 평가에 유창성(fluency), 일관성(coherence), 충실성(faithfulness), 관련성(relevance) 지표가 포함되었음
  - 언어 모델의 생성 능력이 발전하면서 유창성과 일관성의 중요도가 낮아졌음
- 새롭게 평가해야 할 평가 기준들
  - 환각: 창의적인 작업에는 바람직하지만 사실 관계가 중요한 작업에는 치명적
  - 안정성: 생성된 출력이 사용자와 사회에 해를 끼칠 수 있는지 평가 (모든 종류의 유해성 및 편향성을 포함하는 개념)

**사실 일관성과 비일관성**

- 사실 비일관성(factual inconsistency)
  - 탐지하고 측정하기 위한 많은 기법들이 개발되고 있음
- 사실 일관성(factual consistency)
  - 2가지 방식으로 검증 가능
    - 명시적으로 제공된 사실(컨텍스트)을 기준으로 검증
    - 공개된 지식을 기준으로 검증
  - **국소적 사실 일관성(local factual consistency)**
    - 출력을 컨텍스트에 기반해 평가
    - 작업이 제한된 영역에서 중요
  - **전역적 사실 일관성(global factual consistency)**
    - 출력을 공개된 지식에 기반해 평가
    - 챗봇, 사실 확인, 시장 조사 등 광범위한 작업에서 중요
  - 명시적 사실에 대해 확인하는 것이 훨씬 쉬움
  - 만약 컨텍스트가 주어지지 않으면, 신뢰할 만한 출처를 찾고 사실을 도출한 다음 평가해야 함
  - 가장 어려운 부분은 무엇이 사실인지 판단하는 것
- 사실 관계 판단 기법
  - 자체 검증(self-verification)
    - SelfCheckGPT
    - 모델이 서로 일치하지 않는 여러 출력을 생성할 때 원래 출력이 환각일 가능성이 높다는 가정을 따름
    - 평가할 응답 R이 주어지면 N개의 새로운 응답을 생성하고, R이 새로운 응답과 얼마나 일치하는지 측정
    - 효과적이지만 응답 평가를 위해 많은 AI 질의가 필요
  - 지식 강화 검증(knowledge-augmented verification)
    - 구글 딥마인드의 증강 사실성 평가기(Search-Augmented Factuality Evaluator, SAFE)
    - 검색 엔진 결과를 활용해 응답을 검증
- 주어진 컨텍스트와의 일치 여부는 NLP 작업인 **텍스트 함의(textual entailment)** 로 표현 가능
  - **자연어 추론(Natural Language Interface, NLI)** 이라고도 불림
  - 두 진술 간의 관계를 파악하는 작업
  - 전제(premise)와 가설(hypothesis)이 주어지면, 전제가 참일 때 가설이 참인지, 거짓인지, 혹은 중립인지 판단
  - 함의: 가설은 전제로부터 추론 가능 → 사실 일관성
  - 모순: 가설은 전제와 모순됨 → 사실 비일관성
  - 중립: 전제는 가설을 함의하지도, 모순되지도 않음 → 일관성 판단 불가능
- 일반적인 AI 평가자를 사용하는 대신 사실 일관성 예측에 특화된 평가 모델을 학습시킬 수도 있음
  - (전제, 가설) 쌍을 입력받아 함의, 모순, 중립 등 미리 정의된 클래스 중 하나를 출력하는 방식
- 사실 일관성은 검색 증강 생성(Retrieval-Augmented Generation, RAG) 시스템의 핵심 평가 기준
  - 질의가 주어지면 RAG 시스템은 모델 컨텍스트 보완을 위해 외부 DB에서 관련 정보 검색
  - 생성된 응답은 검색된 컨텍스트와 사실 일관성이 있어야 함

**안전성**

- 안전성 분류 방법은 OpenAI의 콘탠츠 중재 엔드포인트와 Meta의 Llama Guard에 정의된 분류법을 참고할 것
- AI 모델이 생성할 수 있는 위험 콘텐츠들
  - 욕설과 노골적인 내용들
  - 파괴적 행동을 부추기는 유해한 추천 및 지침
  - 차별적 행동을 포함하는 혐오 발언
  - 위협과 자세한 묘사를 포함하는 폭력
  - 고정관념
  - 정치적, 종교적 이데올로기에 대한 편향성
- 유해성을 측정하는 일반적인 벤치마크들
  - RealToxicityPrompts
  - BOLD

### 지시 수행 능력(instruction-following capability)

"이 모델이 주어진 지시를 얼마나 잘 따르는가?"

- 지시 수행 능력은 JSON 형식, 정규 표현식(regex) 등 구조화된 출력이 필요한 애플리케이션에서 필수적
- 구조화된 출력 생성 뿐만 아니라 다양한 지시사항에 따를 수 있어야 함

**지시 수행 기준**

- 대표적인 벤치마크
  - IFEval
    - 구글의 벤치마크
    - 모델이 예상된 형식에 맞는 출력을 생성할 수 있는지에 초점을 맞춤
    - 키워드 포함, 길이 제한, 글머리 기호 개수, JSON 형식 등
  - INFOBench
    - IFEval에 더해서 내용 제약, 언어 지침, 문체 규칙 등을 평가할 수 있음
    - 예/아니오 로 답할 수 있는 질의 형태의 기준 목록을 만들어서 활용하는 방식
- 위 2개의 벤치마크는 모델들이 지시를 얼마나 잘 따르는지 감을 잡기엔 좋지만 실제 세계의 지시 중 많은 것을 놓치고 있음
  - 지표가 좋다고 항상 좋은 성능을 내지는 못할 수도 있음

**역할 연기**

- 모델에게 가상 캐릭터나 페르소나를 가정하도록 요청하는 것
- 역할 연기의 2가지 목적 수행
  - 사용자가 상호 작용할 수 있는 캐릭터 연기 (게임, 대화형 스토리텔링 등 엔터테인먼트 용도가 많음)
  - 모델 출력 품질 향상을 위해 프롬프트 엔지니어링 기법으로 역할 연기를 활용
- LMSYS의 챗봇 연구 결과에서 가장 많은 활용 사례 순위 8위에 들었음
  - 1위: 소프트웨어 오류 및 해결 방안 논의 (10.43%)
  - 2위: AI 도구, 소프트웨어 설계, 프로그래밍 관련 문의 (7.29%)
  - 3위: 지리, 여행, 세계 문화 관련 문의 (6.96%)
  - 4위: 텍스트 요약 및 상세 설명 요청 (6.83%)
  - 5위: 비즈니스 전략 및 제품 개발/개선 (6.36%)
  - 6위: 파이썬 코딩 지원 및 예제 요청 (6.14%)
  - 7위: 텍스트 번역, 재작성, 요약 요청 (6.06%)
  - 8위: **대화 속 다양한 인물 역할 수행 (역할 연기) (5.83%)**
  - 9위: 노골적이고 선정적인 이야기 창작 요청 (5.71%)
  - 10위: 제시된 지문에 기반한 질의응답 (5.59%)
- 역할 연기 능력을 평가하는 벤치마크들
  - RoleLLM
    - 유사도 점수와 AI 평가자를 사용하여 평가
  - CharacterEval
    - 사람 평가자를 사용해 역할 연기를 5점 척도로 평가하는 보상 모델을 학습시켰음
- 모델이 캐릭터를 유지하는지, 말투와 지식은 어떤지 평가해야 함

### 비용과 지연 시간

- 파레토 최적화(Pareto optimization)
  - 타협 가능한 수준을 명확히 정하고 최적화하는 방식
  - 비용과 지연 시간은 종종 상충 관계에 있기 때문
- 파운데이션 모델의 지연 시간 지표
  - 첫 토큰까지 걸리는 시간
  - 토큰당 시간
  - 토큰 간 시간
  - 질의당 시간
- 지연 시간은 기반 모델뿐 아니라 프롬프트 및 샘플링 변수의 영향도 받음
- 비용 문제는 입출력 토큰 수와 직결됨
- 자체 호스팅 모델의 경우 주요 비용은 컴퓨팅 자원
  - GPU는 보통 16-80GB 메모리로 갖추고 있으며, 이를 통해 70억-650억개의 파라미터를 가짐
  - 자체 호스팅의 경우, 규모가 커질수록 토큰당 비용을 크게 줄일 수 있음
  - 따라서 기업은 규모에 따라 모델 API를 사용할지, 자체 호스팅 모델을 사용할지 결정해야 함

## 4.2 모델 선택

- 어떤 모델이 가장 중요한가 X / 애플리케이션에 가장 적합한 모델인가 O
- 애플리케이션 개발 도중 여러 모델 조정 기법을 사용하면서 모델을 반복적으로 선택해야 함
- 모델 조정 기법의 선택 과정 2단계
  - 달성 가능한 최고 성능 파악하기
  - 비용-성능 축에 모델을 배치하고 투자 대비 최고 성능의 모델 선택
- 하지만 실제 모델 선택 과정은 훨씬 복잡함

### 모델 선택 과정

- 모델의 속성 구별
  - **하드 속성**
    - 변경이 불가능하거나, 변경하는 것이 비현실적
    - 모델 제공업체의 결정(라이선스, 학습 데이터, 모델 크기) 또는 자체 정책(개인 정보 보호, 제어)의 결과인 경우가 많음
    - 변경이 어려운 부분이기 때문에 일부 활용 사례에서는 사용 가능한 모델의 풀이 크게 줄어들 수도 있음
  - **소프트 속성**
    - 기꺼이 변경할 수 있음
    - 정확도, 유해성, 사실 일관성과 같이 개선 가능한 속성들
    - 속성을 얼마나 개선할 수 있는지 추정할 때 낙관적 태도와 현실적 태도 사이에서 균형을 맞추기 까다로울 수 있음
  - 하드 속성과 소프트 속성의 정의는 모델 활용 사례에 따라 다름
    - 자체 호스팅인지 아닌지에 따라 또 달라짐
- 전반적인 평가 과정 4단계 (순환적)
  - 하드 속성 기준으로 걸러내기
  - 공개된 정보를 활용해 실험해 볼 가장 유망한 모델들 추려내기
  - 자체 평가 파이프라인을 통해 최적의 모델 찾기
  - 운영 환경에서 지속적으로 모니터링

### 모델 자체 개발 vs 상용 모델 구매

- 대부분의 기업이 처음부터 파운데이션 모델을 개발하지는 않을 것
- 그러므로 결국 '상용 모델 API 사용 vs 오픈 소스 모델 직접 호스팅' 으로 좁혀짐

**오픈 소스, 오픈 가중치, 모델 라이선스**

- 오픈 소스 모델(open-source model)의 의미
  - 원래는 그저 다운로드해서 사용할 수 있는 모든 모델을 가리키는 용도
  - 하지만 일부는 학습 데이터까지 공개되었을 때 오픈으로 봐야한다고 주장함
- 오픈 데이터 사용 시 모델 아키텍처, 학습 과정 등을 조정하면서 더 유연하게 모델을 활용할 수 있음
  - 또한 모델을 이해하기 더 쉽게 해줌
- 데이터 공개 여부를 구분하는 용어
  - 오픈 웨이트(open-weight): 데이터 없이 공개된 모델
    - 오픈 소스 모델의 대다수가 여기에 해당함
    - 공개적인 검증 과정에서 비판받거나 잠재적인 소송의 대상이 될 수 있기 때문
  - 오픈 모델(open-model): 데이터와 함께 공개된 모델
- 라이선스는 오픈 소스 모델의 또 다른 중요한 속성
  - 파운데이션 모델 이전에도 MIT, 아파치 2.0, GNU GPL, BSD 등 수많은 라이선스들이 있었는데, 오픈 소스 모델은 이를 더 복잡하게 만들었음
  - 많은 모델이 자체적인 고유 라이선수로 출시되었음
  - 결국 라이선스를 고를 때 이런 질문들을 확인해야 함
    - 라이선스가 상업적 사용을 허용하는가?
    - 상업적 사용을 허용한다면, 어떤 제한이 있는가?
    - 라이선스가 모델의 출력을 다른 모델의 학습이나 개선에 사용하는 것을 허용하는가?

**오픈 소스 모델과 모델 API 비교**

- 추론 서비스: 모델을 호스팅하고 사용자 질의를 받아서, 모델을 실행해 질의에 대한 응답을 생성하고, 사용자에게 응답을 반환하는 서비스
  - 큰 모델을 위한 추론 서비스 개발은 쉽지 않기 때문에, 많은 기업들이 직접 개발하기를 원치 않음
  - 이로 인해 오픈 소스 모델 위에서 동작하는 많은 서드파티 추론 및 파인튜닝 서비스가 생겨났음
- 모델 API: 사용자가 상호작용하는 인터페이스
  - AWS, Azure, GCP 등 클라우드 제공업체들 모두 인기 있는 오픈 소스 모델에 대한 API 접근 제공
- 모델 개발자는 모델 개발 후 오픈 소스로 공개하거나, API를 통한 접근을 허용하거나, 둘 다 지원할 수도 있음
- 많은 모델 개발자들이 모델 서비스 제공업체
- 일반적으로 성능이 낮은 모델은 오픈 소스로 공개하고, 성능이 좋은 모델은 유료 API 형태로 제공하거나 자사 서비스 기능으로 사용
- 상용 모델은 개발사가 제공하는 독점 API를 통해서만 접근 가능 / 반면 오픈 소스 모델은 여러 API 제공업체에서 지원
- 모델을 직접 호스팅할지, 모델 API를 사용할지 결정할 때 고려해야 할 7가지
  - **데이터 프라이버시**
    - 삼성이 실수로 ChatGPT를 통해 회사 기밀을 유출시킨 사례가 있음
    - 일부 국가는 특정 데이터를 국경 밖으로 전송하는 것을 금지하는 법률도 있음
    - AI 모델은 사용자로부터 학습 샘플을 받아서 기억할 수 있으며, 이는 유출되거나 악용될 수도 있음
  - **데이터 계보(data lineage)와 저작권**
    - 대부분의 모델은 어떤 데이터로 학습되었는지를 투명하게 공개하지 않음
    - AI 관련 지적재산권법은 계속 진화하고 있음
    - AI 응용 프로그램의 특허 가능성은 혁신에 대한 사람의 기여도에 달려 있음
    - 저작권이 있는 데이터로 학습한 모델을 써서 제품을 만들었을 때 해당 제품의 지적재산권을 지킬 수 있을지도 불분명
    - 이와 같은 우려 때문에 많은 기업을 상용 모델을 선택하게 됨
  - **성능**
    - 다양한 벤치마크에서 오픈 소스 모델과 독점 모델 간의 격차가 많이 줄어들고 있는 추세
    - 하지만 근미래에도 여전히 최고 성능의 오픈 소스 모델이 최고 성능의 독점 모델을 따라잡지는 못할 것
  - **기능성**
    - 모델에 필요한 기능들
      - 확장성: 추론 서비스가 원하는 지연 시간 및 비용을 유지하면서 애플리케이션 트래픽을 감당
      - 함수 호출: RAG와 에이전트 활용 사례에 필요한 외부 도구 사용
      - 출력 구조: JSON 등의 구조화된 출력
      - 출력 가드레일: 유해하거나 부적절한 콘텐츠 생성 방지
    - 위 기능들은 구현이 어렵고 시간이 많이 걸리기 때문에 API 제공 업체를 이용하게 됨
      - 하지만 모델 API를 사용하면 제공되는 기능만 사용할 수 있게 됨
      - 많이 활용하게 되는 기능들은 분류 작업, 평가, logprobs인데, 모델 제공업체는 logprobs를 비공개하거나 제한적임
      - 또한 프롬프트 엔지니어링만 가능하고 파인튜닝은 할 수 없는 경우가 많음
  - **비용 (API 비용 vs 엔지니어링 비용)**
    - 사용량이 많아져서 API 비용이 감당하기 어려워졌다면 직접 모델을 호스팅하는 방안을 검토하게 됨
    - 엔지니어링에는 모델 최적화, 추론 서비스 확장 및 유지, 가드레일 제공 등 많은 비용이 발생
    - 오픈이든 독점이든 상관 없도록 표준 API를 따르는 모델이 필요
    - 그래서 많은 모델 개발자들은 가장 인기 있는 모델의 API를 모방하는 방식으로 모델을 출시
  - **제어, 접근성, 투명성**
    - 오픈 소스 모델은 커스터마이징 가능성 때문에 관심을 끌고 있음
    - 독점 모델은 잠재적 소송으로부터의 보호를 위해 검열이 과도해질 우려가 있음
    - 상용 모델은 모델 접근 권한을 갑자기 잃게 될 우려도 있으며, 갑자기 프롬프트가 예상대로 동작하지 않게 될 수 있음
  - **온디바이스 배포**
    - 디바이스에서 모델을 실행하는 경우 서드파티 API 사용 불가
    - 실제로 모델을 로컬에서 실행하는 게 필요한 경우가 많음

|  | 모델 API | 모델 자체 호스팅 |
| --- | --- | --- |
| **데이터 프라이버시** | 모델 제공자에게 기밀 유출 우려 | 데이터를 외부로 보낼 필요는 없지만, 데이터 계보 및 학습 데이터 저작권 검증이 어려움 |
| **성능** | 성능이 좋다면 비공개일 것 | 최고의 오픈 소스 모델도 상용 모델에 비해 뒤처질 것 |
| **기능** | 확장 지원 가능성이 높지만, logprobs를 노출해줄 가능성이 낮음 | 함수 호출 및 구조화된 출력이 제한적 |
| **비용** | API 비용 | 모델 최적화, 호스팅, 유지보수 등에 필요한 엔지니어링 비용 |
| **제어, 접근성, 투명성** | 모델 변경 및 버전 관리의 투명성 부족 | 모델에 대한 유지보수 책임 존재 |
| **온디바이스 배포** | 인터넷에 연결되지 않았다면 사용 불가 | 실제로 실행하기 까다로울 수 있음 |

### 공개 벤치마크 탐색하기

- AI 활용 사례의 증가에 따라 벤치마크의 수도 빠르게 늘어나고 있음
- AI 모델이 개선됨에 따라 새로운 벤치마크의 도입이 더 필요해짐
- **평가 하네스(evaluation harness)** : 여러 벤치마크를 실행하고 결과를 수집하는 도구
  - EleutherAI - [LM Evaluation Harness](https://github.com/EleutherAI/lm-evaluation-harness)
  - OpenAI - [Evals](https://github.com/openai/evals)

**벤치마크 선택 및 집계**

- 벤치마크 결과를 집계해서 모델의 순위를 매기면 리더보드가 만들어짐
- 리더보드를 만들 때 고려해야 할 2가지
  - 리더보드에 어떤 벤치마크를 포함할 것인지
  - 벤치마크 결과를 어떻게 집계해서 모델 순위를 매길 것인지
- **공개 리더보드**
  - 자신만의 리더보드를 만들기 위해 공개 리더보드를 살펴보는 것도 좋음
  - 컴퓨팅 자원은 유한하기 때문에 모든 측면을 다루지는 못함
  - 허깅페이스의 6개 벤치마크 예시
    - ARC-C: 초등학교 수준의 과학 문제 풀이 능력 측정
    - MMLU: 초등 수학, 미국 역사, 컴퓨터 과학, 법학 등 57개 과목에서 지식 및 추론 능력 측정
    - HellaSwag: 이야기 및 영상에서 문장이나 장면의 완성을 예측하는 능력 측정, 상식과 일상에 대한 이해 테스트
    - TruthfulQA: 모델이 사실에 기반한 답변을 생성하는지 측정
    - WinoGrande: 대명사 참조 해결 능력 측정
    - GSM-8K: 초등학교 수준의 수학 문제 풀이 능력 측정
  - 하지만 공개 리더보드들도 서로 다른 벤치마크들을 사용하기 때문에 모델 간 순위를 공식적으로 확인하는 용도는 아님
  - 벤치마크를 선정할 때 중요한 점은 각 벤치마크들이 상관관계가 높지 않도록 하는 것
  - 평균 점수 vs 평균 승률
    - 평균 점수: 모든 벤치마크의 점수를 동등하게 취급하며, 같은 가중치를 둠
    - 평균 승률: 모델이 다른 모델에 비해 더 좋은 점수를 얻는 비율을 시나리오별로 평균을 냄
  - 공개 리더보드는 전반적인 성능을 파악하는 데 유용하지만, 리더보드가 어떤 능력을 측정하려 하는지 이해하는 게 중요
- **공개 벤치마크로 맞춤형 리더보드 만들기**
  - 우리 애플리케이션에 필요한 핵심적인 벤치마크 목록을 수집해야 함
  - 또한 벤치마크를 얼마나 신뢰할 수 있는지도 평가해야 함
  - 벤치마크들이 서로 다른 점수 혹은 척도를 사용한다는 것도 문제
- **공개 벤치마크의 데이터 오염(data contamination)**
  - 모델이 평가되는 벤치마크 데이터로 학습한 경우 발생
  - 데이터 오염이 일어나는 방식
    - 높은 점수를 얻기 위해 의도적으로 학습한 경우도 있겠지만, 대부분은 의도치 않게 발생함
    - 오늘날 많은 모델이 인터넷에서 수집한 데이터로 학습하기 때문
    - 이는 기존 벤치마크가 포화 상태가 되고, 새로운 벤치마크에 대한 니즈를 느끼는 이유가 됨
  - 하지만 고품질 벤치마크 데이터로 인한 오염이 오히려 최고의 모델을 만들어줄 수도 있음
  - 데이터 오염 다루기
    - 오염 감지 및 데이터 정화가 필요
    - **n-gram 중복**
      - 학습 데이터에 평가 샘플의 특정 시퀀스가 포함되어 있는지 확인
      - 정확도는 높지만 시간 및 비용이 많이 소모될 수 있음
    - **퍼플렉시티**
      - 퍼플렉시티가 낮다면 예측하기 쉽다는 뜻이므로, 지나치게 낮은 경우 이미 학습했을 가능성이 높음
      - 정확도는 떨어지지만 훨씬 적은 자원으로 가능
    - 오염된 평가 예시를 아예 제거하는 것은 불가능해졌음

**⇒ 공개 벤치마크는 나쁜 모델을 걸러내는 용도, 자체 평가 파이프라인으로 용도에 가장 잘 맞는 모델을 찾아내는 노력이 필요하다**

## 4.3 평가 파이프라인(evaluation pipeline) 설계하기

### 1단계: 시스템의 모든 구성 요소 평가하기

- 각 구성 요소의 중간 출력과 E2E 출력은 독립적으로 평가되어야 함
  - 가능하다면 애플리케이션을 작업별로, 또한 턴(여러 단계 및 메시지)별로 평가해야 함
- 구성 요소를 독립적으로 평가함으로써 시스템이 정확히 어디서 실패하는지 파악 가능
- **turn-based evaluation**
  - 각 출력물의 품질을 평가
- **task-based evaluation**
  - 시스템이 작업을 완료했는지를 평가
  - 작업을 완료하는 데 몇 번의 대화가 필요했는가?
  - 작업 간 경계를 구분하기 힘들 때가 있음

### 2단계: 평가 가이드라인(evaluation guideline) 만들기

- 평가 파이프라인에서 가장 중요한 단계
- 애플리케이션이 해야 할 일 뿐만 아니라 하면 안 되는 일도 정의하는 것이 중요

**평가 기준 정의하기**

- 어떤 출력이 '좋은 출력'인지 정의하는 것이 중요
- 정확한 응답이 항상 좋은 응답은 아님
- 좋은 응답의 3가지 기준
  - **관련성**: 응답이 사용자 질의와 관련 있음
  - **사실 일관성**: 응답이 컨텍스트와 사실적으로 일치
  - **안전성**: 응답이 유해하지 않음

**예시와 함께 평가 기준표 만들기**

- 이진법, 1-5, 0-1 등 다양한 척도를 고민해봐야 함
- 이진 평가 시스템의 경우, 0은 사실 불일치, 1은 사실 일치
- 3가지 값을 사용해서 -1은 모순, 0은 중립, 1은 함의로 정의할 수도 있음
- 어떤 평가 시스템을 사용할지는 데이터 및 필요에 따라 달라짐
- 팀과 함께 모호한 기준을 없애기 위해 많은 논의가 필요

**평가 지표를 비즈니스 지표와 연결하기**

- 애플리케이션의 지표는 비즈니스 문제의 컨텍스트에서 고려되어야 함
- 평가 지표가 비즈니스 지표에 미치는 영향을 이해하고 계획이 수립되어야 할 것
- 유용성 임계값 설정에도 도움이 됨
- 많은 애플리케이션들이 DAU, WAU, MAU 등의 고착화(stickiness) 지표에 초점을 맞춤
- 일부는 월간 대화 수, 방문 기간 등 참여 지표(engagement metrics)에 초점을 맞춤
- 어떤 지표를 우선시할지에 대해서도 균형 있는 선택을 해야 함

### 3단계: 평가 방법과 데이터 정의하기

**평가 방법 선택하기**

- AI 평가자: 관련성, 의미적 유사도, 사실 일관성을 측정하는 데 유용
- 동일한 기준에 여러 평가 방법을 혼합할 수도 있음
- logprobs를 사용 가능하다면 사용하는 것이 좋음
  - 모델이 생성된 토큰을 얼마나 확신하는지 측정하는 데 사용 → 분류에 특히 유용
  - 생성된 텍스트에 대한 모델의 퍼플렉시티를 평가할 때도 사용 가능 → 유창함, 사실 일관성 평가에 사용
- 가능한 한 자동 지표를 사용하되, 필요하다면 사람 평가를 병행해야 함
- 평가 방법들은 개발 단계 뿐만 아니라 운영 환경에서도 사용될 수 있도록 해야 함
  - 운영 환경에서는 실험 단계의 참조 데이터는 사용 못하겠지만, 실제 사용자의 피드백을 사용할 수 있음

**평가 데이터 주석 달기**

- 애플리케이션 평가를 위해 주석이 달린 예시 세트를 선별해야 함
- 턴 기반 평가, 작업 기반 평가 모두 주석이 달린 예시 세트가 필요
- 가능하다면 운영 환경의 데이터 사용
  - 애플리케이션에 사용할 수 있는 자연스러운 레이블이면 좋음
  - 혹은 사람 또는 AI를 사용해 데이터 레이블 지정 가능
- 이 단계의 성공 여부 또한 평가 기준표의 명확성에 달려 있음
- 평가를 위한 주석 가이드라인을 파인튜닝을 위한 지시 데이터 생성에 재사용할 수도 있음
- 데이터 슬라이스
  - 시스템에 대한 세분화된 이해를 도움
  - 데이터를 하위 집합으로 나누고 각 하위 집합에 대한 시스템 성능 개별 평가
  - 달성 가능한 목표들
    - **편향 축소**
    - **디버깅**
    - **애플리케이션 개선 영역 발굴**
    - **심슨의 역설 회피**
      - 모델 A가 집계된 데이터 상에서는 모델 B보다 성능이 좋지만, 데이터의 모든 하위 집합에서는 모델 B가 더 성능이 좋게 나타나는 현상
  - 데이터 슬라이스 기준들 예시
    - 등급: 유료 사용자, 무료 사용자 등
    - 트래픽 출처: 모바일, 웹 등
    - 사용량
- 일반적으로 평가 세트의 예시 수는 평가 결과가 신뢰 가능할 만큼 크되, 비용을 절감할 수 있을 만큼 작아야 함
- 평가 부트스트랩을 여러 번 실행했을 때, 만약 결과가 크게 달라진다면 더 큰 평가 세트가 필요하다는 뜻
- 평가 결과는 시스템을 독립적으로 평가할 뿐만 아니라 시스템을 비교하는 데도 사용됨

OpenAI의 평가 샘플 수 추정치 제안

| 감지할 차이 | 95% 신뢰도에 필요한 표본 크기 |
| :---: | :---: |
| 30% | ~ 10 |
| 10% | ~ 100 |
| 3% | ~ 1,000 |
| 1% | ~ 10,000 |

**평가 파이프라인 평가하기**

- 신뢰성 확보 및 더 효율적인 방법 탐색에 도움이 됨
- 평가 파이프라인의 품질에 대해 고려해야 할 사항들
  - **올바른 신호를 제공하고 있는가?**
    - 더 나은 응답이 실제로 더 높은 점수를 받는가?
    - 더 나은 평가가 더 나은 비즈니스 결과로 이어지는가?
  - **얼마나 신뢰할 수 있는가?**
    - 여러 번 실행할 때 결과가 달라지지는 않는가?
    - 재현성을 높이고 분산을 줄여야 함
  - **지표 간 상관관계는 어떠한가?**
    - 상관관계가 높다면 둘 다 사용할 필요가 없음
  - **애플리케이션에 얼마나 많은 비용과 지연 시간을 추가하는가?**

**반복**

- 요구사항 및 사용자 행동의 변화에 따라 평가 기준도 진화하고 평가 파이프라인도 반복적으로 개선되어야 함
- 이를 반복하는 와중에도 평가 파이프라인은 일정 수준의 일관성을 유지해야 함
- 평가 파이프라인을 반복적으로 개선하면서 적절한 실험 추적을 수행해야 함
  - 평가 기준표, AI 평가자에 사용된 프롬프트 및 샘플링 구성 등 모든 변수를 기록해둘 것
