# 5. 프롬프트 엔지니어링

- 프롬프트 엔지니어링: 모델이 원하는 결과를 생성하도록 지시를 다듬는, 가장 쉽고 일반적인 **모델 조정(model adaptation)** 기법
- 파인튜닝과 달리 모델의 가중치를 변경하지 않고도 모델의 응답을 조정
- 따라서 파인튜닝 같이 더 많은 자원이 필요한 기법으로 넘어가기 전에 프롬프팅을 최대한 활용해 보는 것이 좋음
- 프롬프트 엔지니어링은 별것 아니라고 오해받기 쉬움
  - 하지만 문제를 해결하기 위한 창의적인 접근법이 될 수 있음
  - 체계적인 실험 방법과 평가 과정을 통해 다른 ML 실험처럼 엄중히 수행될 수 있음
- 프롬프트 엔지니어링 = 사람 ↔ AI 커뮤니케이션
- 5장에서는 효과적인 프롬프트 작성법, 프롬프트 공격 방어법을 다룰 것

## 5.1 프롬프트 소개

- 프롬프트가 일반적으로 포함하는 요소들
  - **작업 설명**: 모델이 수행해야 할 일, 역할 및 출력 형식 포함
  - **작업 수행 방법 및 예시**: 모델이 작업을 수행하는 방법 및 예시
  - **작업**: 모델이 수행해야 할 구체적인 작업
- 프롬프트 엔지니어링의 필요도는 모델이 프롬프트 변화에 얼마나 **강건하게(robust)** 반응하는지에 따라 달라짐
  - 프롬프트가 약간만 바뀌어도 응답이 완전히 달라진다면 프롬프트 엔지니어링의 필요성이 커짐
  - 프롬프트를 무작위로 변경하면서 출력의 변경을 확인하면 모델의 강건성 측정 가능
  - 모델이 더 강력해질수록, 모델이 더 강건해짐

### 인컨텍스트 학습(In-Context Learning): 제로샷과 퓨샷

- **인컨텍스트 학습(In-Context Learning, ICL)**
  - 프롬프트를 통해 모델에게 무엇을 해야 할지 가르치는 것
  - 이 방법을 통해 모델은 원래 학습 목적과 다른 작업이더라도 프롬프트 내 행동을 통해 학습할 수 있음
  - 모델이 지속적으로 새로운 정보를 학습할 수 있게 해주므로, 모델이 계속 발전할 수 있게 만들어줌
- **Shot**
  - 프롬프트에 제공된 각 예시를 샷(shot)이라고 부름
  - 모델에게 프롬프트의 예시들을 통해 학습하도록 가르치는 방식을 **퓨샷(few-shot)** 학습이라고 부름
  - 만약 5개의 예시가 있다면 5-shot 학습
  - 만약 예시가 전혀 제공되지 않으면 **제로샷(zero-shot)** 학습
  - 실험을 통해 애플리케이션에 필요한 적정 샷 갯수를 찾아야 함
  - 샷 갯수는 모델의 최대 컨텍스트 길이에 의해 제한되며, 샷 갯수가 많을수록 프롬프트가 길어져 추론 비용이 증가
- 용어 모호성: 프롬프트와 컨텍스트
  - 아직도 용어 통합이 안 되어 있으며 논의가 이어지고 있음
  - 프롬프트: 모델에 입력되는 전체 내용
  - 컨텍스트: 모델이 주어진 작업을 수행할 수 있도록 모델에 제공되는 정보
- 프롬프트 엔지니어링은 원하는 프로그램을 활성화할 수 있는 적절한 프롬프트를 찾는 과정이라고 할 수 있음

### 시스템 프롬프트와 사용자 프롬프트

- 대부분의 모델 API는 프롬프트를 시스템 프롬프트와 사용자 프롬프트로 나눌 수 있는 옵션 제공
- 시스템 프롬프트: 작업 설명
- 사용자 프롬프트: 작업 자체
- ChatGPT를 비롯한 거의 모든 생성형 AI 애플리케이션에는 시스템 프롬프트가 존재
- 일반적으로 애플리케이션 개발자가 제공하는 지시는 시스템 프롬프트로, 사용자가 제공하는 지시는 사용자 프롬프트로 구성됨
- 시스템 프롬프트와 사용자 프롬프트가 주어지면 모델은 일반적으로 템플릿에 맞게 하나의 프롬프트로 결합
- 모델별로 서로 다른 채팅 템플릿을 사용하며, 같은 모델이라도 버전에 따라 템플릿이 달라질 수 있음
- 시스템 프롬프트가 사용자 프롬프트보다 성능을 더 향상시키는 경향이 있음
  - 시스템 프롬프트가 최종 프롬프트의 맨 앞에 위치하기 때문에
  - 모델이 시스템 프롬프트를 더 중요하게 인식하도록 학습되었을 수 있음

### 컨텍스트 길이와 컨텍스트 효율성

- 모델 제공업체들은 경쟁적으로 컨텍스트 길이 확장을 해오고 있으며, Gemini-1.5 Pro는 2M(200만) 토큰의 컨텍스트 길이를 지원
- 연구에 따르면 모델은 프롬프트의 중간보다 시작과 끝에 제시된 지시를 더 잘 이해함
- **NIAH(Needle-in-a-Haystack)** 테스트
  - 프롬프트의 다양한 부분에 대한 효과를 평가하는 방법
  - 무작위 정보(바늘)를 프롬프트(건초더미)의 다양한 위치에 삽입하고 모델에게 이를 찾도록 요청
  - Liu 등의 연구 결과, 모델들이 중간보다 시작과 끝에 가까울 때 정보를 훨씬 더 잘 찾아내는 것으로 나타났음
- 만약 모델 성능이 컨텍스트 길이가 늘어남에 따라 더 나빠진다면, 프롬프트를 더 간결하게 만드는 방법을 고려해야 함
- 프롬프트의 핵심 구성 요소들
  - 시스템 프롬프트
  - 사용자 프롬프트
  - 예시
  - 컨텍스트

## 5.2 프롬프트 엔지니어링 모범 사례

- OpenAI, Anthropic, Meta, Google 등 주요 LLM 제공업체들의 프롬프트 엔지니어링 튜토리얼과, 모범 사례들을 소개할 것
- 모델마다 특별히 잘 반응하는 프롬프트 작성법이 있으므로, 해당 모델에 최적화된 프롬프트 엔지니어링 가이드를 따르는 것이 좋음

### 명확하고 명시적인 지시 작성하기

**모델이 해야 할 일을 모호함 없이 설명하기**

- 점수 체계 같은 것이 있다면 명확히 설명할 것

**모델에게 특정 페르소나 부여하기**

- 모델은 페르소나에 따라 같은 질문에도 다른 대답을 할 수 있음

**예시 제공하기**

- 모델이 어떻게 응답해야 할지에 대한 모호함을 줄여줄 수 있음
- 입력 토큰 길이가 걱정된다면 더 적은 토큰을 사용하는 예시 형식을 선택해야 함

**출력 형식 지정하기**

- 비용 및 지연 시간이 우려된다면 모델이 간결하게 응답하도록 하는 것이 좋음
- JSON 등 형식이 필요하다면 지정할 것
- 분류 등 구조화된 출력이 필요하면 마커를 활용할 수 있을 것

### 충분한 컨텍스트 제공하기

- 충분한 컨텍스트는 모델의 성능 향상에 도움이 됨
- 컨텍스트는 또한 환각 현상을 줄이는 데도 도움이 됨
  - 필요한 정보가 없을 때 신뢰성이 떨어지는 내부 지식에 의존하게 되어서 환각이 발생할 수 있음
- **컨텍스트 구성(context construction)**
  - 주어진 질의에 필요한 컨텍스트를 수집하는 과정을 뜻함
  - 컨텍스트 구성 도구에는 RAG 파이프라인 등의 데이터 검색과 웹 검색도 포함됨

### 복잡한 작업을 단순한 하위 작업으로 나누기

- 복잡한 작업은 하위 작업들을 가지고, 하위 작업들은 저마다의 고유 프롬프트를 가짐
  - 이러한 하위 작업들은 서로 연결됨
- 각 하위 작업이 얼마나 작아야 하는지는 사용 사례, 성능, 비용, 지연 시간의 균형에 따라 달라짐
  - 최적의 분해와 연결 방식을 찾기 위한 실험이 필요
- 프롬프트 분해의 추가 이점들
  - **모니터링**: 각 하위 작업의 성능을 개별적으로 모니터링 가능
  - **디버깅**: 문제가 발생했을 때 어느 하위 작업에서 문제가 발생했는지 파악하기 쉬움
  - **병렬화**: 여러 하위 작업을 병렬로 처리하여 처리 시간 단축 가능
  - **노력 절감**: 복잡한 프롬프트보다 단순한 프롬프트 작성이 쉬움
- 프롬프트 분해의 단점들
  - 중간 단계가 많아지면 지연 시간이 늘어남
  - 일반적으로 더 많은 모델 질의를 수행하게 되므로 비용이 증가할 수도 있음
    - 하지만 작은 프롬프트가 더 적은 토큰을 사용하게끔 오히려 최적화할 수도 있음

### 모델에게 생각할 시간 주기

- **생각의 사슬(Chain of Thought, CoT)**
  - 모델이 단계별로 생각하도록 명시해서 문제에 더 체계적으로 접근하도록 유도하는 것
  - 생각의 사슬은 환각 현상도 줄여준다는 [발표](https://www.linkedin.com/blog/engineering/generative-ai/musings-on-building-a-generative-ai-product)도 있음
  - 생각의 사슬을 적용하는 가장 간단한 방법: 프롬프트에 "단계별로 생각하세요" 또는 "결정 과정을 설명해 주세요"와 같은 문구를 추가하는 것
    - 단계를 구체적으로 명시하거나, 각 단계에 대한 예시를 제공할 수 있을 것
- **자기 비판(self-critique)**
  - 모델에게 자신의 출력을 검토하도록 요청하는 것
  - 모델이 문제에 대해 비판적으로 생각하도록 유도
- 위 둘은 프롬프트 분해와 마찬가지로 지연 시간을 증가시킬 수도 있음

### 프롬프트 반복하며 개선하기

- 모델들은 저마다의 독특한 특성을 가짐
- 모델을 더 잘 알기 위해 여러 프롬프트를 시도해 보고, 프롬프트 가이드도 읽어보자
  - 모델 플레이그라운드가 있다면 활용해볼 수도 있을 것
- 프롬프트 버전을 관리하고 실험 추적 도구를 사용해서 평가 지표 및 평가 데이터 표준화하기
- 각 프롬프트는 하위 작업보다 전체 시스템 컨텍스트에서 평가할 것

### 프롬프트 엔지니어링 도구 평가하기

- 프롬프트를 직접 만들고 최적화하는 작업은 시간이 많이 걸리고 어려울 수 있음
- 이런 이유로 프롬프트 엔지니어링을 지원하고 자동화하는 다용한 도구가 개발되었음
- 전체 프롬프트 엔지니어링 워크플로 자동화 도구
  - OpenPrompt
  - DSPy
- 프롬프트 최적화 도구는 평가 데이터에서 평가 지표를 최대화하는 프롬프트 혹은 프롬프트 체인을 자동으로 찾아냄
- AI 기반 프롬프트 최적화 도구
  - Promptbreeder
    - 진화 전략을 활용해 프롬프트를 선택적으로 **번식**시키는 방식
    - 기본 프롬프트로 시작해 AI 모델을 통해 다양한 변이(mutator)를 만들어냄
    - 변이 프롬프트 모음은 어떤 방식으로 변형할지 방향을 제시
    - 가장 효과적인 것을 선택해 다시 변이를 만드는 과정 반복
  - TextGrad
- 프롬프트 엔지니어링의 일부분 지원 도구
  - Guidance
  - Outlines
  - Instructor
- 프롬프트 엔지니어링 도구 사용시 주의사항들
  - API 호출이 얼마나 발생하는지 추적 관리할 것
  - 도구 자체에 결함이 있을 가능성도 고려할 것

### 프롬프트 정리 및 버전 관리하기

- 프롬프트를 코드와 분리해서 관리하는 것이 좋음
  - 재사용성: 여러 애플리케이션에서 동일한 프롬프트 재사용 가능
  - 테스트: 코드와 프롬프트를 별도로 테스트 가능
  - 가독성: 프롬프트가 코드와 분리되어 있어 가독성 향상
  - 협업: 여러 개발자가 함께 작업하기 용이
- 각 프롬프트에 메타데이터를 추가해 어떤 용도로 만들어진 것인지 기록해두면 좋음
  - 프롬프트를 파이썬 객체로 감싸서 관리하는 방법도 유용함
- 프롬프트 템플릿에 포함할 수 있을 만한 정보들
  - 모델 엔드포인트 URL
  - 온도, top-p 등 샘플링 파라미터
  - 입력 스키마
  - 예상 출력 스키마
- 여러 도구들은 프롬프트 저장을 위해 `.prompt` 파일 형식을 제안했음
  - [Dotprompt](https://genkit.dev/docs/js/dotprompt/)
  - [Humanloop](https://humanloop.com/docs/getting-started/overview)
  - [Promptfile](https://github.com/promptfile/promptfile)
  - 이 파일들을 Git 저장소에 함께 관리하면 버전 관리 가능
  - 하지만 모든 애플리케이션이 강제로 새 파일을 사용하도록 강제하게 됨
- 대부분의 팀은 각 프롬프트 버전을 명확히 구분 및 관리하는 별도의 프롬프트 카탈로그를 사용
  - 여러 애플리케이션이 각자 다른 버전의 프롬프트를 사용 가능
  - 프롬프트 카탈로그는 각 프롬프트에 대한 메타데이터, 검색 기능을 제공해야 할 것
  - 프롬프트 카탈로그는 의존하고 있는 애플리케이션을 추적하고, 프롬프트 업데이트마다 애플리케이션 담당자에게 알림을 보내면 좋을 것

## 5.3 방어적 프롬프트 엔지니어링

- 3가지 주요 프롬프트 공격 유형들
  - **프롬프트 추출(prompt extraction)** : 프롬프트를 추출하여 애플리케이션을 복제하거나 악용하는 것
  - **탈옥과 프롬프트 주입(jailbreaking and prompt injection)** : 모델이 나쁜 행동을 하도록 유도하는 것
  - **정보 추출(information extraction)** : 모델의 학습 데이터나 컨텍스트를 노출하게 만드는 것
- 프롬프트 공격이 초래할 수 있는 위험들
  - **원격 코드 또는 도구 실행**
  - **데이터 유출**
  - **사회적 해악**
  - **잘못된 정보**
  - **서비스 중단 및 전복**
  - **브랜드 위험**

### 독점 프롬프트와 역 프롬프트 엔지니어링

- 좋은 프롬프트를 공유하기 위한 수많은 저장소들이 생겨남
  - [prompts.chat](https://github.com/f/prompts.chat)
  - [PromptHero](https://prompthero.com/)
  - [Cursor Directory](https://cursor.directory)
  - [PromptBase](https://promptbase.com/)
- 대부분의 팀이 프롬프트를 독점적인 것으로 여기며, 일부는 특허권을 받아야 한다고 주장함
- 기업들이 프롬프트 소유권을 주장함에 따라 역 프롬프트 엔지니어링이 더 유행하게 됨
- **역 프롬프트 엔지니어링(reverse prompt engineering)**
  - 특정 애플리케이션에 사용된 시스템 프롬프트를 추론하는 과정
  - 악위적인 행위자들은 유출된 시스템 프롬프트를 이용해 애플리케이션을 복제하거나 조작할 수 있음
  - 주로 애플리케이션 출력을 분석하거나 모델의 전체 프롬프트를 말하게 하는 방식으로 이루어짐
- 한 AI 연구자는 **"시스템 프롬프트를 작성할 때 언젠가 공개될 것을 가정하고 작성해야 한다"** 라고 언급했음
- 독점 프롬프트를 유지하는 것은 지속적인 관리 및 업데이트가 필요하므로, 오히려 경쟁 우위를 차지하는 것보다 더 부담이 될 수 있음
