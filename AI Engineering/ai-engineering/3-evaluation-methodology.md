# 3. 평가 방법론

- AI 애플리케이션을 개발하는 데 가장 큰 난관은 **평가(evaluation)**
- 평가는 중요하고도 복잡하기 때문에 두 장에 걸쳐 자세히 다룰 것
  - 이번 장에서는 개방형 모델 평가에 사용되는 다양한 방법들, 작동 원리, 한계를 다룰 것
  - 다음 장에서는 이런 방법들을 활용해 애플리케이션에 적합한 모델을 선택하고 평과 과정을 구축하는 방법을 살펴볼 것
- 평가의 목적: 위험을 줄이고 기회를 발견하는 것
  - 시스템의 취약점을 파악하고 그에 맞춰 평가 설계를 진행해서 시스템의 견고함을 확보해야 함
- 많은 파운데이션 모델은 언어 모델이므로, 이번 장에서는 **교차 엔트로피**와 **퍼플렉시티** 등의 언어 모델 평가 지표도 살펴볼 것
- 파운데이션 모델은 개방형이라서 평가가 더욱 어려움
  - 여전히 사람 평가자의 역할이 필수적
  - 하지만 이는 비용과 시간이 많이 들기 때문에 평가 과정을 자동화하는 것도 중요
  - 이 책에서는 정확한 평가와 주관적 평가를 모두 포함하는 **자동 평가**에 초점을 둘 것
- 주관적 평가는 AI 응답을 AI가 평가하는 접근 방식
  - AI 평가자가 어떤 모델과 프롬프트를 사용하느냐에 따라 점수가 달라지므로 주관적
  - 아직은 고민할 만한 포인트가 많다

## 3.1 파운데이션 모델 평가의 어려움

*ML 모델도 평가가 어려웠는데, 파운데이션 모델이 등장하면서 평가가 더 어려워졌다.*

파운데이션 모델이 평가가 어려운 이유

1. AI 모델이 똑똑해질수록 평가는 더 어려워짐
2. 파운데이션 모델의 개방적 특성으로 인해, 정답을 기준으로 성능을 평가하는 방식이 유효하지 않음
3. 대부분의 파운데이션 모델은 블랙박스로 취급되므로, 모델의 출력 결과만 보고 평가할 수밖에 없음

평가 벤치마크들의 부적절성

- AI가 발전하면 벤치마크도 이에 맞춰 계속 진화해야 함
- 모델이 벤치마크에서 완벽한 점수를 달성했다면 해당 벤치마크는 더 이상 재활용할 수 없음

범용 모델의 평가 범위 확장

- 작업 특화 모델에서는 평가할 작업이 명확하지만, 범용 모델은 평가할 작업이 많을 뿐만 아니라 새로운 작업을 발견하는 것도 포함해야 함
- 이제 평가는 AI의 잠재력과 한계를 탐구하는 책임까지 더해진 셈

AI 엔지니어링 파이프라인의 다른 부분에 비해 관심이 뒤처져 있음

- 딥마인드 Balduzzi 논문 - *"알고리즘 개발에 비해 평가 개발은 체계적인 관심을 거의 받지 못했다."*
- GitHub AI 레포지토리들의 카테고리별 인기 순위
  - 1위: 모델링 및 학습
  - 2위: AI 오케스트레이션
  - 3위: 추론 최적화
  - 4위: 프롬프트 엔지니어링
  - 5위: AI 인터페이스
  - 6위: 에이전트
  - 7위: **평가**
- 불충분한 투자는 불충분한 인프라로 이어지고 체계적인 평가를 수행하기 어렵게 만듦

## 3.2 언어 모델링 지표 이해하기

- 파운데이션 모델의 근본은 언어 모델이므로, 언어 모델링 지표를 이해하는 것이 도움이 될 것
- 대부분의 자기회귀 언어 모델은 교차 엔트로피나 퍼플렉시티를 사용해 학습됨
- 문자당 비트(BPC), 바이트당 비트(BPB)는 교차 엔트로피의 변형된 형태
- 위 4개 지표는 서로 밀접하게 연관되어 있고, 하나의 값으로 나머지 셋을 계산할 수 있음
- 3.2는 수학적 내용이 많은데, 지표 해석 방법에만 집중해도 좋음

### 엔트로피

- 엔트로피는 토큰이 평균적으로 얼마나 많은 정보를 담고 있는지 측정함
- 엔트로피가 높을수록 토큰이 더 많은 정보를 담고 있으며, 토큰 표현을 위해 더 많은 비트가 필요함
- 엔트로피가 낮으면 다음에 올 것을 예측하기 더 쉬워짐

### 교차 엔트로피

- 교차 엔트로피는 언어 모델이 데이터셋의 내용을 얼마나 예측하기 어려워하는지를 보여주는 지표
- 교차 엔트로피가 달라지는 2가지 요인
  - 학습 데이터의 예측 가능성 (학습 데이터의 엔트로피)
  - 언어 모델이 파악한 분포와 학습 데이터의 실제 분포와의 차이
- $H(P,Q) = H(P) + D_{KL}(P||Q)$
  - $H(P,Q)$: 교차 엔트로피
  - $H(P)$: 학습 데이터 엔트로피
  - $D_{KL}(P||Q)$: 쿨백-라이블러 발산으로 측정한 두 분포의 차이
- 언어 모델은 교차 엔트로피를 최소화하도록 학습됨
- 만약 언어 모델이 학습 데이터를 완벽하게 학습하면, KL 발산이 0이 되고 모델의 교차 엔트로피는 학습 데이터의 엔트로피와 정확히 일치해짐

### 문자당 비트와 바이트당 비트

- 엔트로피 및 교차 엔트로피는 비트 단위를 사용함
  - 언어 모델의 교차 엔트로피가 6비트라면, 각 토큰을 표현하는 데 6비트가 필요하다는 뜻
- 모델마다 토큰화 방식이 다르기 때문에 토큰당 비트 수로 모델을 비교할 수 없음
- **문자당 비트(Bits Per Character, BPC)**
  - 토큰당 비트 수가 6이고 평균적으로 각 토큰이 2개의 문자로 이루어져 있다면, BPC = 6 / 2 = 3
  - 문제점: 문자 인코딩 방식의 다양함
    - ASCII는 문자당 7비트를 사용하지만, UTF-8은 문자당 1~4바이트 사용
    - 그래서 BPB가 더 표준화되었음
- **바이트당 비트(Bits Per Byte, BPB)**
  - 원본 학습 데이터의 1바이트를 표현하는 데 필요한 비트 수
  - BPC가 3이고 각 문자가 7비트라면, BPB = 3 / (7 / 8) = 3.428
- BPB가 3.428이라면, 원본 1바이트(8비트)를 3.428비트로 표현한다는 뜻이며, 이는 원본 텍스트를 원래의 절반 이하로 압축할 수 있다는 의미

### 퍼플렉시티(Perplexity)

- 퍼플렉시티: 엔트로피와 교차 엔트로피의 지수 함수
- 종종 PPL로 줄여 씀
- 실제 분포 P와 학습된 분포 Q에 대한 퍼플렉시티
  - $PPL(P) = 2^{H(P)}$
  - $PPL(P,Q) = 2^{H(P,Q)}$
- 교차 엔트로피가 다음 토큰 예측이 얼마나 어려운지를 측정한다면, 퍼플렉시티는 다음 토큰 예측의 불확실성을 측정
  - **불확실성**이 높을수록 다음 토큰으로 가능한 선택지가 많다는 뜻
- 측정 단위 관련
  - 텐서플로, 파이토치 등 인기 있는 ML 프레임워크는 엔트로피 및 교차 엔트로피 단위로 `nat`(자연 로그) 사용
  - nat는 자연로그의 밑인 `e` 사용
  - nat를 단위로 사용하면, 퍼플렉시티는 e의 지수 함수가 됨
    - $PPL(P,Q) = e^{H(P,Q)}$
  - nat 사용이 혼란스럽기 때문에 많은 사람이 모델의 성능을 말할 때 교차 엔트로피 대신 퍼플렉시티를 사용하는 경우도 있음

### 퍼플렉시티 해석과 활용 사례

- 이 책에서는 퍼플렉시티를 언어 모델링의 기본 지표로 사용할 것
- 퍼플렉시티가 낮을수록 모델이 데이터를 더 잘 예측한다는 뜻
- 다음 예측의 불확실성이 클수록 퍼플렉시티가 높아진다는 점을 기억해둘 것
- 퍼플렉시티를 해석할 때 사용할 수 있는 일반적인 규칙들
  - **구조화된 데이터일수록 퍼플렉시티가 낮다**
  - **어휘 크기가 클수록 퍼플렉시티가 높다**
  - **컨텍스트 길이가 길수록 퍼플렉시티가 낮다**
- 퍼플렉시티는 언어 모델 학습의 성능 지표 외에도 다양한 AI 엔지니어링 작업에서 유용함
  - 모델의 성능을 간접적으로 보여주는 좋은 지표
  - 텍스트가 모델의 학습 데이터에 포함되었는지 탐색하고, 학습 데이터의 중복 제거에도 활용될 수 있음
  - '우리 강아지는 여가 시간에 양자역학을 가르친다' 같은 비정상 텍스트를 탐지하는 데 사용될 수 있음
- SFT, RLHF 등 사후 학습을 거치면 보통 퍼플렉시티가 더 높아짐
  - 특정 작업을 잘 수행하도록 학습되므로, 오히려 다음 토큰 예측 능력이 떨어지게 되므로

## 3.3 정확한 평가

- AI 평가자는 주관적이며, 평가 결과는 평가 모델과 프롬프트에 따라 달라짐
- 정확한 점수를 산출하는 2가지 평가 방식을 다룰 것
  - **기능적 정확성(functional correctness)**
  - **참조 데이터의 유사도 측정(reference-based similarity)**
- 3.3은 폐쇄형 응답이 아닌 개방형 응답의 평가에 중점을 둘 것

### 기능적 정확성

- 애플리케이션이 의도한 대로 동작하는지를 측정하기 때문에 모든 애플리케이션의 성능을 평가하는 궁극적 지표
- AI 이전부터 단위 테스트를 통해 검증되어 왔음
- 특히 AI의 코드 생성 능력에 대해서 기능적 정확성이 자동적인 평가 지표로 자주 사용됨
  - 평가할 문제를 내주고 문제마다 k개의 샘플을 생성해서 문제 해결 여부를 확인하는 방식
  - pass@k: k개의 샘플 중 하나라도 문제를 해결하면 성공으로 간주하는 방식
    - k = 3, 10개 문제 중 5개를 해결했다면, 그 모델의 pass@3 점수 = 50%
- 게임 봇 또한 기능적 정확성을 자동으로 평가할 수 있음
  - 봇이 얻는 점수로 성능 측정

### 참조 데이터 유사도 측정

- 자동 평가가 안된다면 AI의 출력을 참조 데이터와 비교해 평가하는 것이 일반적
- 하나의 응답에는 여러 개의 참조 응답이 있을 수 있음
- 참조가 필요한 평가 지표는 참조 기반 지표(reference-based metric), 참조가 필요 없는 평가 지표는 참조 불필요 지표(reference-free metric)라고 부름
- 참조 데이터를 얼마나 빨리 많이 만들 수 있는지가 중요
  - 참조 데이터는 보통 사람이 생성하지만, 점점 AI가 생성하는 경우가 늘고 있음
- 생성된 응답이 참조 응답과 비슷할수록 더 좋은 것으로 간주
- 두 개방형 텍스트 간 유사도 측정 방법 4가지
  - **비교** : 평가자에게 두 텍스트가 같은지 판단하도록 요청
  - **정확한 일치** : 생성된 응답이 참조 응답 중 하나와 정확히 일치하는지
  - **어휘적 유사도** : 두 텍스트가 얼마나 비슷한지
  - **의미적 유사도** : 생성된 응답이 의미(semantic)에서 참조 응답과 얼마나 가까운지
- 비교 방식은 사람이나 AI 평가자가 수행 가능 (다음 절에서 다룸)
- 정확한 일치 방식의 점수는 이진(일치 혹은 불일치), 나머지 두 점수는 연속적인 척도(0~1 또는 -1~1)

**정확한 일치**

- 간단한 수학 문제, 일반 상식 질의, 퀴즈 형태 질의 등 짧고 정확한 응답을 기대하는 작업에 적합
- 답이 같더라도 표현 방식이 다를 수 있기 때문에, 참조 응답이 포함되어 있으면 인정하는 방식도 있음
- 간단한 작업을 넘어서면 정확한 일치는 거의 작동하지 않음

**어휘적 유사도**

- 텍스트를 더 작은 토큰으로 나누어서 비교
- **근사 문자열 매칭(approximate string matching)**
  - fuzzy matching이라고도 불림
  - 두 텍스트가 얼마나 비슷한지 측정하기 위해, 한 텍스트를 다른 텍스트로 바꾸는 데 필요한 편집 횟수를 셈
    - 편집 횟수 = **편집 거리(edit distance)**
  - 편집 연산의 종류
    - 삭제: brad → bad (b 삭제)
    - 삽입: bad → brad (r 삽입)
    - 대체: bad → bed (a를 e로 대체)
    - 일부 퍼지 매처는 두 글자를 서로 바꾸는 것도 하나의 편집으로 취급
- **n-gram 유사도**
  - 개별 토큰이 아닌 토큰의 연속 시퀀스인 n-gram의 겹침을 기준으로 측정
  - 1-gram (unigram): 하나의 토큰, 2-gram (bigram): 두 토큰의 집합
  - 'My cats scare the mice' → 'my cats', 'cats scare', 'scare the', 'the mice' 4개의 bigram
  - 참조 응답의 n-gram 중 몇 퍼센트가 생성된 응답에도 있는지를 측정
- 어휘적 유사도의 일반적인 지표
  - **BLEU(Bilingual Evaluation Understudy)**
  - **ROUGE(Recall-Oriented Understudy for Gisting Evaluation)**
  - **METEOR(Metric for Evaluation of Translation with Explicit ORdering)**
  - **CIDEr(Consensus-based Image Description Evaluation)**
- 어휘적 유사도 측정의 단점
  - 포괄적인 참조 응답 세트를 만들어야 함
  - 참조 응답도 틀릴 수 있음
  - 점수가 높더라도 반드시 기능적 정확성이 높지 않을수도 있음

**의미적 유사도**

- 측정을 위해 우선 텍스트를 임베딩이라는 숫자 표현으로 변환해야 함
- 임베딩 유사도 점수
  - 두 임베딩이 정확히 같으면 1
  - 두 임베딩이 반대되면 -1
- 텍스트 뿐 아니라 이미지, 오디오 등 모든 데이터 유형의 임베딩에 대해 계산 가능
- 의미적 텍스트 유사도를 위한 지표들
  - BERTScore
  - MoverScore
- 의미적 유사도의 신뢰성은 임베딩 모델의 품질에 따라 달라짐
- 임베딩 알고리즘 실행을 위한 상당한 계산 능력과 시간 소모는 단점이 될 수 있음

**임베딩 소개**

- 컴퓨터는 숫자 기반으로 작동하기 때문에, 모델은 입력을 컴퓨터가 처리할 수 있는 숫자 표현으로 변환해야 함
- 임베딩: 원본 데이터의 의미를 담으려는 숫자 표현
- 임베딩 벡터의 크기(임베딩 벡터 원소 개수)는 보통 100에서 10,000 사이
- 임베딩을 만들기 위해 특별히 학습된 모델들
  - BERT
  - CLIP
  - Sentence Transformers
- 많은 ML 모델은 입력을 벡터로 표현하기 위한 임베딩 생성 단계를 포함
- 임베딩의 품질은 해당 작업의 유용성을 기준으로도 평가 가능
  - 분류, 주제 모델링, 추천 시스템, RAG 등
  - **MTEB(Massive Text Embedding Benchmark)** : 임베딩 모델을 평가하기 위한 벤치마크
- 서로 다른 유형의 데이터에 대한 통합 임베딩을 만들고자 하는 시도가 이어지고 있음
  - CLIP : 텍스트와 이미지를 같은 공간에 임베딩
  - ULIP : 텍스트, 이미지, 포인트 클라우드를 같은 공간에 임베딩
  - 이를 **멀티모달 임베딩 공간**이라고 부름

## 3.4 AI 평가자 (AI as a Judge)

- AI 평가자 = AI를 사용해 AI를 평가하는 접근 방식
  - LLM 평가자라고도 부름

### AI 평가자를 쓰는 이유

- 사람 평가자에 비해 빠르고, 사용도 쉽고, 비용도 저렴하고, 참조 데이터가 없어도 됨
- AI 모델에게 정확성, 반복성, 유해성, 건전성, 환각 등 어떤 기준으로든 출력을 평가하도록 요청 가능
- AI 모델은 사람의 의견을 바탕으로 만들어졌기 때문에, 대중적인 관점에서 판단을 내림
- AI는 평가할 뿐만 아니라 자신의 평가를 설명할 수도 있음

⇒ 다양한 애플리케이션에서 유용하며, 사람만큼 정확하진 않더라도 프로젝트의 개발 방향을 잡기에는 충분함

### AI 평가자 사용법

**일부 AI 도구가 제공하는 내장 AI 평가 기준**

- [Azure AI Foundry](https://azure.microsoft.com/ko-kr/products/ai-foundry)
  - 사실 기반성, 관련성, 일관성, 유창함, 유사성
- [MLflow.metrics](https://mlflow.org/docs/latest/api_reference/python_api/mlflow.metrics.html#generative-ai-metrics)
  - 신뢰성, 적합성
- [LangChain Criteria Evaluation](https://docs.langchain.com/oss/python/langchain/overview)
  - 간결성, 관련성, 정확성, 일관성, 유해성, 악의성, 유용성, 논란 가능성, 여성 혐오, 무신경함, 범죄성
- [Ragas](https://docs.ragas.io/en/latest/concepts/metrics/index.html)
  - 신뢰성, 응답 관련성

⇒ AI 평가 기준은 표준화되어 있지 않고, 점수는 평가자 역할을 하는 AI 모델 및 프롬프트에 따라 달라짐

**AI 평가자를 프롬프트하는 일반적인 방법**

- 모델이 수행할 작업
- 모델을 평가할 때 따라야 할 기준
- 점수 체계
  - 분류: 좋음/나쁨, 관련됨/관련 없음, 없음/중립 등
  - 이산적인 숫자 값: 1~5점, 1~10점 등
  - 연속적인 숫자 값: 0~1, -1~1 등 → 유사성 정도를 평가할 때 사용

⇒ 일반적으로 숫자 기반보다 텍스트 기반(분류)을 더 잘 다룸, 수치 점수 체계 중에서는 이산 점수가 더 잘 작동, 일반적인 이산 점수 체계는 1~5점

**정리**

- 프롬프트에 대한 예시가 있으면 평가 성능이 더 좋아짐
- AI 평가자는 단순히 모델만 있는 것이 아니라 모델과 프롬프트를 모두 포함하는 시스템

### AI 평가자의 한계

- AI로 AI를 평가하는 것은 신뢰성이 떨어진다고 여겨지는 경향
- AI 평가자는 애플리케이션에 상당한 비용 및 지연을 초래할 수 있음

**비일관성**

- AI 평가자는 AI 애플리케이션과 마찬가지로 확률적
- 샘플링 변수를 사용하거나, 프롬프트에 평가 예시를 포함하는 식으로 일관성을 높일 수 있음
  - 하지만 예시를 많이 포함할수록 프롬프트가 길어지고, 긴 프롬프트는 추론 비용을 높임
- 하지만 높은 일관성이 높은 정확도를 의미하지는 않을 수 있음
  - 평가자가 같은 실수를 일관되게 할 수도 있기 때문

**평가 기준의 모호성**

- 사람이 설계한 지표에 비해 AI 평가자의 지표는 표준화되지 않았음
- 오픈 소스 도구들은 충실성(faithfulness)이라는 내장된 기준을 가지지만 지시와 점수 체계가 모두 다름
- 평가 방법 표준화에는 신뢰성 보장을 위한 시간이 더 필요하다

**비용과 지연 시간 증가**

- 강력한 모델로 응답을 평가하는 것은 비용이 많이 들고 시간이 오래 걸림
- 응답 품질, 사실 일관성, 유해성 등 여러 기준에 대해 평가하려면 API 호출 횟수는 더더욱 늘어날 것
- 하지만 사람 평가자에 비하면 훨씬 저렴함
- 운영 환경의 경우 지연 시간과의 트레이드오프도 고려해야 함

**AI 평가자의 편향**

- **자기 편향(self-bias)**
  - 자신이 생성한 응답을 더 선호하는 경향
- **첫 위치 편향(positional bias)**
  - 여러 응답 중 첫 번째 응답을 선호할 수 있음
  - 반대로 사람의 경우에는 마지막에 본 응답을 선호하는 경향이 있음 (최근성 편향, recency bias)
  - 순서를 바꿔가며 같은 테스트를 여러 번 반복하거나, 세심하게 작성된 프롬프트로 완화할 수 있음
- **장황성 편향(verbosity bias)**
  - 품질과 관계없이 더 긴 응답 선호
  - 모델이 더 강력해질수록 장황성 편향은 완화되는 경향이 있음

**평가자로 활용 가능한 모델**

- 평가자는 평가받는 모델보다 강력해야 하는가, 약해야 하는가, 비슷해야 하는가?
- **더 강력한 평가자**
  - 얼핏 보면 이치에 맞는 선택지
  - 약한 모델이 낮은 비용 및 지연 시간으로 응답을 생성하고, 강력한 모델이 백그라운드에서 일부 응답을 평가하는 방식
  - 문제점
    - 강력한 모델을 평가하기 위한 더 강력한 모델을 어떻게 구하는가?
    - 어떤 모델이 더 강력한지를 어떻게 판단하는가?
- **자기 평가(self-evaluation)**
  - 자기 편향 때문에 편법처럼 들릴 수 있는 방식
  - 하지만 기본적인 검증에 매우 유용함
  - 모델에게 자기 평가를 요청하면 응답 수정 및 개선을 유도할 수 있음
- **더 약한 평가자**
  - 아직 논의가 끝나지 않은 문제
  - 일부는 평가자가 생성자보다 더 쉬운 작업이라고 주장하고 있음
- 필자는 평가에 특화된 작은 모델이 필요하다고 생각함
  - **보상 모델**
    - (프롬프트, 응답) 쌍을 입력받고 점수를 출력하는 모델
    - 보상 모델은 수년간 RLHF에서 성공적으로 사용되고 있는 방식
  - **참조 기반 평가자**
    - 유사도 점수나 품질 점수 출력
    - 참조 응답 간의 유사도 점수를 출력하는 모델
  - **선호도 모델**
    - 주어진 프롬프트에 대해 어느 응답이 더 나은지 출력
    - 아마 특화된 평가자 중에서 가장 기대되는 방향일 것
    - 사람의 선호도를 예측함으로써 많은 가능성을 열어줌
    - 선호도 데이터를 확보하지 않아도 선호도를 정확하게 예측할 수 있게 될 것

## 3.5 비교 평가를 통해 모델 순위 정하기

- **개별 평가**: 각 모델을 독립적으로 평가하고 점수를 기준으로 순위를 매김
- **비교 평가**: 모델들을 서로 비교해 평가하고 비교 결과로 순위 계산
- 응답의 품질이 주관적일 때는 보통 비교 평가가 더 쉬움
- 많은 경우 **정확성**을 기준으로 평가
- 선호도 기반 투표는 잘못된 신호를 줄 수 있음
  - 사용자에게 선택을 요청하는 것은 사용자의 불만을 일으킬 수 있음
  - 사용자의 비교 의견을 수집할 때 어떤 질의가 선호도 투표에 적합할지 판단하는 것도 어려움
  - 이미 알고 있는 것을 물어볼 때에 적합
- 모델 간의 비교를 **경기(match)** 라고 부름
  - 경기를 통해 모델들의 승률을 계산하고, 평가 알고리즘을 사용해 모델의 순위 계산
  - 평가 알고리즘은 일반적으로 먼저 비교 결과에서 모델 점수를 계산하고 이 점수로 모델의 순위를 매김
  - 스포츠 및 비디오 게임 분야에서 사용되는 Elo, Bradley-Terry, TrueSkill 등 활용
  - 모델 순위 매기기는 과거 경기 결과로 순위를 계산해 미래 경기 결과를 예측하는 예측 문제나 다름없음

### 비교 평가의 과제들

- 점수 기반 평가에서는 벤치마크와 지표 설계가 가장 어렵고, 점수로 모델 순위를 매기는 건 쉬움
- 반면 비교 평가에서는 비교 결과 수집과 모델 순위 매기기 모두 까다로움

**확장성 병목**

- 비교 평가는 기본적으로 데이터가 많이 필요
- 비교할 모델 쌍의 수는 모델 수의 제곱에 비례해서 증가
- **전이성**을 활용하면 비교 횟수를 줄일 수 있음
  - A가 B보다 순위가 높고, B가 C보다 순위가 높다면 A가 C보다 순위가 높다고 추론
  - 하지만 전이성 가정이 AI 모델에 적용될 수 없다는 비판도 있음
  - 사람들의 선호도가 반드시 전이성을 가지지 않을 수도 있음
  - 서로 다른 모델 쌍이 서로 다른 평가자와 프롬프트로 평가되기 때문에 비전이성 발생 가능
- 새로운 모델 평가의 어려움
  - 새로운 모델을 기존 모델과 비교해야 하며 순위를 바꿔야 할 수도 있음
- 비공개 모델 평가의 어려움
  - 직접 비교 데이터를 수집하거나, 공개 순위표 제공업체에 비공개 평가를 의뢰해야 함
- 확장성 병목 현상은 더 나은 **매칭 알고리즘**으로 완화 가능
  - 어떤 모델 쌍의 결과를 확신할 수 있다면 해당 모델 쌍의 비교는 건너뛸 수 있음
  - 효율적인 매칭 알고리즘은 전체 순위의 불확실성을 최대한 줄일 수 있는 경기를 선택

**표준화와 품질 관리의 부재**

- 커뮤니티에 비교를 맡길 수 있을 것
- 누구나 [웹사이트](https://huggingface.co/datasets/lmsys/chatbot_arena_conversations)에 접속해서 프롬프트를 입력하면 익명의 두 모델의 응답을 평가할 수 있음
- 장점: 다양한 비교 결과, 조작 가능성
- 단점: 표준화 및 품질 관리를 강제하기 어려움
  - 인터넷에 접속할 수 있는 사람이라면 누구나 아무 프롬프트를 사용해 모델 평가 가능
  - 다양한 선호도를 파악할 수 있지만, 모든 활용 사례에 적합한 선호도인지는 미지수
  - **크라우드소싱(crowdsourcing)** 방식의 비교는 실제 현장에서 실제로 어떻게 쓰이는지 제대로 반영하지 못함
  - 단순한 프롬프트는 응답의 난이도가 낮아 모델들의 성능 차이를 구별하기 어려움
- 보완 방법들
  - 사용자가 미리 정해진 프롬프트만 사용하도록 제한하기 → 다양한 활용 사례 X
  - 신뢰할 수 있는 평가자만 사용하기 → 비용이 많이 들고 비용 횟수가 크게 줄어듦
  - 비교 평가를 제품에 통합해서 사용자가 작업하는 동안 모델을 평가하도록 하기 → 사용자의 무성의한 선택이 노이즈를 발생시킬 수 있음

⇒ 이 때문에 사람 평가자보다 AI가 선호되기도 함, AI가 전문가보다 뛰어나진 못해도 무작위 인터넷 사용자보다 더 신뢰할 수 있기 때문

**비교 성능에서 절대 성능으로**

- 비교 평가는 어떤 모델이 더 나은지 알려주지만, 모델이 얼마나 좋고 어떤 사례에 충분히 좋은지는 알려주지 않음
- 비교 평가만으로는 성능 향상이 추가 비용을 감당할 만한 가치가 있는지 판단하기 어려움

### 비교 평가의 미래

- 비교 평가에는 이렇듯 많은 한계가 있지만 여러 장점이 있음
- 두 출력을 비교하는 것이 각 출력에 구체적인 점수를 매기는 것보다 쉬움
  - 특히 모델이 더 강력해지면 사람 평가자가 구체적인 점수를 매기는 것은 불가능해질 수 있음
