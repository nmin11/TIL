# 9. 추론 최적화

- 9장에서는 모델을 더 빠르고 저렴하게 만드는 방법에 집중
- 추론 최적화는 모델, 하드웨어, 서비스 수준에서 접근 가능
  - 모델: 학습된 모델의 크기를 줄이거나 더 효율적인 아키텍처 개발
  - 하드웨어: 더 성능이 좋은 하드웨어 설계
- 추론 서비스는 주어진 하드웨어에서 모델을 실행해 사용자 요청을 처리
  - 특정 하드웨어에 최적화된 모델용 기법들을 활용 가능
  - 이를 통해 사용량과 트래픽 패턴을 고려해서 리소스를 효율적으로 배분하고 지연 시간 및 비용을 줄일 수 있음
- 추론 최적화는 모델 연구자, 애플리케이션 개발자, 시스템 엔지니어, 컴파일러 설계자, 하드웨어 아키텍트 등이 함께 작업하는 융합 분야
- 9장에서는 AI 추론에서 발생하는 병목 현상과 이를 해결하는 기법들을 다룰 것
  - 주로 모델과 서비스 수준의 최적화에 집중
  - AI 가속기도 간단히 살펴볼 것
- 성능 지표와 트레이드오프도 다룰 것
- 오픈 소스 모델이 늘어나면서 자체 추론 서비스를 만드는 팀이 많아졌음
  - 꼭 자체적으로 만들지 않더라도 추론 최적화 기법들의 원리를 이해해두면 추론 서비스와 프레임워크를 평가하는 데 도움이 됨

## 9.1 추론 최적화 이해하기

- AI 모델의 생명주기
  - 학습: 모델을 만드는 과정
  - 추론: 만들어진 모델에 입력을 넣어서 결과를 얻는 과정
- 모델을 직접 학습하거나 파인튜닝하지 않는다면 주로 추론만 다루게 됨

### 추론 개요

- 추론 서버
  - 운영 환경에서 모델 추론을 실행하는 구성 요소
  - 여러 모델들을 호스팅하고 필요한 하드웨어 사용 가능
  - 애플리케이션에서 요청이 들어오면 리소스를 할당해서 적절한 모델을 실행하고 결과를 사용자에게 반환
  - 추론 서버는 추론 서비스라는 더 큰 시스템의 한 부분
- 추론 서비스
  - 요청을 받아서 적절한 곳으로 전송
  - 요청이 추론 서버에 닿기 전에 전처리하는 역할도 담당
  - OpenAI나 구글에서 제공하는 모델 API들이 추론 서비스

**연산 병목**

- 최적화는 병목을 찾아내서 해결하는 일
- 추론 서버도 담당하는 추론 작업의 연산 병목을 해결하도록 설계되어야 함
- 주요 연산 병목들
  - **연산 제약(compute-bound)**
    - 작업을 끝내는 데 걸리는 시간이 연산량에 따라 결정되는 경우
  - **메모리 대역폭 제약(memory bandwith-bound)**
    - 시스템 내부 데이터 전송 속도에 제약을 받는 경우
    - 특히 메모리와 프로세서 간 데이터 전송 속도가 핵심 병목 지점
- **산술 강도(arithmetic intensity)**
  - 수학적으로 연산이 어느 쪽 제약을 받는지 구분하는 방법
  - 메모리 1바이트에 접근할 때 몇 번의 산술을 하는지 측정
- **roofline chart**
  - NVIDIA Nsight 같은 프로파일링 도구들이 제공
  - 작업이 연산 제약인지 메모리 대역폭 제약인지 보여줌
  - 하드웨어 성능 분석에서 자주 쓰이는 방법
- 연산 제약 해결 방법: 더 많은 칩에 분산시키거나 연산 성능이 더 좋은 칩을 활용하거나
- 메모리 대역폭 제약 해결 방법: 대역폭이 더 넓은 칩 활용
- 모델 구조와 작업 종류에 따라 연산 병목도 다르게 나타남
  - 스테이블 디퓨전 등 이미지 생성 모델 추론은 보통 연산 제약
  - 자기회귀 언어 모델 추론은 보통 메모리 대역폭 제약
- 언어 모델 추론의 제약 예시
  - **prefill**
    - 모델이 입력 토큰을 병렬 처리
    - 한 번에 처리 가능한 토큰 개수는 하드웨어가 정해진 시간에 실행 가능한 연산량에 달려 있음
    - 따라서 **연산 제약**에 해당
  - **decode**
    - 모델이 출력 토큰을 한 번에 하나씩 생성
    - 보통 큰 행렬들을 GPU에서 불러오는 작업을 포함
    - 그러므로 하드웨어가 얼마나 빨리 데이터를 메모리로 불러올 수 있는지에 따라 제한됨
    - 따라서 **메모리 대역폭 제약**에 해당
- LLM 추론 서버에서 병목이 어디서 생기는지는 컨텍스트 길이, 출력 길이, 배치 요청 방식에 따라 달라짐

**온라인과 배치 추론 API**

- **온라인 API**
  - 지연 시간 최적화
  - 요청이 들어오면 바로 처리
  - 온라인 또한 지연 시간에 큰 영향을 주지 않는 선에서는 배치 처리가 가능
  - 대부분 스트리밍 모드 지원
  - 적합 케이스
    - 챗봇
    - 코드 자동 생성
  - 추론 방식: 요청이 들어온 후에 예측 연산
- **배치 API**
  - 비용 최적화
  - 다양한 최적화 기법 사용 가능
  - 적합 케이스
    - 합성 데이터 생성
    - 정기 보고서 작성
    - 모든 데이터를 다시 처리해야 하는 새 모델로의 마이그레이션
    - 대규모 고객층을 위한 개인화된 추천 혹은 뉴스레터 생성
  - 추론 방식: 요청이 들어오기 전에 예측을 미리 연산

### 추론 성능 지표

**지연 시간**

- 사용자 질의 전송 시점부터 완전한 응답을 받기까지 걸리는 시간

**TTFT (Time To First Token)**

- 첫 토큰까지 걸리는 시간
- 사용자가 질의를 보낸 후 첫 번째 토큰이 나오기까지 걸리는 시간
- 프리필 단계에 해당하며 입력 길이에 따라 달라짐

**TPOT (Time Per Output Token)**

- 출력 토큰당 시간
- 첫 번째 토큰 이후 각 토큰이 얼마나 빨리 생성되는지 측정
- 토큰이 생성되는 대로 사용자가 읽어야 하는 스트리밍 모드의 경우, TPOT는 사람의 읽기 속도보다 빨라야 함
  - 초당 6-8개 토큰 정도

**TBT (Time Between Tokens) & ITL (Inter Token Latency)**

- 토큰 간 시간 및 토큰 간 지연 시간
- 둘 다 출력 토큰 사이사이 걸리는 시간을 측정

**전체 지연 시간**

- TTFT + (TPOT * 출력 토큰 수)
- 전체 지연 시간이 같더라도 TTFT와 TPOT가 다르면 사용자 경험도 달라짐

**Time to Publish**

- 일부 팀은 사용자가 실제로 보는 첫 토큰까지의 시간이라는 의미로 **공개까지의 시간** 지표를 사용

**처리량(Throughput)**

- 추론 서비스가 모든 사용자의 요청을 통틀어서 초당 몇 개의 출력 토큰을 만들어낼 수 있는지 측정
- 입력 토큰 처리(프리필)와 출력 토큰 생성(디코드)의 처리량을 모두 고려해야 함
- 처리량 측정 단위
  - 주로 tokens/s(TPS) 로 측정
  - 여러 사용자에게 서비스한다면 tokens/s/user 로 측정하기도 함
  - RPS(Request Per Second) 로 측정하는 애플리케이션도 많음
  - 파운데이션 모델 기반 애플리케이션은 요청 완료가 오래 걸릴 수 있어서 RPM(Request Per Minute)을 많이 사용
- 보통 처리량이 높을수록 비용이 낮아짐
- 요청당 비용 지표로 추론 서버의 효율성 비교 가능
- **goodfoot**: SLO(Software-Level Objective)를 만족하는 초당 요청 개수

**활용률(Utilization)**

- 리소스가 얼마나 효율적으로 사용되고 있는지 측정
- 보통 전체 사용 가능 용량 중에서 실제로 사용 중인 비율로 표시
- **MFU (Model FLOP/s Utilization)**
  - 시스템이 최고 FLOP/s로 동작할 때 달성할 수 있는 이론상 최대 처리량 대비 실제 처리량(tokens/s)
  - 연산 제약 작업에서 높게 나타나는 경향
  - 학습은 추론보다 예측 가능하므로, 학습 MFU > 추론 MFU
  - 추론에서 프리필은 연산 위주이고 디코딩은 메모리 대역폭 위주이므로, 프리필 MFU > 디코딩 MFU
- **MBU (Model Bandwitdth Utilization)**
  - 사용 가능 메모리 대역폭 중 실제로 쓰이는 비율
  - 사용 중인 메모리 대역폭 = 파라미터 수 * 파라미터당 바이트 * tokens/s
  - MBU = (파라미터 수 * 파라미터당 바이트 * tokens/s) / (이론적 대역폭)
  - 대역폭 제약 작업에서 높게 나타나는 경향
- 활용률은 서비스가 얼마나 효율적인지 보여주지만, 결국 중요한 것은 더 빠르고 저렴하게 처리하는 것

### AI 가속기

- 소프트웨어가 얼마나 빠르고 저렴하게 돌아가는지는 어떤 하드웨어에서 실행되느냐에 달려 있음
- 그러므로 하드웨어를 제대로 이해하면 훨씬 깊이 있는 최적화 가능
- AI 모델의 발전은 늘 하드웨어의 발전과 함께 얽혀 왔음

**가속기의 정의**

- 단순 정의: 특정 종류의 연산 작업을 빠르게 처리하도록 만들어진 칩
- AI 가속기 = AI 작업 전용으로 설계됨
- 가장 널리 쓰이는 AI 가속기 = GPU
- CPU는 범용 작업용으로, GPU는 병렬 처리용으로 만들어짐
- 최근에는 추론 전용 칩이 떠오르고 있음
  - 보편적 시스템에서는 추론 비용이 학습 비용을 넘어설 수 있다는 이론
  - 학습은 보통 처리량에 집중하고, 추론은 지연 시간 줄이는 데 집중
  - 따라서 추론용 칩들은 대용량 메모리 보다는 낮은 정밀도와 빠른 메모리 접근에 최적화된 경우가 많음
- 특정 모델 아키텍처에 특화된 칩들도 있음
  - 트랜스포머 전용 칩
  - 데이터센터용 칩
  - 스마트폰, 노트북 등 소비자용 기기를 위한 칩
- 한 칩 안에는 여러 데이터 유형에 최적화된 다양한 **연산 유닛(compute unit)** 이 함꼐 포함될 수 있음
  - 스칼라, 벡터, 텐서 등

**연산 성능**

- 보통 칩이 정해진 시간에 수행할 수 있는 연산 수로 측정
- 가장 많이 쓰이는 지표: FLOP/s
  - 초당 부동 소수점 연산 횟수 측정
  - 하지만 현실적으로 애플리케이션이 최대 FLOP/s 수치를 달성할 가능성은 거의 없음
- 칩이 1초에 수행 가능한 연산 횟수는 수치 정밀도에 따라 달라짐
  - 정밀도가 높을수록 칩의 실행 가능 연산이 줄어듦

**메모리 크기와 대역폭**

- GPU는 수많은 코어가 병렬 작동하고, 메모리에서 코어들로 데이터를 계속 옮겨야 하므로 데이터 전송 속도가 중요
- 특히 거대한 가중치 행렬과 학습 데이터를 다루는 AI 모델에서는 데이터 전송 속도가 더 중요
- 그래서 GPU 메모리는 CPU 메모리에 비해 더 넓은 대역폭과 더 낮은 지연 시간이 필요
- 가속기의 메모리는 크기와 대역폭으로 평가됨
  - 가속기가 들어가는 전체 시스템의 맥락 안에서 평가되어야 함
- GPU 같은 가속기는 보통 3단계 메모리와 상호작용
  - **CPU 메모리 (DRAM)**
    - 가속기는 보통 CPU와 함께 사용되므로 CPU 메모리에 접근 가능
    - CPU 메모리는 보통 대역폭이 가장 낮은데, 크기는 다양함
  - **GPU 고대역폭 메모리 (HBM)**
    - GPU 전용 메모리
    - CPU 메모리보다 빠르게 접근 가능
    - 훨씬 높은 대역폭 및 빠른 데이터 전송 속도
  - **GPU 온칩 SRAM**
    - 칩 안에 바로 들어 있는 메모리
    - 자주 쓰는 데이터와 지시를 거의 바로 접근할 수 있도록 저장
    - SRAM으로 만들어진 L1, L2 캐시 포함 (일부는 L3 캐시도 포함)
    - 캐시들은 레지스터 파일과 공유 메모리 같은 다른 부품들과 함께 더 큰 온칩 메모리의 일부가 됨
    - SRAM은 크기가 작고 데이터 전송 속도가 매우 빠름
- GPU 최적화의 상당 부분은 메모리 계층 구조을 어떻게 최대한 활용하느냐에 달려 있음

**전력 소모**

- 칩은 연산할 때 트랜지스터를 사용하며, 각 연산은 트랜지스터를 켰다 껐다 하면서 에너지를 소모
- GPU에는 수십억에서 수백억 개의 트랜지스터가 들어 있음
- 수십억 개의 트랜지스터는 엄청난 양의 에너지를 소비하고 상당한 열을 발생시킴
- 냉각 시스템에도 전기가 필요하므로 전체 에너지 소비가 더 늘어남
- **열 설계 전력(Thermal Design Power, TDP)**
  - 최대 전력 소모량 = 칩 최대 부하 상태에서 뽑아낼 수 있는 최대 전력
  - TDP = 칩이 일반적인 작업을 할 때 냉각 시스템이 방출해야 하는 최대 열
  - 전력 소비를 정확히 측정한 건 아니지만, 예상 전력 소모량을 나타내는 지표
  - 최대 전력 소모량은 대략 TDP의 1.1배에서 1.5배 정도

## 9.2 추론 최적화

- 추론 최적화는 모델, 하드웨어, 서비스 수준에서 수행 가능
- 속도와 비용을 위한 모델 최적화는 모델 품질을 떨어뜨리지 않도록 조심해야 함

### 모델 최적화

- 목표: 모델 자체를 수정해서 효율성을 높이는 것
- 트랜스포머 아키텍처의 자기회귀 언어 모델들은 추론을 리소스 집약적으로 만드는 3가지 특성이 있음
  - 모델 크기
  - 자기회귀 디코딩
  - 어텐션 메커니즘
- 위 문제들을 해결하기 위한 방법들을 알아볼 것

**모델 압축**

- 모델 크기를 줄이기 위한 여러 기법들
- 모델이 작아지면 속도도 빨라질 수 있음
- 이미 2가지 모델 압축 기법을 다뤘음
  - 양자화: 모델의 정밀도를 낮춰서 메모리 사용량을 줄이고 처리량을 늘리는 것
  - 증류: 작은 모델이 큰 모델의 동작을 따라하도록 학습시키는 것
- **Pruning**
  - 모델 증류를 통해 더 적은 파라미터로 큰 모델의 동작을 잡아낼 수 있다는 것을 알 수 있음
  - 그렇다면 큰 모델 안에 전체 모델의 동작을 담아내는 파라미터의 부분집합이 존재하지 않을까? ⇒ 프루닝의 핵심 아이디어
  - 신경망에서 프루닝의 2가지 의미
    - 신경망 노드 전체 제거
      - 아키텍처를 바꾸고 파라미터 개수를 줄임
    - 예측에 도움이 되지 않는 파라미터를 찾아서 0으로 설정
      - 전체 파라미터 개수를 줄이지 않고 0이 아닌 파라미터 개수만 줄임
      - 모델이 더 희소해져서 모델 저장 공간도 줄고 연산도 빨라짐
  - 프루닝한 모델을 그대로 써도 되지만, 추가 파인튜닝을 통해 남은 파라미터를 조정하고 성능을 복구할 수 있음
  - 하지만 실제로 프루닝을 쓰는 경우가 많지 않음
    - 원본 모델 아키텍처에 대한 이해가 필요해서 하기 어려움
    - 다른 방법들에 비해 성능 향상 폭이 적은 경우가 많음
    - 모든 하드웨어 아키텍처가 프루닝의 회소성 이점을 제대로 활용하도록 설계되어 있지는 않음
- **가중치 전용 양자화(weight-only quantization)**
  - 이 분야에서 가장 인기 많은 방법
  - 쓰기 쉬우며 많은 모델에서 바로 작동하고 효과가 뛰어남

**자기회귀 디코딩 병목 현상 극복하기**

- 자기회귀 언어 모델은 토큰을 하나씩 순차적으로 생성
- 그러므로 자기회귀 생성 과정을 조금만 개선해도 사용자 경험이 크게 향상될 수 있음
- 이 병목을 극복하기 위한 기법들이 빠르게 발전되고 있음
- **추측 디코딩(Speculative Decoding)**
  - **추측 샘플링(speculative sampling)** 이라고도 부름
  - 더 빠르지만 성능이 낮은 모델을 사용해 토큰 시퀀스를 생성한 다음, 목표 모델이 이를 검증하는 방식
  - 더 빠른 모델은 초안 출력을 제안하기 때문에 초안 모델 혹은 제안 모델이라고도 불림
  - 작동 방식
    - 초안 모델이 K개 토큰 시퀀스 생성
    - 목표 모델이 K개 토큰을 병렬로 검증
    - 목표 모델이 초안 시퀀스를 순서대로 검증하며, 처음으로 예측이 엇갈리는 지점 앞까지의 토큰들만 수락
    - 목표 모델이 수락 가능한 초안 토큰 개수 j개만큼 수락 후, 목표 모델이 추가로 하나의 토큰을 직접 생성
    - 이후 다시 처음으로 돌아가서 초안 모델이 K개의 토큰을 다시 생성하고 다시 검증하는 과정을 반복
    - 만약 초안 토큰이 하나도 수락되지 않으면, 이 반복은 목표 모델이 생성한 하나의 토큰만 만들어 냄
  - 모든 초안 시퀀스가 거부된다면, 목표 모델이 전체 응답을 직접 생성하게 되므로 오히려 지연 시간이 늘어날 수 있음
  - 위 상황을 피하기 위한 3가지 통찰
    - 검증은 병렬화 가능하지만 생성은 순차적
      - 목표 모델이 토큰 시퀀스를 검증하는 시간은 직접 생성하는 시간보다 짧음
      - 추측 디코딩은 사실상 디코딩의 연산 방식을 프리필 방식으로 바꾸는 효과가 있음
    - 출력 토큰 시퀀스에서 예측이 쉬운 토큰들을 잘 맞추는 약한 초안 모델을 사용해서 초안 토큰 수락률 높이기
    - 디코딩은 메모리 대역폭에 제약을 받기 때문에 디코딩 과정에서 보통 남는 FLOP을 검증에 활용 가능
  - 초안 토큰 수락률은 도메인에 따라 다름
    - 코드처럼 정해진 구조를 따르는 텍스트는 보통 수락률이 더 높음
    - 검증해야 할 K 값이 클수록 목표 모델의 검증 횟수는 줄지만 초안 토큰 수락률이 낮아짐
  - 비교적 구현이 쉽고 모델 품질을 바꾸지 않기 때문에 많이 연구되고 있음
- **참조 기반 추론(Retrieval-Augmented Generation, RAG)**
  - 응답할 때 입력의 토큰들을 참조해야 하는 경우가 많음
  - 핵심 아이디어: 모델이 반복되는 토큰을 새로 생성하는 대신, 입력을 바로 복사해서 생성 속도를 높이면 어떨까?
  - 추측 디코딩과도 비슷하지만, 모델을 사용해 초안 토큰을 생성하는 대신, 입력에서 초안 토큰을 가져온다는 차이점
  - 핵심 과제: 각 디코딩 단계에서 컨텍스트에서 가장 관련성 높은 텍스트 구간을 찾아내는 알고리즘을 개발하는 것
  - 가장 간단한 방법: 현재 토큰들과 일치하는 텍스트 구간을 찾는 것
  - 추측 디코딩과 달리, 추가 모델이 필요 없음
  - 검색 시스템, 코딩, 멀티 턴 대화 처럼 컨텍스트와 출력 사이에 상당한 중복이 있는 생성 시나리오에서만 유용
- **병렬 디코딩(Parallel Decoding)**
  - 초안 토큰으로 자기회귀 생성을 빠르게 하는 대신, 순차적 의존성을 없애는 것이 목표
  - 바로 앞 토큰이 뭔지 모르는 상태에서 토큰 시퀀스를 병렬로 생성하려고 시도
  - 기존 시퀀스만 알아도 다음 몇 개 토큰을 충분히 예측 가능하기 때문에 가능한 방법
  - 병렬 토큰들은 동일한 디코더로도 생성할 수 있고, 서로 다른 디코딩 헤드들로도 생성할 수 있음
  - 병렬 토큰들은 순차적으로 생성되지 않았기 때문에, 서로 잘 맞는지 검증이 필요
  - 병렬 디코딩의 가장 중요한 부분이 검증과 통합
  - Lookahead 디코딩의 Jacobi method 예시
    - K개 미래 토큰이 병렬로 생성됨
    - K개 미래 토큰에 대해 컨텍스트 일관성, 서로 의미가 통하는지 여부 검증
    - 하나 이상의 토큰이 검증에 실패하면, 실패한 토큰들만 다시 만들거나 고침
    - 모델은 생성된 모든 토큰이 검증을 통과하고 최종 출력에 통합될 때까지 생성된 토큰들을 계속 정제
- **어텐션 메커니즘 최적화**
  - 다음 토큰 생성에는 이전 모든 토큰의 키와 값 벡터가 필요
  - 매번 토큰들의 키와 값 벡터를 다시 연산하는 대신, 이전 토큰 생성 때 연산한 벡터들을 재사용
  - 재사용을 위해 키와 값 벡터를 저장하는 **KV 캐시(KV Cache)** 활용
    - KV 캐시는 학습이 아닌 추론에만 사용됨
    - 학습에서는 시퀀스의 모든 토큰을 미리 알 수 있기 때문에 한꺼번에 병렬 계산 가능
    - 기존의 토큰 하나 생성 시 어텐션 연산량은 시퀀스 길이에 제곱에 비례해서 증가 = O(n²)
    - KV 캐시를 사용하면 토큰 하나 생성 시 어텐션 연산량은 시퀀스 길이에 비례해서 증가 = O(n)
    - KV 캐시 크기는 배치 크기가 클수록 더 커짐
    - KV 캐시 크기는 결국 사용 가능한 하드웨어 저장 공간에 의해 제한됨
    - 캐시가 크면 롱 컨텍스트 처리 시 애플리케이션에 병목 현상을 일으킴
    - 캐시 크기가 크면 메모리에 로드하는 데도 시간이 오래 걸리고, 느린 응답을 유발할 수 있음
    - 메모리 = 2 * B * S * L * H * M
      - B: 배치 크기
      - S: 시퀀스 길이
      - L: 트랜스포머 레이어 수
      - H: 모델 차원
      - M: 캐시 수치를 저장하는 메모리 형식 (FP16, FP32 등)
  - 어텐션 메커니즘을 효율적으로 만들기 위한 많은 기법들이 개발되었음
    - 어텐션 메커니즘 재설계
    - KV 캐시 최적화
    - 어텐션 연산을 위한 커널 작성
- **어텐션 메커니즘 재설계**
  - 어텐션 메커니즘 효율화 기법들은 작동 방식 자체를 변경
  - 이 기법들은 추론 최적화에 도움이 되지만, 모델 아키텍처 변경으로 인해 학습 또는 파인튜닝 단계에서만 적용 가능
  - **Local-windowed attention**
    - 새 토큰 생성 시 이전 모든 토큰에 어텐션을 적용하는 대신, 인접한 고정 크기 윈도우 안에 있는 토큰에만 어텐션 적용
    - 유효 시퀀스 길이가 고정 윈도우 크기로 줄어들어서, KV 캐시와 어텐션 연산이 모두 줄어듦
    - 글로벌 어텐션과 섞어서 쓸 수도 있음
  - **Cross-layer attention & Multi-query attention**
    - 키-값 쌍 개수를 줄여서 KV 캐시의 메모리 사용량을 줄임
    - Cross-layer attention
      - 인접한 레이어들끼리 키와 값 벡터 공유
      - 공유하는 레이어 개수가 늘어나는 만큼 KV 캐시 크기가 줄어듦
    - Multi-query attention:
      - 쿼리 헤드들끼리 키-값 벡터를 공유
  - **Group query attention (GQA)**
    - 멀티 쿼리 어텐션을 발전시킨 방법
    - 모든 쿼리 헤드에 하나의 키-값 쌍 세트만 사용하는 대신, 쿼리 헤드들을 더 작은 그룹으로 나누고 같은 그룹 안에서만 키-값 쌍 공유
    - 쿼리 헤드 수와 키-값 쌍 수 사이에서 더 유연하게 조정 가능
- **KV 캐시 크기 최적화**
  - 추론 시 메모리 병목 현상을 완화하고, 롱 컨텍스트를 다루는 애플리케이션에서 더 큰 배치 크기를 가능하게 해줌
  - 추론 프레임워크 vLLM은 Paged Attention을 도입해서 인기를 얻었음
    - KV 캐시를 비연속적인 블록으로 나눠 메모리 단편화를 줄이고, 메모리를 유연하게 공유해서 LLM 서빙 효율을 개선
  - 그밖에 기법들
    - KV 캐시 양자화
    - 적응형 KV 캐시 압축
    - 선택적 KV 캐시
- **어텐션 연산을 위한 커널 작성**
  - 어텐션 점수 연산 방식을 살펴보고, 연산을 더 효율적으로 만드는 방법을 찾아냄
  - 연산을 실행하는 하드웨어를 고려할 때 가장 효과가 좋음
  - 커널: 특정 칩에 최적화된 코드
  - FlashAttention
    - 어텐션 연산에 최적화된 가장 잘 알려진 커널 중 하나
    - 트랜스포머 기반 모델에서 많이 사용되는 여러 연산을 하나로 융합해서 더 빠르게 실행되도록 했음

**커널과 컴파일러**

- kernel
  - GPU나 TPU 같은 특정 하드웨어 가속기에 맞춰 최적화된 특수 코드
  - 보통 연산량이 많고 반복적으로 실행해야 하는 작업을 처리하도록 만들어짐
  - 종종 병렬로 돌려서 가속기 성능을 최대한 끌어냄
- 행렬 곱셈, 어텐션 연산, 합성곱 연산 같은 일반적 AI 연산은 모두 각기 다른 하드웨어에서 더 효율적인 전용 커널을 가짐
- 커널을 작성하려면 기반 하드웨어 아키텍처를 깊게 이해해야 함
  - 캐시, 글로벌 메모리, 공유 메모리, 레지스터 등 메모리 계층 구조에 대한 이해
  - 여러 레벨 간의 데이터 접근 및 이동 방식
  - 게다가 일반적으로 저수준 프로그래밍 언어로만 작성됨
- 이러한 진입 장벽으로 인해 커널 작성은 소수만 하는 신비로운 기술이었음
- 하지만 추론 최적화 수요가 증가하고 가속기가 널리 퍼지면서 커널 작성 관심도가 높아졌음
- 커널 작성으로 연산 속도를 높이는 4가지 일반적인 기법들
  - **벡터화**
    - 루프나 중첩 루프에서 데이터를 하나씩 처리하는 대신, 메모리에서 연속 데이터를 동시에 처리
    - 데이터 I/O 연산이 적어져 지연 시간이 줄어듦
  - **병렬화**
    - 입력 배열(혹은 n차원 배열)을 동시 처리 가능한 독립적인 덩어리로 나누어 연산 속도를 높임
  - **루프 타일링**
    - 하드웨어의 메모리 레이아웃과 캐시에 맞게 루프의 데이터 접근 순서 최적화
    - 하드웨어에 따라 달라지므로, 효율적인 CPU 타일링이 GPU에서는 잘 작동하지 않을 수도 있음
  - **연산자 융합**
    - 여러 연산자를 단일 패스로 결합해 불필요한 메모리 접근을 회피
    - 다른 3개 기법들에 비해 모델의 특정 연산자와 아키텍처를 더 깊이 알아야 함
- 커널은 하드웨어 아키텍처에 최적화됨
  - 새로운 하드웨어 아키텍처가 나올 때마다 새로운 커널을 개발해야 한다는 뜻
- 모델 스크립트는 해당 모델을 실행하기 위해 해야 할 일련의 연산들을 정의
  - 이 코드를 GPU 같은 하드웨어에서 실행하기 위해 하드웨어 호환 언어로 변환해야 함 (lowering)
- compiler
  - 모델 스크립트가 특정 하드웨어에서 실행되도록 코드를 로어링 하는 도구
  - ML 모델과 하드웨어 사이를 연결
  - 로어링 과정에서 연산들을 대상 하드웨어에서 더 빠르게 돌아가는 전용 커널로 바꿈
  - 독립적인 도구일 수도 있고, ML 및 추론 프레임워크에 통합되기도 함
