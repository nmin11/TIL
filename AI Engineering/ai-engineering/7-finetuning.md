# 7. 파인튜닝

- 파인튜닝: 모델 전체나 일부를 추가로 학습시켜 특정 작업에 맞게 모델을 조정하는 과정
- 파인튜닝은 모델의 가중치를 조정하는 방식으로 모델을 변화시킴
- 파인튜닝을 통해 모델의 다양한 측면을 향상시킬 수 있음
  - 도메인별 능력
  - 안전성 강화
  - 하지만 무엇보다 많이 사용되는 목적은 모델의 지시 수행 능력 향상
- **"언제 파인튜닝을 하고 언제 RAG을 해야 하는가?"**
  - 7장에서는 파인튜닝을 해야 하는 이유와 그렇지 않은 이유를 살펴볼 것
  - 더 나아가 파인튜닝과 다른 방법 중 무엇을 선택할 때 도움이 되는 프레임워크도 소개할 것
- 파인튜닝은 프롬프트 기반 방법에 비해 훨씬 많은 메모리가 필요
  - **PEFT(Parameter-Efficient Fine-Tuning)** : 메모리 효율성을 높이는 파인튜닝 기법
- 파인튜닝은 모델을 직접 학습시키는 과정이므로 ML 지식이 필요
  - 이 책에서는 ML 기초를 전부 다루기 보다는 몇 가지 핵심 개념만 다룰 것

## 7.1 파인튜닝 개요

- 파인튜닝은 기본적인 능력을 갖춘 기본 모델을 토대로 시작
- 파인튜닝의 목표: 모델이 특정 작업을 충분히 잘 수행하도록 만드는 것
- 파인 튜닝은 **전이 학습(Transfer Learning, TL)** 의 한 형태
  - 전이 학습: 한 과제에서 학습한 지식을 다른 과제에 적용하는 방법
  - 딥러닝 초기부터 전이 학습은 학습 데이터를 구하기 어려울 때 좋은 해결책이 되어 왔음
  - 전이 학습은 **표본 효율성(sample efficiency)** 을 높여 모델이 더 적은 예시로도 같은 행동을 학습할 수 있게 함
  - 전이 학습 외에 또 다른 접근법으로는 **특성 기반 전이(feature-based transfer)** 가 있음
    - 모델이 학습한 지식을 고정한 채, 모델에서 Feature를 추출하고, 이 Feature를 기반으로 새로운 모델을 학습시키는 방식
- 이상적인 파인튜닝은 모델이 이미 학습할 것들을 학습한 상태에서 단지 모델의 행동을 다듬는 과정임
- 파인튜닝은 모델 학습 과정의 일부로, 사전 학습의 확장이라고 볼 수 있음
- 모델 학습은 보통 자기 지도 학습 방식의 사전 학습에서 시작됨
  - 모델은 자기 지도 학습을 통해 레이블 없는 대량의 데이터로부터 학습
  - 비싼 작업별 데이터로 사전 학습 모델을 파인튜닝하기 전에, 저렴한 관련 분야 데이터로 자기 지도 학습을 적용해볼 수 있음
  - 이러한 자기 지도 파인튜닝을 **지속적 사전 학습(Continual Pretraining)** 이라고 부름
- **Infilling finetuning**
  - 지도 파인튜닝을 통해 다음 토큰을 예측하거나 빈칸을 채우도록 모델을 조정하는 방식
  - 자기회귀 방식으로 사전 학습된 모델이라도 infilling finetuning을 적용할 수 있음
    - 바로 다음 토큰만 예측하는 것이 아닌, 빈칸을 채우는 것처럼 학습시키는 것이 가능하다는 뜻
- 자기 지도 학습 이후
  - 자기 지도 학습을 통해 모델은 방대한 지식을 얻지만, 사용자가 원하는 작업에 바로 활용하기는 어려움
  - 지도 파인튜닝을 통해, 모델과 사람의 실제 선호와의 격차를 줄여야 함
- 지도 파인튜닝
  - 고품질 주석 데이터를 활용해 모델을 사람의 사용 방식과 선호도에 맞게 조정하는 과정
  - (입력, 출력) 쌍으로 모델을 학습시킴
  - 입력은 지시가 될 수 있으며, 출력은 응답이 될 수 있음
  - 응답은 책 요약처럼 개방형 형태일 수도 있고, 분류 작업처럼 폐쇄형 형태일 수도 있음
- 선호도 파인튜닝
  - 강화 학습을 통해 사람의 선호도를 최대화하는 응답을 생성하도록 조정하는 과정
  - 주로 (지시, 선호 응답, 비선호 응답) 형식의 비교 데이터가 필요
- **Long-context finetuning**
  - 컨텍스트 길이를 늘리는 파인튜닝
  - 보통 위치 임베딩 조정 같은 모델 구조의 수정이 필요
  - 롱 시퀀스는 토큰의 가능한 위치가 많다는 뜻이며, 이를 위치 임베딩이 처리할 수 있어야 하므로
  - 다른 파인튜닝 기법보다 까다로운 방법
  - 결과로 얻어진 모델은 숏 시퀀스에서 성능이 오히려 떨어질 수 있음
- 파인튜닝은 모델 개발자와 애플리케이션 개발자 모두 수행 가능
  - 모델 개발자: 주로 모델 출시 전 다양한 파인튜닝 기법으로 모델을 사후 학습시킴
  - 애플리케이션 개발자: 대개 이미 사후 학습된 모델을 더 정교하게 파인튜닝

## 7.2 파인튜닝이 필요한 경우

- 파인튜닝은 프롬프트 기반 방법에 비해 많은 데이터, 고사양 하드웨어 뿐만 아니라 ML 전문가도 필요
- 그러므로 보통 프롬프트 기반 방법을 충분히 시도한 후에 파인튜닝을 시도하는 것이 일반적
- 그러나 실전에서는 두 접근법을 함께 활용해야 하는 경우가 많음

### 파인튜닝을 해야 하는 이유

- 파인튜닝의 주요 목적: 일반 능력 및 특정 작업 수행 능력 모두 향상시키는 것
- 범용 모델의 경우 특정 작업에 대해서는 성능이 떨어질 수 있으므로 파인 튜닝이 필요
- 편향 완화
  - 파인튜닝의 흥미로운 사례 중 하나
  - 파인튜닝 과정 중 신중하게 선별된 데이터를 사용해 기본 모델의 편향을 상쇄할 수 있음
- 큰 모델보다는 작은 모델을 파인튜닝하는 경우가 훨씬 일반적
  - 메모리가 적어 파인튜닝이 쉽고, 운영 환경에서도 저렴하고 빠르게 이용 가능
- **증류(distillation)**
  - 큰 모델이 생성한 데이터로 작은 모델을 학습시켜, 마치 큰 모델처럼 작동하게 만드는 방식
  - 큰 모델의 지식을 더 작은 효율적으로 전달
  - 복잡한 혼합물에서 핵심 성분을 추출하는 '증류' 과정과 유사
  - 동일한 작업에서 특정 작업에 파인튜닝된 작은 모델이 더 큰 기본 모델보다 뛰어난 성능을 보일 수도 있음
- 파운데이션 모델 초기에는 강력한 모델들의 상업적 이용이 제한되었지만, 오픈 소스 커뮤니티가 활발해지면서 파인튜닝은 매력적인 선택지가 되었음

### 파인튜닝을 하지 말아야 하는 이유

**1. 특정 작업 성능은 향상될 수 있지만, 다른 작업에서 오히려 성능이 떨어질 수 있음**

- 다양한 프롬프트를 사용해야 한다면 답답해질 수 있음
- 대응 방법들
  - 포함하는 모든 질의에 대해 파인튜닝
  - 작업별로 다른 모델 사용
  - 별도 모델들을 합치는 모델 병합
- 프로젝트 실험을 막 시작했다면, 파인튜닝은 가장 먼저 시도할 방법이 아님
  - 초기 투자가 크고 지속적인 관리가 필요
  - 주석이 달린 데이터를 수동으로 모으는 과정에 비용이 많이 듦

**2. 모델 학습 방법에 대한 지식 필요**

- 파인튜닝 대상 모델을 찾기 위해 기본 모델을 평가해야 함
- 파인튜닝 프레임워크 및 API의 도움을 받을 수 있음
  - 하지만 여전히 다양한 학습 파라미터를 이해해야 함
  - 학습 과정에 대한 모니터링도 필요
  - 문제 발생 시 디버깅 능력도 필요
- 예시: 옵티마이저 작동 방식, 적절한 학습률, 필요한 학습 데이터 양, 과적합/과소적합 해결 방법, 전체 과정에서 모델을 평가하는 방법 등

**3. 파인튜닝된 모델 서빙 방법**

- 직접 호스팅할 것인지 아님 API 서비스를 사용할지
- 호스팅 및 운영 방법에 대한 숙지 필요
- 모델을 모니터링하고 유지보수하며 업데이트하기 위한 정책과 예산을 수립해야 한다는 점
- 파인튜닝된 모델을 지속적으로 개선하는 와중에도 새로운 기본 모델들이 빠른 속도로 개발되고 있을 것
- 그렇다면 언제 또 새로운 모델을 기반으로 파인튜닝할지 결정할 것인가?

**현실적 관점**

- 실제로 더 좋은 모델로 바꿔도 성능 향상은 미미한 경우가 많음
- 그러므로 5장의 모범 사례를 따라 프롬프팅부터 시작하는 것이 좋음
  - 체계적으로 정리된 프롬프팅을 시도해보자!

*도메인 특화 작업 파인튜닝*

- 범용 모델을 특정 작업에 맞게 파인튜닝한다는 주장을 조심할 것
- 범용 모델이 강력해질수록 특정 도메인 작업에도 능숙해져 도메인 특화 모델보다 오히려 더 나은 성능을 보일 수 있음

**파인튜닝과 프롬프팅의 조화**

- 파인튜닝과 프롬프팅 실험 모두 체계적인 접근 방식이 필요
- 프롬프팅의 평가 파이프라인, 데이터 주석 가이드라인, 실험 추적 방법 등은 파인튜닝의 기반이 되어줌
- 프롬프트 캐싱
  - 프롬프트 캐싱 도입 전에는 파인튜닝의 토큰 사용 최적화에 큰 도움이 되었음
  - 프롬프트 캐싱을 통해 반복적인 프롬프트 부분을 저장 및 재사용할 수 있음
  - 하지만 여전히 프롬프트와 함께 사용 가능한 예시 수는 최대 컨텍스트 길이에 제한을 받고, 파인튜닝에서는 예시 수에 제한이 없음

### 파인튜닝과 RAG

- 프롬프팅으로 모델 성능을 최대한 끌어올렸다면, RAG과 파인튜닝 중에서 고민하게 됨
- 이 선택은 모델 오류 원인이 정보 부족인지, 행동 방식 문제인지에 따라 달라짐
- 정보가 부족하다면 RAG 시스템이 효과적
- 정보 부족으로 인해 오류가 발생하는 상황 2가지
  - 모델이 정보를 가지고 있지 않은 경우
  - 모델의 정보가 오래된 경우
- 시사 문제, 최신 정보 관련 질의에는 RAG이 파인튜닝된 모델보다 더 좋은 성능을 보임
- 행동 방식에 문제가 있는 경우
  - 모델의 출력이 사실 관계는 맞지만 요청 작업과 전혀 무관한 경우
  - 모델이 원하는 출력 형식을 제대로 따르지 못하는 경우
- **Semantic parsing**
  - 모델이 정해진 형식에 맞춰 출력을 생성하는 능력
  - 자연어를 JSON 같은 구조화된 형식으로 변환하는 과정
  - 이 작업에는 종종 파인튜닝이 필요
- 요약: **파인튜닝을 형식을 위한 것이고, RAG은 사실을 위한 것**
- 모델이 정보와 행동에서 모두 문제가 있다면 RAG부터 시작하는 것이 좋음
  - 더 쉽게 접근 가능하기 때문에
  - RAG을 구현할 때도 벡터 DB 보다는 BM25 같은 단순 키워드 기반 검색부터 시작하는 것이 좋음
- RAG은 파인튜닝보다 더 큰 성능 향상을 가져올 수 있음
- RAG과 파인튜닝은 상호 배타적이지 않으며, 둘을 함께 사용함으로써 애플리케이션의 성능을 극대화할 수 있음
- 모델 조정의 전체적인 과정 추천
  - 프롬프트만으로 모델이 원하는 작업을 수행하도록 시도 (엔지니어링 모범 사례 활용, 프롬프트 버전 관리)
  - 프롬프트에 더 많은 예시 추가 (1~50개 정도 사례)
  - RAG 연결 (키워드 기반 검색 같은 기본적인 방법부터 시작)
  - 모델의 오류 유형에 따라 취사 선택
    - 정보 오류 → 임베딩 기반 검색을 적용한 고급 RAG
    - 형식 및 안전성 관련 오류 → 파인튜닝
  - 더 큰 성능 향상을 위해 RAG과 파인튜닝을 함께 적용
