# 7. 파인튜닝

- 파인튜닝: 모델 전체나 일부를 추가로 학습시켜 특정 작업에 맞게 모델을 조정하는 과정
- 파인튜닝은 모델의 가중치를 조정하는 방식으로 모델을 변화시킴
- 파인튜닝을 통해 모델의 다양한 측면을 향상시킬 수 있음
  - 도메인별 능력
  - 안전성 강화
  - 하지만 무엇보다 많이 사용되는 목적은 모델의 지시 수행 능력 향상
- **"언제 파인튜닝을 하고 언제 RAG을 해야 하는가?"**
  - 7장에서는 파인튜닝을 해야 하는 이유와 그렇지 않은 이유를 살펴볼 것
  - 더 나아가 파인튜닝과 다른 방법 중 무엇을 선택할 때 도움이 되는 프레임워크도 소개할 것
- 파인튜닝은 프롬프트 기반 방법에 비해 훨씬 많은 메모리가 필요
  - **PEFT(Parameter-Efficient Fine-Tuning)** : 메모리 효율성을 높이는 파인튜닝 기법
- 파인튜닝은 모델을 직접 학습시키는 과정이므로 ML 지식이 필요
  - 이 책에서는 ML 기초를 전부 다루기 보다는 몇 가지 핵심 개념만 다룰 것

## 7.1 파인튜닝 개요

- 파인튜닝은 기본적인 능력을 갖춘 기본 모델을 토대로 시작
- 파인튜닝의 목표: 모델이 특정 작업을 충분히 잘 수행하도록 만드는 것
- 파인 튜닝은 **전이 학습(Transfer Learning, TL)** 의 한 형태
  - 전이 학습: 한 과제에서 학습한 지식을 다른 과제에 적용하는 방법
  - 딥러닝 초기부터 전이 학습은 학습 데이터를 구하기 어려울 때 좋은 해결책이 되어 왔음
  - 전이 학습은 **표본 효율성(sample efficiency)** 을 높여 모델이 더 적은 예시로도 같은 행동을 학습할 수 있게 함
  - 전이 학습 외에 또 다른 접근법으로는 **특성 기반 전이(feature-based transfer)** 가 있음
    - 모델이 학습한 지식을 고정한 채, 모델에서 Feature를 추출하고, 이 Feature를 기반으로 새로운 모델을 학습시키는 방식
- 이상적인 파인튜닝은 모델이 이미 학습할 것들을 학습한 상태에서 단지 모델의 행동을 다듬는 과정임
- 파인튜닝은 모델 학습 과정의 일부로, 사전 학습의 확장이라고 볼 수 있음
- 모델 학습은 보통 자기 지도 학습 방식의 사전 학습에서 시작됨
  - 모델은 자기 지도 학습을 통해 레이블 없는 대량의 데이터로부터 학습
  - 비싼 작업별 데이터로 사전 학습 모델을 파인튜닝하기 전에, 저렴한 관련 분야 데이터로 자기 지도 학습을 적용해볼 수 있음
  - 이러한 자기 지도 파인튜닝을 **지속적 사전 학습(Continual Pretraining)** 이라고 부름
- **Infilling finetuning**
  - 지도 파인튜닝을 통해 다음 토큰을 예측하거나 빈칸을 채우도록 모델을 조정하는 방식
  - 자기회귀 방식으로 사전 학습된 모델이라도 infilling finetuning을 적용할 수 있음
    - 바로 다음 토큰만 예측하는 것이 아닌, 빈칸을 채우는 것처럼 학습시키는 것이 가능하다는 뜻
- 자기 지도 학습 이후
  - 자기 지도 학습을 통해 모델은 방대한 지식을 얻지만, 사용자가 원하는 작업에 바로 활용하기는 어려움
  - 지도 파인튜닝을 통해, 모델과 사람의 실제 선호와의 격차를 줄여야 함
- 지도 파인튜닝
  - 고품질 주석 데이터를 활용해 모델을 사람의 사용 방식과 선호도에 맞게 조정하는 과정
  - (입력, 출력) 쌍으로 모델을 학습시킴
  - 입력은 지시가 될 수 있으며, 출력은 응답이 될 수 있음
  - 응답은 책 요약처럼 개방형 형태일 수도 있고, 분류 작업처럼 폐쇄형 형태일 수도 있음
- 선호도 파인튜닝
  - 강화 학습을 통해 사람의 선호도를 최대화하는 응답을 생성하도록 조정하는 과정
  - 주로 (지시, 선호 응답, 비선호 응답) 형식의 비교 데이터가 필요
- **Long-context finetuning**
  - 컨텍스트 길이를 늘리는 파인튜닝
  - 보통 위치 임베딩 조정 같은 모델 구조의 수정이 필요
  - 롱 시퀀스는 토큰의 가능한 위치가 많다는 뜻이며, 이를 위치 임베딩이 처리할 수 있어야 하므로
  - 다른 파인튜닝 기법보다 까다로운 방법
  - 결과로 얻어진 모델은 숏 시퀀스에서 성능이 오히려 떨어질 수 있음
- 파인튜닝은 모델 개발자와 애플리케이션 개발자 모두 수행 가능
  - 모델 개발자: 주로 모델 출시 전 다양한 파인튜닝 기법으로 모델을 사후 학습시킴
  - 애플리케이션 개발자: 대개 이미 사후 학습된 모델을 더 정교하게 파인튜닝

## 7.2 파인튜닝이 필요한 경우

- 파인튜닝은 프롬프트 기반 방법에 비해 많은 데이터, 고사양 하드웨어 뿐만 아니라 ML 전문가도 필요
- 그러므로 보통 프롬프트 기반 방법을 충분히 시도한 후에 파인튜닝을 시도하는 것이 일반적
- 그러나 실전에서는 두 접근법을 함께 활용해야 하는 경우가 많음

### 파인튜닝을 해야 하는 이유

- 파인튜닝의 주요 목적: 일반 능력 및 특정 작업 수행 능력 모두 향상시키는 것
- 범용 모델의 경우 특정 작업에 대해서는 성능이 떨어질 수 있으므로 파인 튜닝이 필요
- 편향 완화
  - 파인튜닝의 흥미로운 사례 중 하나
  - 파인튜닝 과정 중 신중하게 선별된 데이터를 사용해 기본 모델의 편향을 상쇄할 수 있음
- 큰 모델보다는 작은 모델을 파인튜닝하는 경우가 훨씬 일반적
  - 메모리가 적어 파인튜닝이 쉽고, 운영 환경에서도 저렴하고 빠르게 이용 가능
- **증류(distillation)**
  - 큰 모델이 생성한 데이터로 작은 모델을 학습시켜, 마치 큰 모델처럼 작동하게 만드는 방식
  - 큰 모델의 지식을 더 작은 효율적으로 전달
  - 복잡한 혼합물에서 핵심 성분을 추출하는 '증류' 과정과 유사
  - 동일한 작업에서 특정 작업에 파인튜닝된 작은 모델이 더 큰 기본 모델보다 뛰어난 성능을 보일 수도 있음
- 파운데이션 모델 초기에는 강력한 모델들의 상업적 이용이 제한되었지만, 오픈 소스 커뮤니티가 활발해지면서 파인튜닝은 매력적인 선택지가 되었음

### 파인튜닝을 하지 말아야 하는 이유

**1. 특정 작업 성능은 향상될 수 있지만, 다른 작업에서 오히려 성능이 떨어질 수 있음**

- 다양한 프롬프트를 사용해야 한다면 답답해질 수 있음
- 대응 방법들
  - 포함하는 모든 질의에 대해 파인튜닝
  - 작업별로 다른 모델 사용
  - 별도 모델들을 합치는 모델 병합
- 프로젝트 실험을 막 시작했다면, 파인튜닝은 가장 먼저 시도할 방법이 아님
  - 초기 투자가 크고 지속적인 관리가 필요
  - 주석이 달린 데이터를 수동으로 모으는 과정에 비용이 많이 듦

**2. 모델 학습 방법에 대한 지식 필요**

- 파인튜닝 대상 모델을 찾기 위해 기본 모델을 평가해야 함
- 파인튜닝 프레임워크 및 API의 도움을 받을 수 있음
  - 하지만 여전히 다양한 학습 파라미터를 이해해야 함
  - 학습 과정에 대한 모니터링도 필요
  - 문제 발생 시 디버깅 능력도 필요
- 예시: 옵티마이저 작동 방식, 적절한 학습률, 필요한 학습 데이터 양, 과적합/과소적합 해결 방법, 전체 과정에서 모델을 평가하는 방법 등

**3. 파인튜닝된 모델 서빙 방법**

- 직접 호스팅할 것인지 아님 API 서비스를 사용할지
- 호스팅 및 운영 방법에 대한 숙지 필요
- 모델을 모니터링하고 유지보수하며 업데이트하기 위한 정책과 예산을 수립해야 한다는 점
- 파인튜닝된 모델을 지속적으로 개선하는 와중에도 새로운 기본 모델들이 빠른 속도로 개발되고 있을 것
- 그렇다면 언제 또 새로운 모델을 기반으로 파인튜닝할지 결정할 것인가?

**현실적 관점**

- 실제로 더 좋은 모델로 바꿔도 성능 향상은 미미한 경우가 많음
- 그러므로 5장의 모범 사례를 따라 프롬프팅부터 시작하는 것이 좋음
  - 체계적으로 정리된 프롬프팅을 시도해보자!

*도메인 특화 작업 파인튜닝*

- 범용 모델을 특정 작업에 맞게 파인튜닝한다는 주장을 조심할 것
- 범용 모델이 강력해질수록 특정 도메인 작업에도 능숙해져 도메인 특화 모델보다 오히려 더 나은 성능을 보일 수 있음

**파인튜닝과 프롬프팅의 조화**

- 파인튜닝과 프롬프팅 실험 모두 체계적인 접근 방식이 필요
- 프롬프팅의 평가 파이프라인, 데이터 주석 가이드라인, 실험 추적 방법 등은 파인튜닝의 기반이 되어줌
- 프롬프트 캐싱
  - 프롬프트 캐싱 도입 전에는 파인튜닝의 토큰 사용 최적화에 큰 도움이 되었음
  - 프롬프트 캐싱을 통해 반복적인 프롬프트 부분을 저장 및 재사용할 수 있음
  - 하지만 여전히 프롬프트와 함께 사용 가능한 예시 수는 최대 컨텍스트 길이에 제한을 받고, 파인튜닝에서는 예시 수에 제한이 없음

### 파인튜닝과 RAG

- 프롬프팅으로 모델 성능을 최대한 끌어올렸다면, RAG과 파인튜닝 중에서 고민하게 됨
- 이 선택은 모델 오류 원인이 정보 부족인지, 행동 방식 문제인지에 따라 달라짐
- 정보가 부족하다면 RAG 시스템이 효과적
- 정보 부족으로 인해 오류가 발생하는 상황 2가지
  - 모델이 정보를 가지고 있지 않은 경우
  - 모델의 정보가 오래된 경우
- 시사 문제, 최신 정보 관련 질의에는 RAG이 파인튜닝된 모델보다 더 좋은 성능을 보임
- 행동 방식에 문제가 있는 경우
  - 모델의 출력이 사실 관계는 맞지만 요청 작업과 전혀 무관한 경우
  - 모델이 원하는 출력 형식을 제대로 따르지 못하는 경우
- **Semantic parsing**
  - 모델이 정해진 형식에 맞춰 출력을 생성하는 능력
  - 자연어를 JSON 같은 구조화된 형식으로 변환하는 과정
  - 이 작업에는 종종 파인튜닝이 필요
- 요약: **파인튜닝을 형식을 위한 것이고, RAG은 사실을 위한 것**
- 모델이 정보와 행동에서 모두 문제가 있다면 RAG부터 시작하는 것이 좋음
  - 더 쉽게 접근 가능하기 때문에
  - RAG을 구현할 때도 벡터 DB 보다는 BM25 같은 단순 키워드 기반 검색부터 시작하는 것이 좋음
- RAG은 파인튜닝보다 더 큰 성능 향상을 가져올 수 있음
- RAG과 파인튜닝은 상호 배타적이지 않으며, 둘을 함께 사용함으로써 애플리케이션의 성능을 극대화할 수 있음
- 모델 조정의 전체적인 과정 추천
  - 프롬프트만으로 모델이 원하는 작업을 수행하도록 시도 (엔지니어링 모범 사례 활용, 프롬프트 버전 관리)
  - 프롬프트에 더 많은 예시 추가 (1~50개 정도 사례)
  - RAG 연결 (키워드 기반 검색 같은 기본적인 방법부터 시작)
  - 모델의 오류 유형에 따라 취사 선택
    - 정보 오류 → 임베딩 기반 검색을 적용한 고급 RAG
    - 형식 및 안전성 관련 오류 → 파인튜닝
  - 더 큰 성능 향상을 위해 RAG과 파인튜닝을 함께 적용

## 7.3 메모리 병목 현상

- 파인튜닝은 메모리를 많이 사용
- 따라서 많은 파인튜닝 기법이 메모리 사용량을 최소화하는 데 중점을 둠
- 메모리 계산을 위해 ML 내부 동작 원리 및 시스템 수준의 컴퓨팅 개념을 깊이 다루게 될 것
  - 기술적으로 다소 어려울 수 있는 부분들

*메모리 병목 현상 이해를 위한 핵심 내용*

1. 파운데이션 모델의 큰 규모로 인해, 추론과 파인튜닝 모두 메모리가 주요 병목 지점이 됨
   - 파인튜닝 > 추론
2. 파인튜닝 중 모델 메모리 사용량에 큰 영향을 미치는 요소들:
   - 전체 파라미터 수
   - 학습 가능한 파라미터 수
   - 수치 표현 방식
3. 학습 가능한 파라미터
   - 많을수록 메모리 사용량이 증가하고, 줄이면 메모리 사용량도 줄일 수 있음
   - **파라미터 효율적 파인튜닝(Parameter Efficient Fine Tuning, PEFT)** 의 핵심 아이디어
4. 양자화
   - 큰 비트 형식을 적은 비트 형식으로 변환해 모델의 메모리 사용량을 줄이는 간단하고 효과적인 방법
5. 추론은 가능한 한 적은 비트를 사용해서 수행
6. 학습은 수치 정밀도에 더 민감
   - 그러므로 더 낮은 정밀도로 모델을 학습하는 것은 어려움
   - 주로 혼합 정밀도 방식을 사용해서 일부 연산은 높은 정밀도로, 나머지는 낮은 정밀도로 수행

### 역전파와 학습 가능한 파라미터

- **학습 가능한 파라미터 수(trainable parameters)**
  - 파인튜닝 과정에서 모델의 메모리 사용량을 결정하는 핵심 요소
  - 정의: 파인튜닝 중 업데이트될 수 있는 파라미터의 수
- 파라미터들의 업데이트 여부
  - 사전 학습 과정: 모든 모델 파라미터가 업데이트 됨
  - 추론 과정: 어떤 파라미터도 업데이트되지 않음
  - 파인튜닝 과정: 일부 또는 모든 모델 파라미터가 업데이트될 수 있음
- 고정된 파라미터: 변경되지 않고 유지되는 파라미터
- 각 학습 가능한 파라미터에 필요한 메모리는 모델 학습 방식에서 비롯됨
- **역전파(Backpropagation)**
  - 신경망 학습에서 주로 사용되는 알고리즘
- 역전파의 학습 단계
   1. **순방향 패스(forward pass)** : 입력값에서 출력값을 계산하는 과정
   2. **역방향 패스(backward pass)** : 순방향 패스에서 집계된 신호를 활용해 모델 가중치를 업데이트하는 과정
- 추론 과정에서는 순방향 패스만 실행, 학습 과정에서는 두 패스 모두 실행
- 역방향 패스의 작동 방식
   1. 순방향 패스에서 계산된 출력값과 예상 출력값 비교 → 둘이 다른 경우 모델의 실수이므로 파라미터 조정 필요
   2. 각 학습 가능 파라미터가 실수에 얼마나 기여했는지 계산 → 이 값을 **gradient** 라고 부름
   3. 해당 그래디언트 값을 사용해 학습 가능 파라미터 값 조정 → 옵티마이저가 파라미터를 얼마나 재조정할지 결정
- 역방향 패스에서 각 학습 가능 파라미터는 그래디언트와 옵티마이저 스테이트 같은 추가 값들을 동반
  - 그러므로 이런 추가 값들을 위한 메모리도 더 많이 필요해짐

### 메모리 계산

- 모델의 메모리 사용량은 모델 자체뿐 아니라 작업 부하, 최적화 기법에 따라 달라짐
- 모든 작업 부하 및 최적화 기법을 헤아리는 것은 현실적으로 불가능하기 때문에, 여기에서는 대략적인 계산식만 다룰 것

**추론에 필요한 메모리**

- 추론 중에는 순방향 패스만 실행
- 모델 가중치를 위한 메모리 필요
- 필요한 메모리 = N(파라미터 수) * M(파라미터에 필요한 메모리)
- 더 필요한 메모리들
  - 활성화(activations) 값을 위한 메모리
  - 트랜스포머 모델의 경우 어텐션 메커니즘을 위한 키-값 벡터(key-value vectors)에도 메모리 필요
- 추가 메모리 사용을 고려한 메모리 = N(파라미터 수) * M(파라미터에 필요한 메모리) * 1.2

**학습에 필요한 메모리**

- **학습 메모리 = 모델 가중치 + 활성화 + 그래디언트 + 옵티마이저 스테이트**
- 역방향 패스 동안 각 학습 파라미터는 그래디언트용 값 하나와 함께 0~2개의 옵티마이저 스테이트가 필요
  - 기본 SGD 옵티마이저: 스테이트 없음
  - 모멘텀 옵티마이저: 학습 가능 파라미터 당 1개 저장
  - Adam 옵티마이저: 학습 가능 파라미터 당 2개 저장
- 활성화 메모리 관련 유의점: 그래디언트 계산을 위해 활성화를 저장하면 모델 가중치 메모리를 훨씬 웃돌게 될 수 있음
- **gradient checkpointing**
  - 순방향 패스에서 활성화 값을 저장하지 않고, 역방향 패스에서 필요할 때마다 다시 계산
  - 메모리를 절약할 수 있지만 계산 시간이 더 오래 걸림
  - **활성화 재계산(activation recomputation)** 이라고도 불림

### 수치 표현 방식

- 수치 표현 방식은 모델의 전체 메모리 사용량에 직접적인 영향을 미침
- 신경망의 수치 값은 전통적으로 부동소수점 수로 표현됨
  - 가장 일반적인 부동소수점 형식은 전기전자기술자협회(IEEE)의 부동소수점 산술 표준(IEEE 754)을 준수하는 FP 계열
    - FP32: 32비트를 사용해 부동소수점 수를 표현 (single precision, 단정밀도)
    - FP64: 64비트를 사용해 부동소수점 수를 표현 (double precision, 배정밀도)
    - FP16: 16비트를 사용해 부동소수점 수를 표현 (half precision, 반정밀도)
- 신경망에서는 FP32와 FP16이 가장 흔하게 사용됨
- BFloat16(BF16), TensorFloat-32(TF32) 또한 AI 워크로드에서 인기 있음
  - BF16은 구글이 TPU에서 AI 성능을 높이기 위해 사용했음
  - TF32는 엔비디아에서 GPU를 위해 설계
- 정수 표현 방식 또한 점점 더 인기를 얻고 있음
  - INT8(8비트 정수), INT4(4비트 정수)
- 각 부동소수점 형식은 보통 숫자의 부호를 나타내기 위해 1비트를 사용
  - 나머지 비트는 범위와 정밀도로 나뉨
  - 범위: 형식이 표현할 수 있는 값의 범위 결정
  - 정밀도: 숫자를 얼마나 정확하게 표현할 수 있는지 결정
- 보통 비트 수가 많은 형식일수록 정밀도가 높다고 간주됨
- 고정밀도 형식의 숫자를 저정밀도 형식으로 변환하면 정밀도가 낮아짐

### 양자화(Quantization)

- 모델 값을 표현하는 데 필요한 비트 수가 적을수록 모델의 메모리 사용량도 줄어듦
- 양자화 = 정밀도를 낮추는 것
- 양자화는 모델의 메모리 사용량을 줄이는 저렴하고도 효과적인 방법
- ML 분야에서 저정밀도는 일반적으로 표준 FP32 보다 적은 비트를 가진 모든 형식을 의미
- 양자화를 위해 결정해야 할 것들
  - **무엇을 양자화할 것인가?**
    - 메모리를 가장 많이 차지하는 것보다, 성능 저하 없이 적용 가능한 부분을 고려해야 함
    - 대체로 활성화 양자화보다는 가중치 양자화를 더 많이 사용함
  - **언제 양자화할 것인가?**
    - 학습 중이나 학습 후에 진행 가능
    - **학습 후 양자화(Post-Training Quantization, PTQ)** : 모델이 완전 학습된 후 양자화
      - 가장 많이 사용되는 방식

**추론 양자화**

- 8비트 이하로 내려가면 FP8, FP4 등의 미니플로트 형식을 사용해 부동소수점 유지 가능
  - 하지만 일반적으로는 INT8, INT4 등 정수 형식으로 변환
- 양자화는 효과적이지만 무한정 적용할 수는 없음
  - 이론상 1비트보다 작게 양자화할 수 없음
- 정밀도 감소의 장점
  - 더 큰 배치 크기를 쓰므로 모델이 더 많은 입력을 병렬 처리
  - 계산 속도를 높여 추론 지연 시간과 학습 시간을 더욱 단축
  - 하지만 형식 변환을 위한 계산에 시간을 더 많이 소모할 수도 있으므로 주의 필요
- 정밀도 감소의 단점
  - 각 변환이 작은 값 변화를 일으키는 경우가 많음
  - 이러한 작은 변화들이 축적되면 모델의 전체적인 성능 저하로 이어질 수 있음
- 추론 학습에는 낮은 정밀도 활용이 표준이 되었음
  - 학습에 높은 정밀도를 사용해 모델 성능을 최대화하고, 추론에는 낮은 정밀도 사용

**학습 양자화**

- PTQ만큼 보편적이진 않지만 점점 인기를 얻고 있음
- 학습 양자화의 2가지 목표
   1. **추론 과정에서 낮은 정밀도에서도 우수한 성능을 보이는 모델을 만드는 것**
   2. **학습 시간과 비용을 줄이는 것**
- **양자화 인식 학습(Quantization-Aware Training, QAT)**
  - 목표: 추론 시 낮은 정밀도에서도 품질이 좋은 모델을 만드는 것
  - 모델은 학습 중에 낮은 정밀도 동작을 시뮬레이션하므로, 낮은 정밀도에서 품질 높은 출력을 생성하도록 학습 가능
  - 그러나 계산이 여전히 높은 정밀도로 이루어지기 때문에 학습 시간을 줄이지는 않음
  - 오히려 낮은 정밀도 동작을 시뮬레이션하는 추가 작업으로 인해 학습 시간 증가
- 모델을 처음부터 낮은 정밀도로 학습하면 정밀도 불일치 문제를 해결하면서도 학습 효율까지 향상시킬 수 있음
- 하지만 정밀도 낮은 학습은 역전파 과정이 낮은 정밀도에 더 민감하게 반응하므로 어려움
- 낮은 정밀도 학습에서 쓰이는 **혼합 정밀도(mixed precision)** 방식
  - 가중치 사본은 높은 정밀도로 유지, 그래디언트나 활성화 등 다른 값은 낮은 정밀도로 유지
  - 가중치 중에서도 민감도에 따라 정밀도 분리
- **자동 혼합 정밀도(Automatic Mixed Precision, AMP)**
  - 모델의 어떤 부분을 어떤 정밀도로 할지 결정해줌
  - 많은 ML 프레임워크에서 지원
- 학습의 단계마다 다른 정밀도 수준을 사용하는 것도 가능
