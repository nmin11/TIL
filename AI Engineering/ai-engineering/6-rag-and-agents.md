# 6. RAG와 에이전트

- 이번 장에서는 각 질의에 적절한 컨텍스트를 구성하는 방법에 초점을 둘 것
- 모델의 지시는 모든 질의에 공통 적용되지만, 컨텍스트는 각 질의에 따라 달라짐
- 컨텍스트를 구성할 때 주로 쓰는 2가지 패턴
  - **검색 증강 생성(Retrieval-Augmented Generation, RAG)**
    - 모델이 외부 데이터 소스에서 관련 정보를 검색할 수 있게 해줌
    - 주로 컨텍스트를 구성하는 데 사용됨
  - **에이전트(Agent)**
    - 웹 검색이나 API 도구 등을 사용해 정보 수집
    - 컨텍스트 구성 이상의 기능 수행 가능
    - 모델이 많은 부분을 자동화할 수 있도록 현실 세계와 직접 상호작용

## 6.1 RAG

- RAG: 외부 메모리 소스에서 관련 정보를 검색해 모델의 생성 능력을 향상시키는 기술
  - 외부 메모리 소스는 내부 DB, 사용자의 이전 채팅 세션, 인터넷 등이 될 수 있음
- **검색 후 생성(retrieve-then-generate)** 패턴
  - Reading Wikipedia to Answer Open-Domain Questions 연구에서 처음 소개되었음
  - 시스템은 먼저 질의와 가장 관련성 높은 5개 위키피디아 페이지를 검색하고, 모델이 페이지 정보를 통해 응답을 생성
- 검색 증강 생성이라는 용어는 Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks 연구에서 처음 사용됨
  - 가용한 지식 전부를 모델에 직접 입력하기 어려운 지식 집약적 작업을 해결하기 위한 방법이었음
- RAG를 통하면 검색기가 주어진 질의와 가장 관련성 높다고 판단한 정보만 모델에 입력
- RAG를 통해 관련 정보에 접근시키고, 이를 통해 환각을 줄일 수 있음
- RAG는 모든 질의에 동일한 컨텍스트를 사용하지 않고, 각 질의에 특화된 컨텍스트를 만들어내는 기술이라고 볼 수 있음
- RAG는 사용자 데이터 관리에도 유용
  - 특정 사용자에 관한 질의가 들어올 때만 해당 사용자 데이터를 컨텍스트에 포함시키는 방식
- 많은 사람들이 충분히 긴 컨텍스트로 RAG의 필요성을 제거할 수 있으리라 생각하지만 한계점들이 존재함
  - 모델의 컨텍스트가 아무리 길어도 더 긴 컨텍스트가 필요해질 것
  - 모델이 롱 컨텍스트를 처리할 수 있어도 반드시 잘 활용하지는 못할 수도 있음
- Anthropic의 경우, 토큰 분량이 200,000 미만이라면 전체 지식 베이스를 프롬프트에 포함시켜도 된다고 조언했음

### RAG 아키텍처

- RAG 시스템의 구성요소
  - 검색기: 외부 메모리 소스에서 정보 검색
  - 생성 모델: 검색된 정보를 기반으로 응답 생성
- 많은 팀들이 이미 만들어진 검색기와 생성 모델을 활용해 RAG 시스템을 구축
  - 하지만 RAG 시스템 전체를 파인튜닝하면 성능이 훨씬 좋아질 것
- RAG 시스템의 성공은 검색기의 품질에 달려 있음
- 검색기의 2가지 주요 기능
  - **indexing**
  - **query**
- 작동 방식 예시
  - 문서 DB의 경우, 각 문서를 **chunk**로 분할
  - 각 질의의 목표는 가장 관련성 높은 데이터 청크를 검색하는 것
  - 데이터 청크를 사용자 프롬프트와 합치기 위해 후처리 작업이 약간 필요
  - 이렇게 만들어진 최종 프롬프트가 생성 모델에 입력됨

### 검색 알고리즘

- 정보 검색 기술은 검색 엔진, 추천 시스템, 로그 분석 등 다양한 분야의 근간이 됨
- 핵심적으로 검색은 질의에 대한 관련성을 기준으로 순위를 매기는 방식
- 따라서 검색 알고리즘은 관련성 점수 계산 방법에 따라 달라짐
- 일반적인 검색 방법 2가지
  - **용어 기반 검색**
  - **임베딩 기반 검색**

**용어 기반 검색**

- 질의에 관련된 문서를 찾는 가장 간단한 방법은 키워드를 사용하는 것
- 이런 접근 방식을 **어휘적 검색(lexical search)** 이라고 불리기도 함
- 키워드 검색 방식의 문제점 2가지
  - 예상보다 많은 문서에서 결과가 집계되고, 모델은 이를 전부 컨텍스트로 활용하기에 충분한 공간을 갖지 못하게 될 수 있음
    - 용어가 자주 나오는 문서들만 선택하도록 하는 것이 하나의 해결책
    - **용어 빈도수(Term Frequency, TF)** : 용어가 문서에 등장하는 횟수
  - 프롬프트가 길어지고, 더 다양한 용어들을 포함하게 될 수 있음
    - 이 경우 중요한 용어를 효과적으로 식별할 방법이 필요해짐
    - 흔하게 등장하는 용어일수록 별로 유용하지 않은 경우가 많음
    - **역문서 빈도(Inverse Document Frequency, IDF)** : 용어가 나타나는 빈도가 적을수록 용어의 중요도를 높게 보는 지표
    - 용어의 IDF = 문서 총 수 / 용어가 포함된 문서 수
- **TF-IDF**: TF와 IDF를 결합한 알고리즘
  - 질의 Q에 대한 문서 D의 TF-IDF 점수 계산 방식
    - 질의 Q의 용어들은 t로 표시
    - 용어 t에 대해 문서 D에서 t의 빈도 = f(t,D)
    - 전체 문서 수 = N
    - t를 포함하는 문서 수 = C(t)
    - 용어 t의 IDF = log(N/C(t))
    - 질의 Q에 대한 문서 D의 TF-IDF = Score(Q,D) = Σ IDF(t) * f(t,D)
- 자주 사용되고 있는 용어 기반 솔루션 2가지
  - Elasticsearch
    - **inverted index** 데이터 구조 활용
    - 이는 용어를 문서로 매핑하는 사전이며, 이를 통해 용어에 관련된 문서들을 빠르게 찾을 수 있게 됨
    - 또한 해당 용어를 포함한 문서 갯수를 담은 문서를 추가 저장할 수 있음
  - BM25
    - 문서 길이를 고려해 용어 빈도 점수를 정규화
    - 긴 문서일수록 용어 포함 가능성이 높고, 용어 빈도 값도 함께 높아지는 것을 고려한 방식
    - 지금도 널리 사용되며, 임베딩 기반 검색과 비교할 때 중요한 기준점이 됨
- 토큰화 관련
  - 질의를 개별 용어로 나누는 과정
  - 가장 간단하게는 질의를 단어별로 나누고 각각을 별도의 용어로 취급하는 것
  - 하지만 개별 단어 방식은 여러 단어로 이루어진 용어들을 놓칠 수 있음
  - 이를 해결하기 위해 n-gram을 하나의 용어로 합치는 방식이 있음
  - 모든 문자를 소문자로 바꾸거나, 구두점을 없애거나, 불용어를 제거해서 용어들을 가려낼 수도 있을 것
  - 이러한 용어 관련 작업들을 처리해 주는 NLTK, spaCy, CoreNLP 등의 라이브러리 활용 가능

**임베딩 기반 검색**

- 텍스트의 겉모습은 그 의미를 제대로 반영하지 못할 때가 많으며, 이로 인해 사용자 의도와 무관한 문서가 반환될 수도 있음
- 임베딩 기반 검색기는 문서의 의미가 질의와 얼마나 가까운지를 기준으로 순위를 매김
  - **의미 기반 검색(semantic retrieval)** 이라고도 부름
- 원본 데이터 청크를 임베딩으로 변환하는 색인화 과정이 더 필요해짐
- **vector database**: 임베딩을 저장하는 데이터베이스
  - 벡터를 저장하는 것은 쉽지만, 벡터 검색은 진짜 어려운 부분
  - 질의가 임베딩으로 변환되면 벡터 DB는 질의 벡터와 유사한 벡터들을 찾아내야 함
  - 따라서 벡터들은 빠르고 효율적인 검색이 가능한 방식으로 색인화 및 저장되어야 함
- 질의 과정 2단계
  - **임베딩 모델**: 색인화에 사용된 것과 동일한 임베딩 모델을 사용해 질의를 임베딩으로 변환
  - **검색기**: 질의 임베딩과 가장 가까운 k개의 데이터 청크를 가져옴
- 실제로는 재순위 모듈이나, 캐시 같은 다른 구성 요소들이 더 포함될 수 있음
- 벡터 검색은 검색, 추천 시스템, 데이터 체계화, 클러스터링, 사기 탐지 등 임베딩을 활용하는 다양한 영역에서도 활용됨
- 벡터 검색은 보통 **최근접 이웃 검색(nearest neighbor search)** 문제로 이어짐
  - 가장 기본적인 방법은 k-NN(k-nearest neighbor) 알고리즘
    - 작동 방식
      - 질의 임베딩과 DB의 모든 벡터 간 유사도 점수를 코사인 유사도 같은 지표를 사용해 계산
      - 모든 벡터를 유사도 점수에 따라 순위를 매김
      - 가장 높은 점수를 얻은 k개의 벡터 반환
    - 한계점: 결과의 정확성을 보장하지만 계산이 많이 필요하고 느림
  - 따라서 큰 데이터셋에서는 보통 근사 최근접 이웃(Approximate Nearest Neighbor, ANN) 알고리즘을 사용
- 대표적인 벡터 검색 라이브러리
  - FAISS(Facebook AI Similarity Search)
  - ScaNN(Scalable Nearest Neighbors)
  - Annoy
  - Hnswlib(Hierarchical Navigable Small World)
- 대부분의 개발자는 벡터 검색을 직접 구현하지 않음
- 보통 벡터 DB는 벡터를 버킷, 트리, 그래프 등으로 구성
- 벡터 검색 알고리즘은 각각 다른 휴리스틱을 사용해 비슷한 벡터들이 저장 공간에서 서로 가까이 위치하도록 만듦
- 벡터는 양자화(정밀도 낮춤)를 하거나 희소하게 만들어 연산에 필요한 컴퓨팅 자원을 절약하기도 함
- 대표적인 벡터 검색 알고리즘들
  - **지역 민감 해싱(Locality Sensitive Hashing, LSH)**
    - 유사한 벡터를 같은 버킷에 해싱하여 유사도 검색 속도를 높이는 방식
    - 정확도를 일부 희생하는 대신 효율성을 얻는 방식
  - **계층적 탐색이 가능한 소규모 세계(Hierarchical Navigable Small World, HNSW)**
    - 노드가 벡터, 엣지가 유사 벡터를 연결하는 간선으로 표현되는 다중 레이어 그래프를 구성하는 방식
    - 검색 시 그래프 엣지를 따라 이동하며 최근접 이웃 탐색
  - **제품 양자화(Product Quantization, PQ)**
    - 각 벡터를 여러 하위 벡터로 분해하여 훨씬 단순한 저차원 벡터로 표현하는 방식
    - 저차원 벡터를 사용하면 거리 계산을 훨씬 빠르게 수행 가능
  - **역파일 색인(Inverted File Index, IVF)**
    - K-평균 클러스터링을 사용해 유사 벡터를 같은 클러스터로 묶는 방식
    - 평균적으로 각 클러스터에 100-10,000개 벡터로 구성하는 것이 일반적
    - 질의가 들어오면 질의 임베딩과 가장 가까운 클러스터 중심을 찾고, 해당 클러스터에 속한 벡터들이 후보 이웃이 됨
  - **근사 최근접 이웃 탐색(Annoy)**
    - 트리 기반 접근 방식
    - 여러 이진 트리를 구축해 벡터를 클러스터로 분할 - 무작위로 그은 선을 기준으로 벡터를 두 그룹으로 나누는 과정 반복
    - 검색 시 트리를 순회하여 후보 이웃 수집
- 그 외 알아둘 만한 알고리즘들
  - 공간 분할 트리 및 그래프(Space-Partitioning Trees and Graphs, SPTAG)
  - 근사 최근접 이웃을 위한 빠른 라이브러리(Fast Library for Approximate Nearest Neighbors, FLANN)

**검색 알고리즘 비교**

- **용어 기반 검색**
  - 색인화 및 질의 모두에서 임베딩 기반 검색보다 빠름
  - 별도의 설정 없이도 용어 검색이 잘 작동함
  - 임베딩 기반 검색보다 훨씬 저렴함
  - 성능 향상을 위해 조정할 수 있는 구성 요소가 적음
- **임베딩 기반 검색**
  - 질의 임베딩 생성 및 벡터 검색이 느릴 수 있음
  - 발전되어 오면서 용어 기반 검색보다 더 좋은 성능을 낼 수도 있음
  - 구성 요소들을 파인튜닝할 수 있음
  - 데이터를 임베딩으로 변환하면 특정 키워드들을 검색하기 어려워질 수 있음
  - 임베딩, 벡터 저장, 벡터 검색 솔루션은 비용이 많이 들 수 있음
- 검색기의 품질은 검색되는 데이터의 품질로 평가 가능
- RAG 평가 프레임워크에서 주로 사용되는 2가지 지표
  - **컨텍스트 정밀도(context percision)** : 검색된 모든 문서 중 질의 관련 문서 비율
  - **컨텍스트 재현율(context recall)** : 질의 관련 문서 중 실제로 검색된 문서 비율
- 위 지표들을 계산하기 위해, 테스트용 질의 목록과 각 질의에 해당하는 문서 집합으로 평가 세트를 구성
  - 각 테스트 질의에 대해, 테스트 문서가 관련 있는지 없는지 주석을 달아둠
  - 그 다음 평가 세트에서 검색기의 정밀도와 재현율 점수를 계산
- 특정 질의에 대한 컨텍스트 재현율을 계산하려면 DB의 모든 문서와 질의 사이 관련성 여부를 주석으로 달아야 함
  - 그래서 운영 환경에서 컨텍스트 정밀도만 지원하는 경우가 많음
- 검색된 문서의 순위가 중요할 때 쓸 수 있는 지표들
  - 정규화된 할인 누적 이득(Normalized Discounted Cumulative Gain, NDCG)
  - 평균 정밀도(Mean Average Precision, MAP)
  - 평균 역순위(Mean Reciprocal Rank, MRR)
- 의미 기반 검색에서는 임베딩 자체의 품질도 평가해야 함
  - 독립적으로 평가 가능
  - 비슷한 문서들이 벡터 공간에서 더 가깝게 위치한다면 좋은 임베딩으로 봄
  - 또한 특정 작업에 얼마나 효과적인지로도 평가됨
  - MTEB(Massive Text Embedding Benchmark)는 검색, 분류, 클러스터링 등 다양한 작업에서의 임베딩 성능 측정
- 검색기의 품질은 전체 RAG 시스템 컨텍스트에서도 평가해야 함
  - 검색기가 시스템이 고품질 응답을 생성하는 데 도움이 되면 좋은 것
- 의미 기반 검색 시스템의 성능 개선 판단 여부
  - 결국 질의 단계에서 비용과 지연 시간을 얼마나 중요하게 여기느냐에 달려 있음
  - 지연 시간: 주로 출력 생성 때 발생하고, 임베딩 생성 및 벡터 검색에 따른 지연은 미미할 수 있음
  - 비용: 데이터가 자주 변경되어 임베딩을 다시 생성해야 한다면 큰 문제가 될 수 있음
- 검색 시스템에서는 색인화와 질의 사이에 적절한 균형이 중요
  - 색인이 더 상세할수록 검색 과정은 더 정확하지만, 색인화 과정이 더 느려지고 메모리가 더 많이 필요해짐
  - HNSW 같은 상세한 색인은 구축하는 데 상당한 시간과 메모리가 필요
  - 반면에 LSH 같은 단순한 색인은 생성하는 데 더 빠르고 메모리가 더 적게 들지만 질의 속도가 느리고 정확도가 떨어짐
- [ANN-벤치마크](https://ann-benchmarks.com/) 웹사이트는 색인화와 질의의 균형을 고려해 4가지 주요 지표로 ANN 알고리즘을 비교
  - **재현율**
  - **초당 질의 수(QPS)**
  - **구축 시간**
  - **색인 크기**
- 요약: RAG 시스템의 품질은 구성 요소와 전체 시스템 모두에서 평가되어야 하며, 이를 위해 다음과 같은 작업들이 수행되어야 함
  - 검색 품질 평가
  - 최종 RAG 출력 결과 평가
  - 임베딩 품질 평가

**검색 알고리즘 결합하기**

- 검색 알고리즘들은 뚜렷한 장점을 가지고 있기 때문에, 보통 여러 접근 방법을 함께 결합해 사용함
- **하이브리드 검색(hybrid search)**
  - 용어 기반 검색과 임베딩 기반 검색을 결합한 것
  - 보통 비용이 적게 드는 용어 기반 시스템으로 후보군을 추려내고, 그 다음 k-최근접 이웃 같은 고비용 방식으로 최상의 결과를 찾아냄
  - 결과를 더 고도화하는 단계를 **reranking**이라고도 부름
- 서로 다른 알고리즘을 앙상블 기법으로 결합할 수도 있음
  - 여러 검색기를 동시에 사용해서 후보를 가져온 다음, 다양한 순위들을 하나로 결합해 최종 순위 생성
  - **역순위 퓨전(Reciprocal Rank Fusion, RRF)**
    - 서로 다른 순위를 결합하는 알고리즘
    - 검색기가 매긴 순위에 따라 각 문서에 점수를 부여하고, 최종 점수는 모든 검색기로부터 받은 점수의 합

### 검색 최적화

- 검색 가능성을 높이는 4가지 전략
  - 청킹 전략
  - 재순위화
  - 질의 재작성
  - 컨텍스트 검색

**청킹 전략**

- 청킹 전략은 검색 시스템의 성능에 상당한 영향을 미칠 수 있음
- 가장 단순한 전략: 고정 크기 청크
- 각 청크가 최대 청크 크기 안에 들어올 때까지 점점 작은 단위로 재귀적으로 분할할 수도 있음
- 청크 간의 겹침도 고려해야 함
  - 중요 컨텍스트가 분리되어 정보가 손실될 수도 있기 때문에
  - 경계 정보가 최소한 하나의 청크에 포함되도록 보장할 수 있을 것
- 청크 크기는 생성 모델의 최대 컨텍스트 길이를 넘지 말아야 함
- 어떤 전략을 사용하든 청크 크기가 중요!
  - 청크 크기가 작은 경우
    - 더 많은 청크로 다양한 정보를 담을 수 있음
    - 검색이 되지 않는 정보가 손실될 수 있음
    - 청크 크기가 줄어들면 색인화할 청크가 늘어나고, 생성 및 저장해야 할 임베딩 벡터도 늘어남

**재순위화**

- 모델의 컨텍스트에 맞추거나 입력 토큰 수를 줄이기 위해 검색된 문서 수를 줄여야 할 때 특히 유용
- 먼저 저비용으로 낮은 정확도의 후보들을 가져오고, 고비용으로 재순위화 하는 방식
- 시간을 기준으로 재순위화 될 수도 있음
- 컨텍스트 재순위화는 항목의 정확한 일치가 덜 중요함

**질의 재작성**

- 다음과 같은 용어로도 불림
  - 질의 재구성(query reformulation)
  - 질의 정규화(query normalization)
  - 질의 확장(query expansion)
- 컨텍스트를 모른다면 사용자의 실제 의도를 반영하여 질의를 재작성해야 함
- 새롭게 작성된 질의는 별도의 컨텍스트 없이도 의미가 명확해야 함
- 신원 확인이나 다른 지식을 통합해야 하는 경우 질의 재작성이 복잡해짐

**컨텍스트 검색**

- 핵심 아이디어: 각 청크에 컨텍스트를 추가해 검색 품질 향상
- 간단한 방법은 메타데이터에 태그나 키워드를 담는 것
- 각 청크에 응답 가능한 질의들을 추가할 수도 있음

*검색 솔루션 평가하기*

- 어떤 검색 방식을 지원하는가? 하이브리드 검색을 지원하는가?
- 어떤 임베딩 모델과 벡터 검색 알고리즘을 지원하는가?
- 데이터 저장량과 트래픽 측면에서 얼마나 확장 가능한가? 트래픽 패턴에 적합한가?
- 데이터 색인화에 얼마나 시간이 걸리는가? 한 번에 얼마나 많은 데이터를 대량으로 처리 가능한가?
- 다양한 검색 알고리즘에 대한 질의 지연 시간은 어느 정도인가?
- 관리형 솔루션인 경우, 가격 체계는 어떻게 되는가?

### 텍스트를 넘어선 RAG

- 앞에서는 외부 데이터 소스가 텍스트 문서인 RAG 시스템을 살펴봤는데, 다른 형식이라면 어떨까?

**멀티모달 RAG**

- 생성 모델이 멀티모달 데이터를 처리 가능하다면, 컨텍스트는 텍스트 뿐만 아니라 이미지, 비디오, 오디오 등도 활용 가능해짐
- 이미지에 제목, 태그, 캡션 등의 메타데이터가 있다면 이를 활용해 검색할 수 있을 것
- 질의가 텍스트이고 문서가 이미지라면, 텍스트와 이미지 모두 벡터로 변환 가능한 멀티모달 임베딩 모델 필요

**표 형식 데이터를 활용한 RAG**

- 대부분의 애플리케이션은 텍스트, 이미지 등의 비정형 데이터 뿐만 아니라 표 형식의 tabular 데이터도 처리
- 표 형식 데이터로 컨텍스트를 보강하는 과정은 일반적인 RAG 워크플로와 상당히 다름
- 데이터 처리 과정
  - **Text-to-SQL**: 사용자 질의 및 제공된 테이블 스키마 기반으로 SQL 쿼리 생성
  - **SQL 실행**: DB에서 쿼리 실행 결과 반환
  - **응답 생성**: 결과에 대한 자연어 응답 생성
- 만약 Text-to-SQL 단계에서 테이블이 너무 많아 모델 컨텍스트에 담을 수 없다면 사용할 테이블 예측 단계가 더 필요해질 수 있음

## 6.2 에이전트

- 많은 사람이 AI의 궁극적 목표로 여기는 것은 **지능형 에이전트**
  - 고전 『Artificial Intelligenct: A Modern Approach』(1995)에서도 AI 연구 분야를 **합리적 에이전트의 연구와 설계**라고 정의
- 파운데이션 모델의 획기적인 발전으로 에이전트 기반 애플리케이션의 문이 열렸음
- 에이전트는 새로운 개념이지만, 모델의 자기 비평, 생각의 사슬(CoT), 구조화된 출력 등 이미 다룬 내용들의 연장선이기도 함

### 에이전트 개요

- Agent: 자신의 환경을 인식하고 그 환경에서 행동할 수 있는 모든 것
  - 다시 말해, 작동하는 환경과 수행할 수 있는 행동들로 정의되는 것
- AI 에이전트가 수행할 수 있는 행동 집합들은 접근할 수 있는 도구에 의해 확장됨
- 에이전트의 환경과 도구들 사이에는 강한 의존성이 있음
- 환경은 에이전트가 잠재적으로 사용할 수 있는 도구를 결정함
- AI 에이전트는 주로 사용자가 입력으로 제공하는 작업을 수행하기 위해 만들어짐
- 에이전트에게 더 강력한 모델이 필요한 2가지 이유
  - **누적되는 오류**: 에이전트는 보통 여러 단계의 작업을 수행하며, 각 단계마다 정확성이 떨어질 수 있음
  - **더 큰 위험성**: 도구 사용을 통해 더 영향력 있는 작업을 수행할 수 있지만, 실패의 여파 또한 커질 수 있음
- 많은 단계가 필요한 작업은 시간 및 비용이 많이 들지만, 에이전트가 자율적으로 행동하게 할 수 있다면 그만한 가치가 있음
- 에이전트의 성공은 사용 가능한 도구들과 AI가 얼마나 계획을 잘 세우는지에 달려 있음

### 도구

- 외부 도구 없는 에이전트도 가능하지만, 능력이 제한적일 수밖에 없음
- 도구(tool)는 에이전트에게 환경을 인식하는 능력과 그 환경에 변화를 주는 능력을 부여해줌
  - 읽기 행동: 에이전트의 환경 인식
  - 쓰기 행동: 환경에 변화 주기
- 에이전트의 도구 목록은 에이전트가 할 수 있는 일을 결정하기 때문에 신중히 고려해야 함
  - 최적의 도구 조합 탐색을 위한 실험이 필요

**지식 증강**

- 텍스트 검색기, 이미지 검색기, SQL 실행기 등의 지식 증강 도구들
- 특히 웹 브라우징은 모델이 구식이 되는 것을 막아줌
  - 최신 정보를 통해 환각을 줄여줄 수 있음
  - 하지만 유해한 콘텐츠에 놀출될 위험도 증가

**능력 확장**

- AI 모델의 고유한 한계를 보완하는 도구들
- 모델에게 새로운 능력을 학습시키는 것보다 도구에 대한 접근 권한을 주는 것이 효율적
- 캘린더, 시간대 변환기, 단위 변환기, 언어 번역기 등의 도구들
- 텍스트 및 이미지 관련 도구들을 활용해서 멀티모달 전용으로 만들 수 있음
- **도구 사용은 심지어 프롬프팅, 파인튜닝보다도 모델의 성능을 크게 향상시킬 수 있음**

**쓰기 행동**

- 모델은 도구를 통해 읽기 행동 뿐만 아니라 쓰기 행동도 수행 가능
- 쓰기 권한을 통해 많은 작업들을 자동화할 수 있음
- 하지만 시스템의 능력만큼 보안 조치에 대한 신뢰도 중요함

*함수 호출(function calling)*

- 많은 모델 제공 업체들이 함수 호출 기능을 통해 모델이 다양한 도구를 사용할 수 있도록 지원하고 있음
- 앞으로 대부분의 모델에서 사실상 표준으로 자리잡을 것

### 계획 수립

- 에이전트의 작업은 목표와 제약 조건으로 정의됨
- 복잡한 작업에는 계획 수립이 필요
- 계획: 작업을 달성하는 데 필요한 단계를 정리한 로드맵
- 효과적인 계획 수립은 보통 모델이 작업을 이해하고, 이를 달성하기 위한 다양한 선택지를 고려한 후, 가장 유망한 경로를 선택하는 과정을 포함

**계획 수립 개요**

- 계획 수립 및 실행을 동일한 프롬프트에서 결합할 수는 있음
  - 프롬프트에서 단계별로 생각하도록 요청한 다음, 해당 프롬프트에서 실행시키게 하는 식
  - 하지만 모델이 달성하지도 못할 수많은 단계의 계획을 세우는 등 비효율적인 작동을 하게 될 수 있음
  - 감독 없는 에이전트가 아무 성과 없이 시간과 API 호출 비용만 낭비하게 될 수 있음
- 따라서 **계획 수립**과 **실행**을 분리해야 함
  - 먼저 에이전트에게 계획 수립을 요청하고, 이것이 검증된 후에만 실행하는 방식
  - 계획은 휴리스틱을 사용해 검증
    - 휴리스틱의 예시
      - 유효하지 않은 행동이 포함된 계획 제거
      - 사용할 수 없는 도구가 포함된 계획 제거
      - X단계보다 많은 모든 계획 제거
- 계획은 AI 평가자를 사용해서 검증할 수도 있음
- 계획은 전체 작업의 모든 단계를 포함할 필요는 없으며, 작은 하위 작업에 대한 계획일 수도 있음
- 멀티 에이전트 시스템의 3가지 구성 요소
  - 계획 생성 요소
  - 계획 검증 요소
  - 계획 실행 요소
- 과정을 더 빠르게 진행하기 위해, 순차적인 계획 생성 대신 병렬적으로 동시에 생성하고 최적의 계획을 선택하도록 요청할 수도 있음
  - 다만 지연 시간과 비용 사이의 균형점을 찾아야 할 것
- 의도 분류
  - 계획 수립에는 작업 뒤의 사용자 의도를 이해하는 것도 필요
  - 사용자가 이 질의로 무엇을 하려 하는가?
  - 의도 분류기는 종종 에이전트의 계획 수립을 돕는 데 사용됨
  - 의도 분류는 별도의 프롬프트나 분류 모델을 사용해 수행 가능
  - 의도 파악은 에이전트가 적절한 도구를 선택하는 데에도 도움이 될 수 있음
  - 일부 질의는 에이전트 처리 범위 밖일 수 있으며, 의도 분류기는 이런 요청을 **IRRELEVANT**로 분류하여 에이전트 계획 수립을 우회할 수 있음
- 계획 생성-계획 검증-계획 실행 3단계는 사람이 수행할 수도 있음
  - AI 기반 자동화도 되겠지만, 과정 개선 및 위험 부담 절감을 위한 노력
  - 복잡한 작업의 경우, 전문가가 계획의 큰 틀을 제공하고 에이전트가 구체화할 수도 있을 것
  - 위험한 작업의 경우에도 사용자의 명시적 승인이 필요할 것
- 에이전트의 일반적인 작업 해결 과정
  - **계획 생성**: 작업 수행 전 계획, 작업 분해라고도 함
  - **성찰 및 오류 수정**: 생성된 계획 평가, 좋지 않은 경우 새로운 계획 생성
  - **실행**: 생성된 계획에 따라 행동 수행, 주로 특정 함수를 호출하는 방식
  - **성찰 및 오류 수정**: 행동 결과를 평가하고 목표 달성 여부 확인, 오류가 있다면 수정, 목표가 완료되지 않았다면 새로운 계획 생성

**계획자 역할을 하는 파운데이션 모델**

- 파운데이션 모델이 얼마나 효과적으로 계획을 세울 수 있는지는 아직 명확히 검증되지 않았음
- 적어도 자기회귀 모델은 계획을 세울 수 없다는 의견이 지배적
  - "LLM의 계획은 그럴듯해 보이지만, 실행 단계에서 여러 문제 및 오류를 일으킬 수 있다."
- **계획 수립은 본질적으로 탐색 문제**
  - 목표로 가는 다양한 경로를 탐색하고, 각 경로의 결과를 예측하며, 가장 유망한 결과를 가진 경로를 선택하는 과정
  - 탐색 과정에는 종종 **백트래킹**도 필요
- 자기회귀 모델은 앞으로 나아가기 때문에 뒤로 돌아가는 백트래킹을 할 수 없다는 주장이 많음
  - 하지만 모델도 경로를 수정할 수 있기 때문에 꼭 그런 것은 아님
  - 모델은 아예 처음부터 경로를 다시 탐색할 수도 있음
- LLM이 계획 수립에 취약한 이유 중 하나는 도구를 제대로 갖추지 못했기 때문일 수 있음
  - 효과적인 계획을 세우는 작업은 가능한 행동뿐 아니라 각 행동의 잠재적 결과도 알아야 함
  - 그러므로 단순히 CoT 프롬프팅 기법처럼 단순히 행동 순서 생성만 요청하는 프롬프팅은 충분치 않을 수 있음
- AI가 계획 수립에 완벽하지 않다 하더라도 계획 수립 시스템의 일부로 활용될 수는 있을 것
  - 검색 도구, 상태 추적 시스템을 결합해 계획 수립 능력을 강화할 수 있을 것

**계획 생성**

- 모델을 계획 생성기로 바꾸는 가장 간단한 방법은 프롬프트 엔지니어링
- 프롬프트에서 에이전트가 사용 가능한 도구들(함수들)을 나열하고, 이 도구들을 사용해서 계획을 생성하는 예제를 만들어둘 수 있을 것
- 종종 함수의 정확한 파라미터 값을 결정할 정보가 부족한 경우가 있음
  - 이런 경우 모델이 사용자 요구를 자주 추측하게 되고, 추측이 틀리게 될 수도 있음을 의미
- 이렇듯 행동 순서와 관련된 파라미터 모두 AI 모델이 생성하므로 환각이 발생할 수 있음
  - 환각은 모델이 존재하지 않는 함수를 호출하거나, 함수를 잘못된 파라미터로 호출하게 될 수 있음
- 에이전트의 계획 수립 능력 향상을 위한 방법들
  - **더 많은 예시가 포함된 더 나은 시스템 프롬프트 작성**
  - **도구와 파라미터에 대해 더 자세히 설명해서 모델이 더 잘 이해하도록 돕기**
  - **함수 자체를 더 단순하게 만들기**
  - **더 강력한 모델 사용**
  - **모델 파인튜닝**

**함수 호출**

- 많은 모델 제공업체가 자사 모델 도구 사용 기능을 함수 호출 방식으로 제공
- 그래서 도구 사용을 종종 **함수 호출(function calling)** 이라고 부름
- 일반적인 함수 호출 과정
  - **도구 목록 생성**
  - **에이전트가 사용할 수 있는 도구 지정**
- 시스템이 항상 각 함수 호출에 어떤 파라미터 값을 사용했는지 알려주도록 설정하는 것이 좋음

**계획의 세부성**

- 계획은 로드맵이며, 다양한 수준의 세부성을 가질 수 있음
- 큰 그림으로 만들지, 세부적으로 만들지, 이 사이에서 절충점을 찾아야 함
- 이를 위해 계층적 계획을 세워야 함
  - 먼저 분기별 계획 같은 큰 그림의 계획 세우기
  - 그 다음 각 분기마다 더 자세한 월별 계획들을 수립하는 방식
- 정확한 함수 이름들은 매우 세부적인 접근법이라 할 수 있음
  - 이는 에이전트 도구 목록이 변해 가면서 문제가 될 수도 있음
  - 도구 목록이 변하면 이에 따라 프롬프트 변경도 필요해질 것
  - 계획 수립자 재활용도 어려워질 것
- 위 문제를 해결하기 위해 특정 함수 이름 대신 일반적인 자연어를 사용하게 할 수 있을 것
  - 더 자연스러운 언어는 계획 생성기가 도구 API 변화에 더 유연하게 대응할 수 있게 해줌
  - 하지만 자연어 행동을 실행 가능한 명령으로 번역해주는 번역기가 필요해지기는 함
  - 그래도 번역이 계획 수립보다 훨씬 간단한 작업일 것이며, 더 가벼운 모델로도 수행 가능할 것

**복잡한 계획**

- **제어 흐름(control flow)** : 행동이 실행되는 순서
  - **순차 실행**
  - **병렬 실행**
  - **If 조건문**
  - **For 반복문**
- AI 에이전트는 일반적인 엔지니어링과는 다르게 AI 모델이 제어 흐름을 결정함
- 에이전트 프레임워크를 평가할 때, 어떤 제어 흐름을 지원하는지 확인하는 것이 중요

**성찰 및 오류 수정**

- 계획의 성공 가능성을 높이기 위해 지속적으로 평가 및 조정
- 성찰은 필수는 아니지만 에이전트의 성공적인 작업 수행에 도움이 됨
- 성찰이 유용하게 활용되는 단계들
  - **사용자 질의를 받은 후 요청 실현 가능 여부를 검토할 때**
  - **초기 계획 수립 이후 계획의 타당성을 확인할 때**
  - **각 실행 단계 이후 계획이 올바른 방향으로 진행되고 있는지 점검할 때**
  - **전체 계획 실행 후 정상 작동 여부를 확인할 때**
- 성찰과 오류 수정은 서로 밀접하게 연결되어 있지만 별개의 메커니즘
  - 성찰을 통해 문제를 파악하고, 이를 해결하기 위해 오류 수정이 수행됨
- 성찰은 자기 비평 프롬프트에 포함시키거나, 별도의 전문 평가 모델을 구성할 수 있음
- ReAct에서 제안된 추론과 행동의 교차 방식은 에이전트의 일반적인 패턴이 되었음
  - **추론(Reasoning)** : 계획 수립과 성찰을 모두 포함하는 개념
  - 각 단계에서 에이전트는 자신의 생각을 설명(계획) → 행동 → 관찰 결과를 분석(성찰) ⇒ 에이전트가 작업이 완료되었다고 판단할 때까지 반복
  - 일반적으로 에이전트는 예시를 참고하여 출력을 생성하도록 프롬프팅 됨
- 성찰은 멀티 에이전트 설정을 통해 분리될 수 있음
  - 강화 학습에서의 **행위자-비평자(Actor-Critic, AC)** 에이전트 방법을 떠올리게 함
- 에이전트가 작업을 완수하지 못했을 경우 성찰을 통해 새로운 계획을 세움
- Reflexion 프레임워크에서는 성찰을 두 모듈로 나눔
  - **평가자(Evaluator)**: 결과를 평가
  - **성찰 모듈(Reflexion Module)**: 무엇이 잘못되었는지 스스로 과정을 되짚어봄
- 계획 생성에 비해 성찰은 구현이 비교적 쉽고 높은 성능 향상을 가져올 수 있음
  - 다만 지연 시간과 비용이 증가함
  - ReAct 및 Reflexion 에서는 프롬프트에 많은 예시를 포함시키는 방식으로 입력 토큰 계산 비용을 높이고 컨텍스트 공간을 줄이는 방식을 사용함

**도구 선택**

- 도구는 작업 성공의 중요한 역할을 수행
- 도구가 많을수록 에이전트 능력은 향상되지만 그만큼 효율적인 사용이 어려워짐
  - 또한 도구가 늘어날수록 모델의 컨텍스트 한계를 초과하게 될 수도 있음
- 도구 선택을 위한 실험 및 분석 방법들
  - 여러 도구 조합으로 에이전트 성능 비교
  - 특정 도구 제거 시 에이전트 성능이 얼마나 떨어지는지 확인
  - 자주 실수하는 도구 탐색
  - 도구 호출 분포를 시각화해서 도구 사용 빈도 파악
- 도구 선택 시 중요하게 여겨야 할 것들
  - 작업마다 필요한 도구가 다름
  - 모델마다 선호하는 도구가 다름
- 에이전트 프레임워크를 평가할 때도 어떤 계획 수립 기능과 도구를 지원하는지 확인 필요
- 자주 활용되는 도구 사용 패턴이 있다면 조합해서 더 복잡한 도구를 만들어낼 수도 있을 것
  - Voyager는 에이전트가 습득한 새로운 도구를 추적 및 재사용할 수 있게 해주는 스킬 매니저를 제안했음

### 에이전트 실패 유형과 평가

- 평가 = 실패를 발견하는 과정
- 에이전트가 수행하는 작업이 복잡할수록 실패 가능성도 높아짐
- 에이전트 평가를 위해 실패 유형을 파악하고, 각 유형이 얼마나 자주 발생하는지 측정해야 함

**계획 수립 실패**

- **도구 사용 실패**
  - 가장 흔한 계획 수립 실패 유형
  - 유효하지 않은 도구
  - 유효한 도구, 유효하지 않은 파라미터
  - 유효한 도구, 잘못된 파라미터 값
- **목표 달성 실패**
  - 계획이 작업을 완수하지 못하거나, 제약 조건을 따르지 않은 경우
  - 특히 작업 시간은 종종 간과되는 제약 조건이지만, 시간이 오래 걸릴수록 에이전트의 유용성은 급격히 떨어짐
- **성찰 오류**
  - 에이전트가 실제로는 작업을 완료하지 못했음에도 완료했다고 확신하는 것
- 계획 수립 실패에 대해 평가하는 방법
  - (task, tool inventory) 쌍으로 구성된 계획 수립 데이터셋을 만드는 방식
  - 각 작업에 대해 K개의 계획 생성 후 다음 지표들 계산
    - 생성된 모든 계획 중 유효 계획의 비율
    - 주어진 작업에 대해, 유효 계획을 얻기 위해 평균적으로 몇 개의 계획을 생성해야 하는지
    - 모든 도구 호출 중 유효한 호출의 비율
    - 유효하지 않은 도구 호출 빈도
    - 유효한 도구가 유효하지 않은 파라미터로 호출되는 빈도
    - 유효한 도구가 잘못된 파라미터 값으로 호출되는 빈도

**도구 실패**

- 올바른 도구를 사용했지만 결과가 잘못된 경우
- 에이전트가 작업에 필요한 적절한 도구들을 갖고 있지 않을 때에도 발생
- 도구 실패는 도구마다 다르게 나타나므로 각 도구는 개별적으로 테스트되어야 함
- 도구 호출 및 결과를 출력해서 검사하고 평가해야 함

**효율성**

- 에이전트가 올바른 도구로 유효한 계획을 세워 작업을 완수하더라도 그 과정이 비효율적일 수 있음
- 에이전트 효율성 평가를 위한 요소들
  - 작업 완료까지 평균적으로 몇 단계가 필요한가?
  - 작업 완료까지 평균적으로 얼마의 비용이 드는가?
  - 각 행동은 보통 얼마나 시간이 걸리는가? 특히 시간 및 비용이 많이 드는 행동이 있는가?

## 6.3 메모리

- 메모리: 모델이 정보를 저장하고 활용할 수 있게 하는 방식
- 메모리 시스템은 RAG은 물론, 에이전트에도 매우 유용
  - RAG: 검색된 정보로 컨텍스트를 보강하고 처리하기 위해 메모리 활용, 대화가 계속될수록 더 많아지는 정보들을 담아둘 수 있음
  - 에이전트: 지시, 예시, 컨텍스트, 도구 목록, 계획, 도구 출력, 성찰 등을 저장하기 위해 메모리 활용
- RAG와 에이전트는 메모리가 특히 더 필요하지만, 모든 AI 애플리케이션에서 메모리는 유용하게 쓰임
- AI 모델의 메모리는 크게 3가지로 나뉨
  - **내부 지식**
    - 모델이 학습한 데이터들은 그 자체로 하나의 기억 시스템이라 할 수 있음
    - 이는 모델이 업데이트되지 않는 한 변하지 않음
    - 모델은 모든 질의에서 이 지식에 접근 가능
  - **단기 메모리**
    - 모델의 컨텍스트 또한 하나의 기억 시스템이라 할 수 있음
    - 모델은 이전 대화 내용을 컨텍스트에 추가해두고, 다음 응답을 생성할 때 이를 참조할 수 있음
    - 다만 모델의 컨텍스트는 작업(질의) 종료 후 사라지므로 단기 메모리라고 볼 수 있음
    - 접근 속도는 빠르지만 용량이 한정되어 있으므로, 주로 현재 작업에 가장 중요한 정보를 저장
  - **장기 메모리**
    - RAG과 같은 외부 데이터 소스 또한 하나의 메모리
    - 작업이 끝나도 정보는 유지됨
    - 모델 자체를 업데이트하지 않고도 수정 및 삭제 가능
    - 단기 메모리의 넘치는 정보를 저장하는 데 활용 가능
- 어떤 정보를 어떤 메모리에 저장할지는 그 정보가 얼마나 자주 사용되는지에 따라 결정됨
- AI 모델에 메모리 시스템을 추가했을 때의 이점들
  - **세션 내 정보 과부하 관리**
    - 작업 실행 중 에이전트는 많은 정보를 얻게 되므로, 에이전트의 최대 컨텍스트를 초과하지 않게 하기 위해 메모리가 필요
  - **세션 간 정보 관리**
    - 대화 기록 유지를 통해 에이전트가 사용자 맞춤형 서비스를 제공해줄 수 있을 것
  - **모델의 일관성 향상**
    - 이전 응답을 기억해둠으로써 더 일관된 응답을 하게 됨
  - **데이터 구조 무결성 유지**
    - 텍스트는 비정형이기 때문에, 메모리에 구조화된 데이터를 저장해두면 데이터의 구조적 무결성을 유지할 수 있음
- AI 모델을 위한 메모리 시스템의 2가지 핵심 기능
  - **메모리 관리**: 메모리를 단기 및 장기 메모리에 저장할지 관리
    - 일반적으로 추가, 삭제 2가지 작업으로 이루어짐
    - 특히 단기 메모리의 경우 모델의 최대 컨텍스트 길이가 있기 때문에 추가 및 삭제 전략이 필요
  - **메모리 검색**: 장기 메모리에서 작업 관련 정보 검색
- 메모리를 효율적으로 사용하기 위한 전략들
  - **FIFO**
    - 메모리를 효율적으로 사용하기 위한 가장 간단한 전략
    - 단기 메모리에 가장 먼저 추가된 정보가 가장 먼저 장기 메모리로 이동하는 방식
    - 대화형 AI의 경우, 최근 N개의 메시지 혹은 마지막 N개의 토큰만 유지하는 방식으로 사용
      - 하지만 초기 대화 부분에서 중요한 부분을 놓치게 만들 우려도 있음
      - 첫 메시지에 대화의 목적이 담겨 있는 경우에 치명적
  - 중복 제거
    - 사람의 언어는 의미의 명확한 전달을 위해 중복을 사용함
    - 중복을 자동으로 감지할 수 있다면 메모리 사용량이 크게 줄어들 것
    - 가장 효과적인 방법은 대화의 요약
