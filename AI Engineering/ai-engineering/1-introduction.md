# 1. 파운데이션 모델을 활용한 AI 애플리케이션 입문

## 1.1 AI 엔지니어링의 부상

### 언어 모델 (Language Model)

- 하나 이상의 언어에 대한 통계 정보를 인코딩
- 주어진 컨텍스트에서 어떤 단어가 나타날 것 같은지를 알려줌
- 언어 모델의 기본 단위는 토큰
  - [OpenAI Tokenizer](https://platform.openai.com/tokenizer)  
  - 토큰은 모델에 따라 문자, 단어, 또는 단어의 일부가 될 수 있음
- 토큰화: 원문을 모델이 정한 길이로 나누는 과정
- 어휘(vocabulary): 모델이 다룰 수 있는 모든 토큰의 집합
  - 소수의 토큰만 사용해 많은 고유 단어 생성 가능
- 언어 모델의 2가지 유형
  - **마스크 언어 모델(masked language model)**
    - 누락된 토큰 전후 컨텍스트를 사용해 시퀀스의 어느 위치에서든 누락된 토큰을 예측하도록 학습
    - 기본적으로 빈칸을 채울 수 있도록 학습됨
    - 감정 분석, 텍스트 분류 등 새로운 텍스트를 만들지 않는 작업에 주로 사용됨
    - 코드 디버깅처럼 앞뒤 코드를 모두 이해해서 오류를 찾아야 하는 전체 컨텍스트 이해가 필요한 작업에도 유용
  - **자기회귀 언어 모델(autoregressive language model)**
    - 이전 토큰들만 보고 시퀀스의 다음 토큰을 예측하도록 학습됨
    - 토큰을 하나씩 순차적으로 생성할 수 있음
    - 텍스트 생성 분야의 대세로 자리 잡아 마스크 언어 모델보다 훨씬 큰 인기
  - 둘의 가장 큰 차이점
    - 자기회귀 언어 모델: 이전 토큰만 사용해서 예측
    - 마스크 언어 모델: 주변 토큰을 사용해서 예측

### 자기 지도 학습 (Self-supervised Learning)

- 다른 모델은 지도 학습이 필요하지만 언어 모델은 자기 지도 학습이 가능
- 지도 학습: 레이블이 있는 데이터를 사용해 ML 알고리즘을 학습하는 과정
  - 비용 및 시간이 많이 듦
- 자기 지도 학습은 데이터 레이블링 병목 현상을 극복해 모델이 학습할 대규모 데이터셋을 만들 수 있게 해줌
- 자기 지도 학습에는 명시적인 레이블이 필요하지 않고, 모델이 입력 데이터로부터 레이블 추론 가능

**⇒ 자기 지도 학습을 통해 언어 모델을 LLM으로 확장할 수 있다.**

### 파운데이션 모델 (Foundation Model)

- 언어 모델은 놀라운 작업을 수행할 수 있지만, 글자에만 국한되어 있음
- 사람은 언어뿐만 아니라 다양한 감각들을 통해서 세상을 인식하므로, AI 또한 글자 이상의 데이터를 처리하는 능력이 꼭 필요함
- 이에 따라 언어 모델은 더 많은 모달리티를 포함하도록 확장되고 있음
- 파운데이션이라는 단어의 의미: 모델들이 AI 애플리케이션에서 갖는 중요성 및 다양성에 맞게 발전시킬 수 있다는 것
- 생성형 멀티모달 모델을 **대규모 멀티모달 모델(Large Multimodal Model, LMM)** 이라고도 부름
- 파운데이션 모델은 한가지 작업에만 국한되지 않고 다양한 작업을 수행할 수 있음
  - 만약 특정 작업에 대한 성능을 올리고 싶다면 파인튜닝(fine-tuning)을 통해 모델을 조정할 수 있음
- 모델이 원하는 결과물을 내놓도록 유도하는 기법들
  - **프롬프트 엔지니어링(prompt engineering)** : 상세한 지시를 모델에 입력하는 방법
  - **검색 증강 생성(Retrieval-Augmented Generation, RAG)** : 외부 지식을 활용해 모델의 답변을 보강하는 방법
  - **파인튜닝(fine-tuning)** : 모델에게 고품질 데이터셋을 제공해 특정 작업에 대한 성능을 올리는 방법

### AI 엔지니어링 (AI Engineering)

- AI 엔지니어링: 파운데이션 모델을 기반으로 애플리케이션을 만드는 과정
- 전통적인 ML 엔지니어링이 모델 자체를 개발하는 것이라면, AI 엔지니어링은 이미 존재하는 모델을 활용함

## 1.3 AI 애플리케이션 기획

### AI가 맡는 역할

- 핵심적 또는 보완적
  - 앱이 AI 없이도 작동할 수 있다면 AI는 '보완적'
- 반응형 또는 선제형
  - 반응형은 사용자의 요청이나 특정 행동에 '응답'하는 방식
  - 선제형은 사용자에게 유용하다고 판단되는 '적절한 시점'에 먼저 정보를 제시
  - 반응형은 지연 시간이 더 짧아야 함
  - 선제형은 더 높은 품질 기준을 요구함
- 동적 또는 정적
  - 동적 방식은 사용자 피드백을 통해 지속적으로 업데이트됨
  - 정적 방식은 주기적으로 업데이트 됨
  - 동적 방식은 파인 튜닝을, 정적 방식은 공유 모델의 업데이트를

### AI 제품 방어 가능성

- 우리가 쉽게 만들 수 있는 것은 경쟁사도 쉽게 만든다
- 제품을 방어할 수 있는 견고한 해자가 있는가?
- 파운데이션 모델 위에 애플리케이션을 만든다는 건 모델들 위에 하나의 계층을 얹는 셈
  - 오픈AI 래퍼, 클로드 래퍼라는 농담이 유행했음
  - 파운데이션 모델이 좋아지면 우리가 얹은 계층도 모델에 흡수되어 쓸모없게 될 수 있음
- AI 분야의 경쟁 우위 3가지: **기술력**, **데이터**, **유통력**
  - 파운데이션 모델을 사용하면 대부분 회사의 핵심 기술은 비슷해질 것
  - 유통력의 우위는 대기업이 가질 것
  - 데이터는 스타트업과 대기업 모두 확보 가능한 것

## 1.4 AI 엔지니어링 스택

- AI 엔지니어링은 ML 엔지니어링에서 발전한 것
- 따라서 기업이 파운데이션 모델을 실험하기 시작할 때, 보통 기존 ML 팀이 담당함
- AI 엔지니어와 ML 엔지니어의 역할은 상당 부분 중복됨

### AI의 3가지 계층

**애플리케이션 개발**

- 모델을 활용해 누구나 애플리케이션 개발 가능
- 가장 활발한 활동이 있었으며, 여전히 빠르게 발전 중
- 모델에 적절한 프롬프트 및 컨텍스트를 제공하는 것도 포함
- 철저한 평가 과정이 필요하며, 좋은 인터페이스도 필요
- 구성요소
  - AI 인터페이스
  - 프롬프트 엔지니어링
  - 컨텍스트 구성
  - 평가

**모델 개발**

- 모델링, 학습, 파인튜닝, 추론 최적화를 위한 프레임워크를 포함해 모델 개발을 위한 도구 제공
- 데이터가 핵심이므로, 데이터셋 엔지니어링도 포함됨
- 역시 철저한 평가 과정이 필요함
- 구성요소
  - 추론 최적화
  - 데이터셋 엔지니어링
  - 모델링 및 학습
  - 평가

**인프라**

- 모델 서빙, 데이터 및 컴퓨팅 관리, 모니터링 도구를 포함하는 인프라
- 구성요소
  - 컴퓨팅 관리
  - 데이터 관리
  - 서빙
  - 모니터링

### AI 엔지니어링 vs ML 엔지니어링

- AI 엔지니어링은 모델링 및 학습 보다는 **모델 조정**에 더 초점을 맞춤
- AI 엔지니어링은 더 많은 컴퓨팅 자원을 소비하며, 더 높은 지연 시간을 발생시키는 모델을 다룸
  - 효율적인 학습 및 추론 최적화에 대한 압박이 크다는 의미
  - 이에 따라 GPU 및 대규모 클러스터를 다룰 줄 아는 엔지니어에 대한 수요가 늘어났음
- AI 엔지니어링은 개방형 출력을 생성할 수 있는 모델을 다룸
  - 개방형 출력은 모델에게 유연성을 제공하지만 그만큼 평가하기 어려움

### 모델 조정 기법

- 모델 조정 기법은 모델 가중치 업데이트 여부에 따라 2가지로 나뉨
- **프롬프트 기반 기법**
  - 모델 가중치를 업데이트하지 않고, 프롬프트 및 컨텍스트를 통해 모델의 출력을 조정
  - 프롬프트 엔지니어링은 비교적 쉽게 시작할 수 있으며 데이터도 거의 필요 없음
  - 사용법이 쉽고 모델을 자유롭게 실험해 볼 수 있음
  - 하지만 복잡한 작업이나 성능에 대한 엄격한 기준이 있는 애플리케이션에는 부적합
- **파인튜닝**
  - 모델 가중치를 업데이트하여 모델의 출력을 조정
  - 더 복잡하고 더 많은 데이터 필요 → 하지만 품질, 지연 시간, 비용을 크게 개선할 수 있음
  - 복잡한 작업에 많이 활용됨

### 모델 개발

- 전통적인 ML 엔지니어링과 가장 밀접하게 연관된 계층
- **모델링과 학습**, **데이터셋 엔지니어링**, **추론 최적화** 라는 3가지 구성 요소가 있음
- 평가도 필요하지만 대부분 애플리케이션 개발 계층에서 먼저 공부하게 되므로 다음에

**모델링과 학습**

- 모델 아키텍처를 고안하고, 학습하고, 파인튜닝하는 과정을 뜻함
- 예시: 구글 - 텐서플로, 허깅페이스 - 트랜스포머, 메타 - 파이토치
- ML 지식 필요
  - 클러스터링, 로지스틱 회귀, 의사결정 트리, 협업 필터링 등 다양한 ML 알고리즘
  - 피드포워드, 순환 합성곱, 트랜스포머 등 신경망 아키텍처
  - 경사 하강법, 손실 함수, 정규화 등 학습 관련 지식
- 파운데이션 모델이 등장하면서 ML 지식은 필수가 아님, 하지만 사용할 수 있는 도구의 폭을 넓혀주고 모델의 예상치 못한 작동에 대한 문제 해결에 도움이 됨
- **학습 단계**
  - 사전 학습(pre-train)
    - 모델을 처음부터 학습하는 것을 의미
    - 모델 가중치가 무작위로 초기화됨
    - LLM의 경우 보통 텍스트 완성을 위해 모델을 학습하는 것도 포함
    - 모든 학습 단계 중에서도 압도적으로 많은 지원 필요
    - 이에 따라 소수만이 경험할 수 있는 전문적인 기술 영역이 되었음
  - 파인튜닝(fine-tuning)
    - 이미 학습된 모델을 추가로 학습하는 과정
    - 모델 가중치는 사전 학습의 값 유지
    - 사전 학습보다는 적은 자원 필요
  - 사후 학습(post-train)
    - 모델 제공업체가 학습을 수행하면 '사후 학습', 개발자가 수행하면 '파인튜닝'이라고 구분함
- *프롬프트 엔지니어링은 학습이 아니다*

**데이터셋 엔지니어링**

- AI 모델의 학습과 조정에 필요한 데이터를 선별하고, 생성하며, 주석을 다는 것을 의미
- 모델 자체가 대중화된 기술이 되었기 때문에 데이터가 주요 차별점이 됨
- AI 엔지니어링에서 데이터 가공은 중복 제거, 토큰화, 컨텍스트 검색, 민감 정보 및 유해 데이터 제거를 포함한 품질 관리에 더 중점을 둠

**추론 최적화(Inference Optimization)**

- 모델을 더 빠르고 저렴하게 만드는 것을 의미
- ML 엔지니어링에서도 항상 중요했는데, 파운데이션 모델이 더 높은 추론 비용과 지연 시간을 발생시키게 되면서 더욱 중요해짐
- 파운데이션 모델은 종종 자기회귀적(autoregressive)이라는 점이 문제인데, 그럼에도 불구하고 지연 시간을 일반적인 인터넷 애플리케이션에 기대되는 100ms 수준으로 낮추는 것이 과제

| 범주 | 전통 ML 모델 개발 | 파운데이션 모델 개발 |
|:---:|:---:|:---:|
| 모델링과 학습 | ML 지식 필수 | ML 지식이 있으면 유리 |
| 데이터셋 엔지니어링 | 특성 공학 역량 중요 | 데이터 중복 제거, 토큰화, 컨텍스트 검색, 품질 관리 등이 더 중요 |
| 추론 최적화 | 중요함 | 더 중요함 |

### 애플리케이션 개발

- 많은 팀들이 같은 모델을 사용한다면 애플리케이션 개발 과정을 통해 차별화를 이뤄야 함
- **평가**, **프롬프트 엔지니어링 및 컨텍스트 구성**, **AI 인터페이스** 라는 3가지 구성 요소가 있음

**평가(Evaluation)**

- 위험을 완화하고 기회를 발견하는 과정
- 모델을 조정하는 전체 주기에 걸쳐 평가가 이루어져야 함
- 모델을 선택하고, 진행 상황을 벤치마크하며, 애플리케이션 배포 준비 여부를 판단하는 데 쓰임
- 운영 중인 시스템의 문제 파악 및 개선 기회 발견에도 필수적
- 평가는 ML 엔지니어링에서도 중요했는데 파운데이션 모델에서는 더 중요함
- 파운데이션 모델은 개방형이기 때문에 출력을 예측할 수 없고, 이에 따라 평가도 어려워짐

**프롬프트 엔지니어링 및 컨텍스트 구성**

- 프롬프트 엔지니어링: 모델 가중치를 변경하지 않고 입력만으로 AI 모델에서 원하는 동작을 이끌어내는 것
- 컨텍스트 구성: 모델에 제공되는 입력의 일부로, 모델이 응답을 생성하는 데 사용하는 정보
- 프롬프트 엔지니어링은 단순히 모델에게 뭘 하라고 말하는 것이 아니라, 작업을 수행하는 데 필요한 컨텍스트 및 도구를 모델에게 제공하는 것

**AI 인터페이스**

- 최종 사용자가 AI 애플리케이션과 상호작용할 수 있는 인터페이스를 만드는 것
- 독립형 혹은 기존 제품에 탑재하는 형태로 개발됨
- AI 인터페이스 예시
  - 독립형 웹, 데스크톱, 모바일 애플리케이션
  - AI 모델에 빠르게 질의 가능한 브라우저 확장 프로그램
  - 채팅 애플리케이션에 통합된 챗봇
  - 플러그인 및 애드온 형태의 통합 API

### 풀스택 엔지니어링

- AI 엔지니어링 워크플로의 변화
  - ML 엔지니어링: 데이터 → 모델 → 제품
  - AI 엔지니어링: 제품 → 데이터 → 모델
