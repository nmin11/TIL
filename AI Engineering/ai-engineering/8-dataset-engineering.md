# 8. 데이터셋 엔지니어링

- **데이터셋 엔지니어링의 목표: 최고의 모델을 학습할 수 있는 데이터셋을 만드는 것**
  - 또한 할당된 예산 내에서 이를 달성하는 것
- 모델을 직접 개발할 여력이 없는 회사들이 적으므로, 많은 회사가 데이터에 주목하고 있음
- 모델이 더 많은 데이터를 요구하게 되면서 데이터 처리가 어려워지고 있으며, 이에 따른 인력 및 인프라 투자가 증가하고 있음
- 데이터셋을 구축할 때 고려해야 할 사항들
  - 데이터 큐레이션
  - 어떤 데이터가 얼마나 필요한지
  - 고품질 데이터
  - 데이터 합성 및 처리 기법들
- 사전 학습, 사후 학습 등 어느 단계인지에 따라 세부 접근 방식의 차이가 있지만, 여기서는 사후 학습 위주로 다룰 것
- **모델 중심 AI(Model-centric AI)** 와 구분되는 **데이터 중심 AI(Data-centric AI)** 가 떠오르고 있음

## 8.1 데이터 큐레이션

- 좋은 데이터는 모델을 더 똑똑하게, 더 안전하게 해주고, 더 긴 컨텍스트를 처리할 수 있게 함
- 나쁜 데이터는 모델의 편향과 환각을 키울 수 있음
- **데이터 큐레이션(Data Curation)** : 모델이 어떻게 학습하는지, 학습에 도움이 되는 자원이 무엇인지 이해하는 분야
- 용도별로 필요한 데이터들
  - 자기지도 학습 파인튜닝 = 데이터 시퀀스 필요
  - 지시 파인튜닝 = (지시, 응답) 쌍 필요
  - 선호도 파인튜닝 = (지시, 선호 응답, 비선호 응답) 쌍 필요
  - 보상 모델 학습 = ((지시, 응답), 점수) 쌍 필요
- 학습 데이터는 모델이 학습했으면 하는 행동을 먼저 보여줘야 함
  - 그러므로 생각의 사슬(CoT), 도구 사용 등 복잡한 행동을 가르치는 게 어려움
- 대화 인터페이스가 있는 애플리케이션용 데이터 큐레이션에서 고려해야 할 데이터 유형들
  - 싱글 턴 데이터: 모델이 개별 지시에 답하는 법을 가르침
  - 멀티 턴 데이터: 모델에게 작업을 해결하는 방법을 가르침
- 데이터 큐레이션에는 신규 데이터 뿐만 아니라 기존 데이터를 정제하는 과정도 포함됨
- 데이터 큐레이션이 따르는 3가지 기준
  - **데이터 품질(data quality)**
  - **데이터 커버리지(data coverage)**
  - **데이터 양(data quantity)**

### 데이터 품질

- 노이즈가 있는 데이터 = 관련이 없거나 일관성이 없는 데이터
- 고품질의 데이터 = 데이터가 자신의 일을 효율적이고 안정적으로 하는데 도움이 되는 데이터
- 데이터 품질을 결정하는 6가지 특성
  - **관련성(relevant)**
    - 학습 예시는 모델이 학습하려는 작업과 관련이 있어야 함
  - **작업 요구사항 부합(aligned with task requirement)**
    - 주석은 작업의 요구사항에 부합해야 함
    - '부합'하다는 표현을 쓴 이유 = 과제에 따라 정확하거나 올바른 응답이 꼭 사용자가 원하는 것이 아닐 수 있기 때문에
  - **일관성(consistent)**
    - 주석은 예시들끼리, 주석자들 간에 일관되어야 함
    - 그러므로 좋은 주석 가이드라인이 필요
  - **올바른 형식(correctly formatted)**
    - 모든 예시는 모델이 기대하는 형식을 따라야 함
  - **충분한 고유성(sufficiently unique)**
    - 데이터의 고유한 예시
    - 중복은 편향을 만들고 데이터 오염을 일으킬 수 있음
  - **규정 준수(compliant)**
    - 데이터는 모든 관련 정책을 지켜야 함

### 데이터 커버리지

- 모델의 학습 데이터는 모델이 풀어야 할 문제들의 범위를 포괄해야 함
- 애플리케이션의 다양한 사용 패턴을 담은 데이터를 확보하는 것이 모델 성능 확보의 핵심
- 좋은 커버리지 확보에는 데이터 다양성이 필수적이므로, 이 개념을 **데이터 다양성(data diversity)** 이라고도 부름
- 예시
  - 사용자마다 자세한 지시, 짧은 지시에 대한 선호도가 다르다면 둘 다 포함해야 함
  - 사용자 질의에 종종 오타가 있다면 오타가 포함된 예시도 포함해야 함
  - 애플리케이션이 여러 프로그래밍 언어를 다룬다면 학습 데이터에 관심 비중이 높은 프로그래밍 언어들이 포함되어야 함
- 실제 애플리케이션 사용 패턴에 맞춰 올바른 데이터 조합을 찾아가야 함
- 혹은 실험을 통해 최적의 데이터 조합을 찾을 수도 있음
- 데이터 다양성은 과제 종류, 주제 다양성, 출력 형식에 따라 달라질 수 있음

### 데이터 양

- 돈이 허락하는 한 많은 데이터
- 데이터 양을 결정하는 요소 3가지
  - **파인튜닝 기법**
    - 전체 파인튜닝이라면 부분 파인튜닝 기법들에 비해 몇 배나 많은 데이터가 필요해질 것
  - **과제 복잡성**
    - 복잡할수록 더 많은 데이터 필요
  - **기본 모델의 성능**
    - 기본 모델이 원하는 성능에 가까울수록 목표에 도달하는 데 필요한 예시가 적음
- 만약 데이터가 적다면 고급 기본 모델에 PEFT 파인튜닝 기법을 사용하는 것이 좋음
- 반대로 데이터가 크다면 더 작은 기본 모델에 전체 파인튜닝을 적용할 수도 있음
- 대규모 데이터셋을 구축하기 전에, 소규모 데이터셋(약 50에서 100개 가량)으로 파인튜닝의 개선 가능성을 살펴보는 것이 좋음
  - 작은 데이터셋 실험을 통해 앞으로 데이터가 얼마나 더 필요한지 추정 가능
- 먼저 저품질 데이터로 파인튜닝한 이후 고품질 데이터로 파인튜닝하면 필요한 고품질 데이터 양을 줄일 수 있음
  - 자기 지도 학습 → 지도 학습
  - 관련성 낮은 데이터 → 관련성 높은 데이터
  - 합성 데이터 → 실제 데이터

### 데이터 수집과 주석

- 데이터 수집의 목표: 사용자 프라이버시를 존중하고 규정을 지키면서 필요한 품질과 다양성을 갖춘 충분한 크기의 데이터셋을 만드는 것
- 데이터 수집이 포함하는 과정들
  - 공개 데이터 수집
  - 독점 데이터 구매
  - 데이터 주석 작업
  - 데이터 합성
- 데이터 수집 전략을 연구하는 분야가 아직은 틈새 영역이지만 점점 발전하고 있음
- 가장 중요한 데이터 소스는 보통 자체 애플리케이션에서 나오는 데이터
  - 사용자가 만든 데이터를 활용해 제품을 지속적으로 개선하는 데이터 플라이휠 구축
  - 완벽하게 관련성이 있으며 과제와도 정확히 맞아 떨어지기 때문에 이상적
  - 사용자 콘텐츠, 사용자 로그 데이터, 사용자 피드백 등
- 데이터 마켓플레이스를 활용하는 것도 좋음
  - 오픈 소스와 독점 데이터 모두 제공
  - 여러 데이터를 조합해서 활용해볼 것
- 공개 데이터셋 리소스들
  - [Hugging Face Datasets](https://huggingface.co/datasets)
  - [Kaggle](https://www.kaggle.com/datasets)
  - Google - [Dataset Search](https://datasetsearch.research.google.com/)
  - [Data.gov](https://data.gov/)
  - [ICPSR](https://www.icpsr.umich.edu/sites/icpsr/find-data)
  - [Open Data Network](https://www.opendatanetwork.com)
  - AWS - [Open Data](https://registry.opendata.aws/)
- 주석 작업
  - 파인튜닝을 위해 자체 데이터에 주석을 달아야 할 때가 많음
  - 주석 과정 자체도 어렵지만 명확한 주석 가이드라인을 만드는 것도 복잡함
  - 링크드인 및 몇몇 팀들은 주석 가이드라인이 AI 엔지니어링 파이프라인에서 가장 까다로운 부분 중 하나였다고 했음
  - 시간과 노력이 많이 드는 과정
  - 모델들이 똑똑하기 때문에 오히려 나오게 된 그럴듯한 결과를 믿었다가 위험한 결과로 이어질 수 있음

## 8.2 데이터 증강 및 합성

- **데이터 증강(data augmentation)** : 기존 데이터에서 새로운 데이터를 만듦
- **데이터 합성(data synthesis)** : 실제 데이터의 특성을 모방하는 데이터 생성
- 증강은 실제 데이터로부터, 합성은 실제가 아님
- 그래도 둘 다 데이터 생성을 자동화하는 것이 목표 → 때로는 두 용어를 같은 의미로 사용하기도 함
- AI는 실제와 매우 유사한 데이터를 정교하게 합성할 수 있게 되었음
- 하지만 합성 데이터가 사람이 직접 만든 데이터를 완전히 대체하지는 못함

### 데이터 합성을 하는 이유

- 데이터 합성을 통해 양, 커버리지, 품질 3개를 커버할 수 있음

**데이터 양 늘리기**

- 대규모 데이터를 AI 모델 학습 및 테스트에 공급할 수 있음

**데이터 커버리지 늘리기**

- 특정한 특성을 가진 데이터를 생성해 모델 성능 개선 및 특정 행동 유도 가능
- 유해 콘텐츠, 희귀 클래스 데이터, 사실 관계 확인 등을 위한 데이터 생성

**데이터 품질 향상**

- 사람의 근본적 한계로 인해, 사람이 만든 데이터보다 고품질의 데이터가 나올 수 있음

**프라이버시 문제 해결**

- 사람이 생성한 데이터에 프라이버시 이슈가 있기 때문에 합성 데이터를 사용하게 되는 경우가 많음

**모델 증류**

- 다른 모델의 행동을 모방하도록 모델을 학습시키는 기법
- 원본 모델과 비슷한 성능을 내면서도 더 저렴하거나 빠른 모델을 만들 수 있음
- 원본 모델이 생성한 데이터로 증류된 모델을 학습시키는 방식

### 전통적인 데이터 생성 기법

- 데이터 합성은 AI 뿐만 아니라 소프트웨어 테스팅, 게임, 로보틱스 분야에서 오랫동안 사용되어 왔음
- **절차적 생성(procedural generation)**
  - 사람이 직접 만들지 않고, 알고리즘으로 데이터를 생성하는 방식
  - 게임에서 레벨, 맵, 아이템, 캐릭터 등의 콘텐츠를 실시간으로 만드는 데 많이 쓰임
- 전통적인 데이터 증강 및 합성 방식들
  - 규칙 기반 방식
  - 시뮬레이션 방식

**규칙 기반**

- 데이터를 생성하는 가장 간단한 방법 = 미리 정해둔 규칙과 템플릿을 사용하는 것
- 기존 데이터에 간단한 변형을 가해서 새로운 데이터를 절차적으로 생성할 수도 있음
  - 이는 데이터 편향을 줄이는 데에도 도움이 됨
- **섭동(perturbation)**
  - 기존 데이터에 노이즈를 넣어서 새로운 데이터를 생성하는 것
  - 사실 노이즈 낀 데이터는 모델의 분류 잘못에 큰 오차를 발생시킬 수 있음
  - 하지만 섭동을 통해 모델 성능을 향상시키고 공격에 더 견고하게 만들 수도 있음
  - 텍스트 및 시각 데이터에도 섭동을 활용해서 데이터를 증강할 수 있음

**시뮬레이션**

- 실제 세계에서 실험을 통해 데이터를 수집하는 것은 고비용 고위험
- 그러므로 실험들을 가상으로 시뮬레이션하게 됨
- 자율주행, 로보틱스 등의 분야에서도 사고 및 물리적 손상 없이 최소 비용으로 여러 실험 수행 가능
- 시뮬레이션은 모델에게 도구 사용법을 가르치는 데이터 생성에도 많이 쓰임
- 시뮬레이션은 사람이 놓치는 행동들을 찾아내는 데 도움이 될 수 있음
  - 질의가 주어지면 다양한 행동 순서를 시뮬레이션
  - 행동 순서들을 실행해서 결과 확인
  - 가장 효율적인 행동을 그 질의에 대한 정답 응답으로 사용
- 현실에서 일어나기 힘든 사건의 데이터를 생성하는 데 유용
  - 이를 통해 더 다양한 가능성들로부터의 학습이 가능해짐

### AI 기반 데이터 합성

- AI는 API 호출, 사람의 행동 등을 시뮬레이션 할 수 있음
- 게임의 경우, AI끼리 빠르게 게임을 플레이하는 '셀프플레이' 방식이 있음
  - 셀프 플레이는 일반적인 에이전트들끼리 서로 다른 전략을 협상하게 하는 데에도 유용
- AI의 바꿔쓰기나 번역 능력으로 기존 데이터셋을 늘릴 수 있음
- 번역
  - AI로 자원이 풍부한 언어의 데이터를 자원이 부족한 언어로 번역해서 저자원 언어 모델 학습을 돕는 경우도 흔함
  - **역번역(back-translation)** 으로 번역 품질을 확인할 수도 있음
  - 자연어 뿐만이 아니라 프로그래밍 언어 사이에서도 번역 가능
  - 예시: 코드 조각으로 시작해서 AI로 설명 및 문서 제작 → AI로 설명과 문서로부터 코드 조각 생성 → 생성된 코드가 원본을 제대로 구현했는지 검증
- 합성 데이터는 사전 학습보다 사후 학습에 훨씬 더 자주 쓰임
  - 사전 학습의 목표는 모델의 '새로운' 지식을 늘리는 것이기 때문
  - 하지만 인터넷 데이터에 의존하는 모델들은 이미 합성 데이터로 사전 학습되고 있을 가능성이 높음
  - 사후 학습에서의 데이터 합성은 지시 데이터 및 선호도 데이터를 포함한 학습 데이터를 만들기 까다롭기 때문에 실시됨

**지시 데이터 합성**

- 지시 파인튜닝에서는 각 예시에 지시와 응답이 포함됨
- **지시 생성**
  - 활용 사례를 충분히 다루는 지시 유형 목록 작성
  - 목록의 항목마다 일정 수의 지시 생성
- **응답 생성**
  - 지시 하나당 하나 또는 여러 개의 응답 생성 가능
- AI는 짧은 지시보다 고품질의 긴 응답을 생성하기 어려우며, 응답이 길수록 환각을 일으킬 확률도 높아짐
- 역지시(reverse instruction)
  - 기존의 긴 고품질 콘텐츠를 가져와서 AI로 그런 콘텐츠를 유도할 수 있는 프롬프트를 생성하는 것
  - 이를 통해 환각은 피하면서 더 고품질의 지시 데이터를 얻을 수 있음
  - 작동 예시
    - 소수의 초기 예시로 약한 모델 학습
    - 약한 모델로 기존 고품질 콘텐츠에 대한 지시를 생성해서 고품질 지시 데이터 생성
    - 새로운 고품질 지시 데이터로 약한 모델을 파인튜닝
    - 원하는 성능을 도달할 때까지 반복
- 합성 데이터를 활용해 더 긴 컨텍스트를 이해하도록 모델을 파인튜닝할 수도 있음
  - 긴 문서를 짧은 덩어리로 나누기
  - 짧은 덩어리 마다 여러 (질의, 응답) 쌍 생성
  - (질의, 응답) 쌍마다 원래의 긴 문서를 컨텍스트로 사용
- 라마 3의 방식
  - AI로 문제 설명을 생성
  - AI로 각 문제의 해결책을 다양한 프로그래밍 언어로 생성
  - AI로 생성된 코드를 테스트할 단위 테스트 생성
  - AI에게 합성된 코드의 오류 수정 지시
  - AI로 생성된 코드를 다양한 프로그래밍 언어로 번역하고, 테스트를 통과하지 못하는 번역된 코드는 걸러냄
  - AI로 코드 설명과 문서 작성 등 코드에 대한 대화 생성, 역번역 검증을 통과하지 못하는 설명 및 문서는 걸러냄

**데이터 검증**

- 데이터 품질은 모델 성능을 결정짓는 중요한 요소
- AI가 생성한 데이터의 품질은 AI 평가자를 통해 수행할 수 있음
- **AI 검증기(AI verifier)**
  - 기능적 정확성으로 검증할 수 없는 합성 데이터는 AI 검증기를 사용하는 것이 일반적
  - AI 검증기는 범용 AI 평가자나 특화된 채점기일 수 있음
  - 검증 문제 구성 방법도 다양
    - 1점부터 5점까지 점수를 매기거나, 단순히 좋음 또는 나쁨으로 분류하거나
- 합성 데이터가 실제 데이터와 닮길 원한다면, 구별이 얼마나 어려운지로 합성 데이터의 품질을 측정할 수 있음
- 실제 데이터와 마찬가지로 합성 데이터도 휴리스틱을 사용해서 걸러낼 수 있음
  - 반복적인 예시
  - 너무 길거나 너무 짧은 예시
  - 같은 지시에 다른 응답을 가진 예시
  - 출력이 입력을 그대로 반복하는 예시
- 합성 데이터를 평가하는 것은 아직 어려운 단계

**AI 생성 데이터의 한계**

- AI 생성 데이터가 사람이 생성한 데이터를 완전히 대체하기 어려운 4가지 이유
  - **품질의 차이**
  - **피상적 모방의 한계**
  - **모델 성능 저하 가능성**
  - **데이터의 계보가 불분명해지는 문제**

**품질 관리**

- 쓰레기가 들어가면 쓰레기가 나온다
- 데이터를 평가할 신뢰할 수 있는 방법, 지표를 개발하는 것이 꼭 필요

**피상적 모방(superfical imitation)**

- 모방을 통해 얻는 성능은 겉보기에만 좋아 보일 수도 있음
- 모방 범위 이외의 영역에서 사실적 정확성 및 일반화에서 어려움을 겪게 될 수 있음
- 더 심각한 것은, 모방이 학생 모델로 하여금 환각을 일으키게 할 수 있다는 것
- 추론 능력의 개선을 위해서는 기본 모델의 품질 향상에 집중해야 함

**모델 성능 저하 가능성**

- 일부 연구에 따르면, AI 생성 데이터로 반복적으로 학습했을 때 모델에 결함이 생기고 성능이 더 떨어진다고 함
- 이 현상을 **모델 붕괴(model collapse)** 라고 부름
- 모델 붕괴는 사전 학습과 사후 학습 모두에서 발생 가능
- AI 모델은 확률이 높은 사건을 생성할 가능성이 높고, 확률이 낮은 사건을 생성할 가능성이 낮음
  - 반복 학습이 진행될수록 더 일반적인 사건만 출력하게 되고 희귀 사건은 잊어버리게 될 것
- 그러므로 합성 데이터와 실제 데이터를 섞으려는 노력이 필요
- 모델의 출력이 원래 학습 분포의 특성을 잘 반영할수록 피드백 루프가 줄어들고 편향이 커질 위험이 줄어듦

**불분명한 데이터 계보**

- AI 모델은 학습 데이터의 영향을 받아 사용자도 모르게 그 내용을 그대로 출력하게 될 수 있음
- 만약 저작권을 위반한 데이터가 있다면 이런 식으로 노출될 수 있으므로 무척 위험해짐
- 명확한 데이터 계보 없이는 모델을 상업적으로 쓸 수 있는지, 성능을 믿을 수 있는지 판단하기 어려움

### 모델 증류(model distillation)

- **지식 증류(knowledge distillation)** 라고도 불림
- 작은 모델(학생)이 큰 모델(교사)을 모방하도록 학습시키는 방식
- 전통적으로 모델 증류의 목표는 배포용으로 더 작은 모델을 만드는 것
- 합성 데이터로 학습한다고 해서 모두 모델 증류인 것은 아님
- 학생 모델의 목표는 교사 모델과 같은 성능
- 하지만 합성 데이터를 사용해서 교사 모델보다 더 크고 강력한 학생 모델을 학습시키는 것도 가능

## 8.3 데이터 처리

- 데이터 처리 효율을 높이는 방법들
  - 컴퓨팅 자원을 절약할 수 있는 순서라면 어떤 순서라도 상관 없음
  - 데이터 처리 스크립트 적용 전에 꼭 테스트를 진행할 것
  - 원본 데이터를 바로 수정하지 말고 사본을 활용할 것

### 데이터 검사

- 원시 데이터셋을 모은 후 가장 먼저 할 일은 데이터 품질 파악
  - 데이터의 기본 정보와 통계
  - 데이터가 어디서 나왔는지
  - 어떻게 처리된 데이터인지
  - 데이터의 다른 활용 사례들은 어떤 것들이 있는지
- 더 세부적으로 봐야 할 것들
  - 어떤 토큰이 더 흔한가 (토큰 분포)
  - 입력 길이, 응답 길이 등을 나타낸 그래프
  - 특별한 토큰이 사용되었는지
  - 데이터 주제와 언어 분포를 구할 수 있는지, 또 주제와 언어들이 과제와 얼마나 관련이 있는지
- 데이터를 이해하기 위해 다양한 통계를 창의적으로 활용 가능
  - 데이터 출처, 시간, 주석자 등으로 나눠서 시각화
  - 더 길거나 짧은 응답, 또는 더 높거나 낮은 점수를 받는 질의 패턴
  - 이상값이 있는지, 이상값의 원인은 무엇이며, 어떻게 처리할 것인지
- 점수의 정규분포
  - 모든 주석자의 점수가 정규분포를 따르는가
  - 특정 주석자가 낮은 점수나 높은 점수만 주도록 편향되어 있지는 않은가
- 데이터를 직접 살펴보려고 노력하자
  - 데이터 탐색 도구들이 많지만 도구들이 직접 보는 것을 대신할 수는 없음
  - OpenAI 그렉 브록맨 '데이터를 살펴보는 건 아마도 ML의 모든 활동 중에서 가장 저평가되어 있지만, 실제로 이는 매우 가치 있는 활동이다.'
  - 예시들이 말이 되는지 확인할 것
  - 주석이 달린 데이터라면 몇 개 질의를 골라 직접 주석을 달아보고, 내 주석과 기존 주석이 일치하는지 확인해보기
- 응답의 사실 일치 확인
  - 예시들이 얼마나 고유한가
  - 같은 질의인데 다른 응답을 가진 예시가 있는가
  - 같은 응답인데 다른 질의를 가진 예시가 있는가

### 데이터 중복 제거

- 중복 데이터는 데이터 분포를 왜곡하고 모델에 편향을 만들 수 있음
- 중복 데이터는 모델 성능에 악영향을 미치며, 시간과 컴퓨팅 자원을 낭비할 수 있음
- 중복의 형태는 다양하며, 일부는 찾아내기도 어려움
- 예시: 문서 데이터셋에서의 중복 데이터 사례들
  - 전체 문서 중복: 같은 문서가 중복으로 나타나는 경우
  - 문서 내 중복: 같은 단락이 한 문서가 중복으로 나타나는 경우
  - 문서 간 중복: 같은 인용구 등이 여러 문서에서 나타나는 경우
- 무엇을 중복으로 볼지는 정의에 따라 달라짐
- 유사성 측정 기법을 중복 제거 작업에 활용할 수 있음
- 데이터 중복 제거의 구체적인 방법들
  - **쌍대 비교**
    - 정확한 일치, n-gram 유사도, 퍼지 매칭, 의미적 유사도 점수 등을 활용해서 각 예시를 다른 모든 예시와 함께 유사도 점수 계산
    - 큰 데이터셋에서는 많은 비용 필요
  - **해싱**
    - 예시들을 서로 다른 버킷으로 해싱
    - 같은 버킷에 속하는 예시들만 유사도 검증
    - MinHash, Bloom Filter 등을 활용
  - **차원 축소**
    - 먼저 데이터 차원을 줄인 다음 쌍대 비교 수행
    - 벡터 검색 기법들 활용 가능

### 데이터 정리 및 필터링

- 불필요한 형식 토큰 없애기
  - 많은 공개 데이터셋은 인터넷에서 긁어온 것이기 때문에 HTML 태그를 지우는 작업 필요
- PII, 민감 데이터, 저작권이 있는 데이터, 유해성 데이터 등 정책에 맞지 않는 데이터들 정리
- '데이터 검증'에서 찾아낸 저품질 데이터는 바로 제거하기 보다는 직접 들여다 보는 과정이 필요
  - 저품질 데이터를 찾아내는 휴리스틱 확보에 활용 가능
- 필요 이상의 데이터도 걸러낼 수 있음
  - 중요도 샘플링을 활용해서 과제에서 가장 중요한 예시들을 찾아낼 수 있음

### 데이터 형식 맞추기

- 각 모델들은 특정한 토크나이저를 사용하고 특정 채팅 템플릿 형식의 데이터를 기대함
- 프롬프트 엔지니어링에서 쓰이던 지시 형식이 파인튜닝에서는 달라질 수 있으므로 유의해야 함
- 파인튜닝 데이터 형식에 따라 모델 성능에 영향을 줄 수 있으므로, 어떤 형식이 가증 좋은지 실험해보는 것이 좋음
