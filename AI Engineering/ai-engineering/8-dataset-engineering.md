# 8. 데이터셋 엔지니어링

- **데이터셋 엔지니어링의 목표: 최고의 모델을 학습할 수 있는 데이터셋을 만드는 것**
  - 또한 할당된 예산 내에서 이를 달성하는 것
- 모델을 직접 개발할 여력이 없는 회사들이 적으므로, 많은 회사가 데이터에 주목하고 있음
- 모델이 더 많은 데이터를 요구하게 되면서 데이터 처리가 어려워지고 있으며, 이에 따른 인력 및 인프라 투자가 증가하고 있음
- 데이터셋을 구축할 때 고려해야 할 사항들
  - 데이터 큐레이션
  - 어떤 데이터가 얼마나 필요한지
  - 고품질 데이터
  - 데이터 합성 및 처리 기법들
- 사전 학습, 사후 학습 등 어느 단계인지에 따라 세부 접근 방식의 차이가 있지만, 여기서는 사후 학습 위주로 다룰 것
- **모델 중심 AI(Model-centric AI)** 와 구분되는 **데이터 중심 AI(Data-centric AI)** 가 떠오르고 있음

## 8.1 데이터 큐레이션

- 좋은 데이터는 모델을 더 똑똑하게, 더 안전하게 해주고, 더 긴 컨텍스트를 처리할 수 있게 함
- 나쁜 데이터는 모델의 편향과 환각을 키울 수 있음
- **데이터 큐레이션(Data Curation)** : 모델이 어떻게 학습하는지, 학습에 도움이 되는 자원이 무엇인지 이해하는 분야
- 용도별로 필요한 데이터들
  - 자기지도 학습 파인튜닝 = 데이터 시퀀스 필요
  - 지시 파인튜닝 = (지시, 응답) 쌍 필요
  - 선호도 파인튜닝 = (지시, 선호 응답, 비선호 응답) 쌍 필요
  - 보상 모델 학습 = ((지시, 응답), 점수) 쌍 필요
- 학습 데이터는 모델이 학습했으면 하는 행동을 먼저 보여줘야 함
  - 그러므로 생각의 사슬(CoT), 도구 사용 등 복잡한 행동을 가르치는 게 어려움
- 대화 인터페이스가 있는 애플리케이션용 데이터 큐레이션에서 고려해야 할 데이터 유형들
  - 싱글 턴 데이터: 모델이 개별 지시에 답하는 법을 가르침
  - 멀티 턴 데이터: 모델에게 작업을 해결하는 방법을 가르침
- 데이터 큐레이션에는 신규 데이터 뿐만 아니라 기존 데이터를 정제하는 과정도 포함됨
- 데이터 큐레이션이 따르는 3가지 기준
  - **데이터 품질(data quality)**
  - **데이터 커버리지(data coverage)**
  - **데이터 양(data quantity)**

### 데이터 품질

- 노이즈가 있는 데이터 = 관련이 없거나 일관성이 없는 데이터
- 고품질의 데이터 = 데이터가 자신의 일을 효율적이고 안정적으로 하는데 도움이 되는 데이터
- 데이터 품질을 결정하는 6가지 특성
  - **관련성(relevant)**
    - 학습 예시는 모델이 학습하려는 작업과 관련이 있어야 함
  - **작업 요구사항 부합(aligned with task requirement)**
    - 주석은 작업의 요구사항에 부합해야 함
    - '부합'하다는 표현을 쓴 이유 = 과제에 따라 정확하거나 올바른 응답이 꼭 사용자가 원하는 것이 아닐 수 있기 때문에
  - **일관성(consistent)**
    - 주석은 예시들끼리, 주석자들 간에 일관되어야 함
    - 그러므로 좋은 주석 가이드라인이 필요
  - **올바른 형식(correctly formatted)**
    - 모든 예시는 모델이 기대하는 형식을 따라야 함
  - **충분한 고유성(sufficiently unique)**
    - 데이터의 고유한 예시
    - 중복은 편향을 만들고 데이터 오염을 일으킬 수 있음
  - **규정 준수(compliant)**
    - 데이터는 모든 관련 정책을 지켜야 함

### 데이터 커버리지

- 모델의 학습 데이터는 모델이 풀어야 할 문제들의 범위를 포괄해야 함
- 애플리케이션의 다양한 사용 패턴을 담은 데이터를 확보하는 것이 모델 성능 확보의 핵심
- 좋은 커버리지 확보에는 데이터 다양성이 필수적이므로, 이 개념을 **데이터 다양성(data diversity)** 이라고도 부름
- 예시
  - 사용자마다 자세한 지시, 짧은 지시에 대한 선호도가 다르다면 둘 다 포함해야 함
  - 사용자 질의에 종종 오타가 있다면 오타가 포함된 예시도 포함해야 함
  - 애플리케이션이 여러 프로그래밍 언어를 다룬다면 학습 데이터에 관심 비중이 높은 프로그래밍 언어들이 포함되어야 함
- 실제 애플리케이션 사용 패턴에 맞춰 올바른 데이터 조합을 찾아가야 함
- 혹은 실험을 통해 최적의 데이터 조합을 찾을 수도 있음
- 데이터 다양성은 과제 종류, 주제 다양성, 출력 형식에 따라 달라질 수 있음

### 데이터 양

- 돈이 허락하는 한 많은 데이터
- 데이터 양을 결정하는 요소 3가지
  - **파인튜닝 기법**
    - 전체 파인튜닝이라면 부분 파인튜닝 기법들에 비해 몇 배나 많은 데이터가 필요해질 것
  - **과제 복잡성**
    - 복잡할수록 더 많은 데이터 필요
  - **기본 모델의 성능**
    - 기본 모델이 원하는 성능에 가까울수록 목표에 도달하는 데 필요한 예시가 적음
- 만약 데이터가 적다면 고급 기본 모델에 PEFT 파인튜닝 기법을 사용하는 것이 좋음
- 반대로 데이터가 크다면 더 작은 기본 모델에 전체 파인튜닝을 적용할 수도 있음
- 대규모 데이터셋을 구축하기 전에, 소규모 데이터셋(약 50에서 100개 가량)으로 파인튜닝의 개선 가능성을 살펴보는 것이 좋음
  - 작은 데이터셋 실험을 통해 앞으로 데이터가 얼마나 더 필요한지 추정 가능
- 먼저 저품질 데이터로 파인튜닝한 이후 고품질 데이터로 파인튜닝하면 필요한 고품질 데이터 양을 줄일 수 있음
  - 자기 지도 학습 → 지도 학습
  - 관련성 낮은 데이터 → 관련성 높은 데이터
  - 합성 데이터 → 실제 데이터

### 데이터 수집과 주석

- 데이터 수집의 목표: 사용자 프라이버시를 존중하고 규정을 지키면서 필요한 품질과 다양성을 갖춘 충분한 크기의 데이터셋을 만드는 것
- 데이터 수집이 포함하는 과정들
  - 공개 데이터 수집
  - 독점 데이터 구매
  - 데이터 주석 작업
  - 데이터 합성
- 데이터 수집 전략을 연구하는 분야가 아직은 틈새 영역이지만 점점 발전하고 있음
- 가장 중요한 데이터 소스는 보통 자체 애플리케이션에서 나오는 데이터
  - 사용자가 만든 데이터를 활용해 제품을 지속적으로 개선하는 데이터 플라이휠 구축
  - 완벽하게 관련성이 있으며 과제와도 정확히 맞아 떨어지기 때문에 이상적
  - 사용자 콘텐츠, 사용자 로그 데이터, 사용자 피드백 등
- 데이터 마켓플레이스를 활용하는 것도 좋음
  - 오픈 소스와 독점 데이터 모두 제공
  - 여러 데이터를 조합해서 활용해볼 것
- 공개 데이터셋 리소스들
  - [Hugging Face Datasets](https://huggingface.co/datasets)
  - [Kaggle](https://www.kaggle.com/datasets)
  - Google - [Dataset Search](https://datasetsearch.research.google.com/)
  - [Data.gov](https://data.gov/)
  - [ICPSR](https://www.icpsr.umich.edu/sites/icpsr/find-data)
  - [Open Data Network](https://www.opendatanetwork.com)
  - AWS - [Open Data](https://registry.opendata.aws/)
- 주석 작업
  - 파인튜닝을 위해 자체 데이터에 주석을 달아야 할 때가 많음
  - 주석 과정 자체도 어렵지만 명확한 주석 가이드라인을 만드는 것도 복잡함
  - 링크드인 및 몇몇 팀들은 주석 가이드라인이 AI 엔지니어링 파이프라인에서 가장 까다로운 부분 중 하나였다고 했음
  - 시간과 노력이 많이 드는 과정
  - 모델들이 똑똑하기 때문에 오히려 나오게 된 그럴듯한 결과를 믿었다가 위험한 결과로 이어질 수 있음
